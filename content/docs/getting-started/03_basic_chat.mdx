# 基础对话

## 概述

Claude Code 不是搜索引擎——不是你问一句它答一句。**有效的对话方式是：把 AI 当成一个能看到你全部代码的同事**。你告诉它上下文（你在做什么、你关心什么问题），它能给出远超复制粘贴到网页版 ChatGPT 的回答。

> **类比**: 在 ChatGPT 里问"这段代码有什么问题"，等于你把一页纸传给一个从未上过你项目的顾问。在 Claude Code 里问同样的话，等于你把顾问请到你的工位上，让他翻你的代码库、跑你的编译命令、看你的 git 历史——然后回答。

**关键转折**：从"随便聊聊"切换到"让 AI 读你的项目"。记住这条黄金规则：

> ⚠️ **启动 `claude` 之前，先 `cd` 到你的项目根目录。** Claude Code 的工作范围以当前目录为起点——它只能看到你 `cd` 进去的项目，看不到你电脑上其他地方的代码。

---

## 快速上手：三种对话模式

以下用嵌入式 C 代码演示三种最常用的对话模式。假设你在一个 STM32 固件项目目录下启动 `claude`。

### 模式一：让 AI 读代码

**场景**: 你接手了一个同事的 GPIO 驱动，想快速理解整体结构。

**你输入**：
```
请分析 drivers/gpio/gpio_mcu.c，解释：
1. 整体代码结构（初始化 → 读写 → 中断处理）
2. 有哪些函数是对外暴露的接口
3. 和 HAL 库的关系是什么
```

**AI 会输出**：
- 文件结构拆解，每个函数的一句话功能
- 标注哪些是 public 接口（如 `GPIO_Init`、`GPIO_WritePin`）
- 指出它调用了哪些 HAL 函数（如 `HAL_GPIO_WritePin`）
- 可能还会画一个调用关系图（Mermaid 格式）

**为什么比 ChatGPT 强**：AI 不只是读了这一个文件——它还会看 `gpio_mcu.h` 的声明、`#include` 的其他头文件、甚至搜索项目中谁调用了这些函数来分析接口的使用方式。

> **你的角色**: 不是"解释者"（不用自己一行行看）——变成"审查者"（判断 AI 的解释对不对）。

### 模式二：让 AI 深层分析

**场景**: 你的 I2C 中断处理偶尔丢数据，怀疑是竞态条件，但不确定。

**你输入**：
```
drivers/i2c/i2c_irq.c 里的 I2C1_EV_IRQHandler 中断处理函数，
请分析是否存在以下问题：
1. 和主循环中 i2c_master.c 的 I2C_WriteBuffer 是否有共享变量未保护
2. 中断嵌套时，寄存器操作是否为原子操作
3. 错误处理路径是否可能死锁
```

**AI 会**：
- 先读 `i2c_irq.c` 的中断处理函数
- 再读 `i2c_master.c` 的写入函数，交叉比对共享变量（如 `tx_buffer`、`tx_index`）
- 逐个检查你的三个问题，指出：共享变量 `tx_index` 在中断和主循环都有写入但没有 volatile 修饰也没有关中断保护——这是潜在竞态
- 给出修复建议（加 `__disable_irq()` / `__enable_irq()` 临界区 或改用 `volatile` + 原子操作）

**为什么这比你自己查快**：你自己查要打开 3-4 个文件来回翻、画变量访问图。AI 一次性读完所有相关文件，交叉比对。

### 模式三：让 AI 改代码

**场景**: 你想给 GPIO 驱动加上错误处理——初始化失败时返回错误码而不是直接 `assert`。

**你输入**：
```
请修改 drivers/gpio/gpio_mcu.c 的 GPIO_Init 函数：
1. 把所有的 assert() 失败改为返回 HAL_StatusTypeDef 错误码
2. 在 gpio_mcu.h 中同步更新函数声明
3. 搜索项目中所有调用 GPIO_Init 的地方，看是否需要同步修改
```

**AI 会**：
- 修改 `GPIO_Init` 函数体（`assert()` → `return HAL_ERROR` 等）
- 修改头文件函数声明
- 搜索项目中所有 `GPIO_Init(` 的调用点，列出需要改的，逐个让你确认

> ⚠️ **AI 改代码时你的角色**：AI 每次编辑文件前都会弹出确认框——你会看到具体的 diff。**你的职责不是改代码，而是审查 AI 的改动是否正确、有没有引入新问题。**

## 对话禁区：避免模糊提问

你提问的清晰度直接决定了 AI 的回答质量。以下是两个典型反例：

| ❌ 模糊提问 | ✅ 有效提问 | 区别 |
|------------|-----------|------|
| "帮我写个程序" | "帮我写一个 STM32F4 的 GPIO 初始化函数，参数是 GPIOA 端口和 Pin 5，推挽输出，无上拉下拉，最大速度 50MHz" | 告诉 AI 输入参数、输出要求、约束条件 |
| "这段代码有问题吗" | "gpio_mcu.c 的 GPIO_WritePin 函数在连续翻转时，ODR 寄存器操作是原子的吗？会不会被中断打断？" | 指定具体文件、具体函数、你担心什么问题 |
| "帮我把 I2C 改好" | "I2C 在读写 EEPROM 时偶尔返回 HAL_TIMEOUT，帮我在 i2c_eeprom.c 里定位可能是哪一步超时，并给出修复方案" | 描述现象、位置、期望的输出 |

**模糊提问的后果**：AI 会猜你的意图，给出一个"通用"答案——而通用答案对你的具体项目大概率没用，甚至改错。

## 对话技巧速查

| 技巧 | 示例 |
|------|------|
| **给文件路径** | "看 `drivers/i2c/i2c_master.c:120-150`" 而不是 "看 i2c 那个文件" |
| **说清上下文** | "这段代码是跑在 STM32F407 上，使用了 HAL 1.8，编译器是 GCC" |
| **限定范围** | "只改 i2c_master.c，不要动其他文件" |
| **分步执行** | "先帮我找到所有调用 I2C_Write 的地方，我看一下再决定怎么改" |
| **给出验收标准** | "改完后这个函数应该返回 HAL_OK 或 HAL_ERROR，不能有 assert" |
| **用 /compact** | 对话太长时输入 `/compact`，AI 会压缩历史，保留关键上下文继续工作 |

## 常见问题

**Q: AI 改错了怎么办？**

直接在对话里说"不对，我期望的是 XXX"。AI 会根据你的反馈修正。如果你的项目用了 git，可以用 `git diff` 看改动，`git checkout -- <file>` 回滚——AI 也能帮你做这些。

**Q: 对话太长了怎么办？**

输入 `/compact`——AI 会压缩对话历史，保留关键决策和修改，释放上下文空间。你仍然可以基于之前的对话继续工作。

**Q: 我在哪里输入命令？**

在 Claude Code 的交互界面里直接输入。以 `>` 开头的是你和 AI 对话，以 `/` 开头的是命令（如 `/help`、`/clear`）。下一篇会详细讲命令。

**Q: AI 给出的代码能直接跑吗？**

不一定。AI 理解你的项目结构但它不是编译器——你需要自己编译验证。把它当成一个写了初稿的同事，你负责审查和测试。

**Q: 我在 IDE 里能用吗？**

能。装好 VS Code 或 JetBrains 插件后，你可以在 IDE 侧边栏里直接对话 Claude Code，和终端里用完全一样。详见 [1-5-IDE集成.md](#)。
