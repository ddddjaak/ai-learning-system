    MCP 插件使用¶

    什么是 MCP¶

    MCP（Model Context Protocol，模型上下文协议） 是 Anthropic 发布的一个开放标准——让 AI 模型能安全、标准化地连接外部工具和数据源。

    一句话：MCP 就是 AI 的 USB 协议。USB 让电脑能即插即用地连接键盘、鼠标、U 盘——MCP 让 AI 能即插即用地连接代码库、文档、数据库、API。任何开发者都可以按 MCP 标准写一个 "MCP Server"，然后 Claude Code 就能直接调用它。

    MCP 的架构¶

    ┌──────────────┐     MCP 协议      ┌─────────────────┐
│  Claude Code │ ←──────────────→ │   MCP Server     │
│  (Host/Client)│    JSON-RPC      │ (你电脑上的进程)  │
└──────────────┘                   └────────┬────────┘
                                            │
                                   ┌────────┴────────┐
                                   │ 实际能力          │
                                   │ • 读你的代码库    │
                                   │ • 查在线文档      │
                                   │ • 执行脚本        │
                                   │ • 存取记忆        │
                                   └─────────────────┘
    • Claude Code = Host，负责发现和调用 MCP 工具
    • MCP Server = 一个独立进程（在你本机运行），提供具体能力
    • 协议 = JSON-RPC over stdio/HTTP，AI 和 Server 之间的标准通信方式

    对你意味着什么¶

    没有 MCP 有 MCP
    AI 只能靠训练数据"回忆"——HAL 库 API 可能是 3 年前的版本 AI 实时查你项目的代码结构（codegraph）+ 拉最新官方文档（context7）
    你要手动复制粘贴代码给 AI AI 直接读你的代码库
    你要手动翻 PDF 手册、粘贴段落给 AI AI 自动拉取对应章节
    每次新会话 AI 都是"白纸一张" memory MCP 让 AI 记住上次的讨论
    复杂推理一次性说完，错了也不知道 sequential-thinking MCP 让 AI 分步推理、自我纠错

    核心价值¶

    场景：你的 I2C 驱动在高低温循环测试中偶尔丢数据。你让 AI 帮忙排查。

    步骤 没有 MCP 的 AI 有 MCP 的 AI
    1. 定位 I2C 中断处理代码 你说"在 i2c_driver.c 大概 200 行左右" AI 用 codegraph 搜 I2C1_EV_IRQHandler → 精确找到第 234 行
    2. 查调用链 你手动 grep I2C_ReadByte → 50 条匹配，大部分是注释 AI 用 codegraph_trace → 3 个精确调用者，含回调注册
    3. 确认 API 用法 AI 凭训练数据给出 HAL_I2C_Master_Receive 的旧版签名 AI 用 context7 拉 HAL v1.8 的最新文档 → API 签名完全匹配
    4. 计算时序参数 你手动用计算器算 PSC/ARR AI 用 node_repl 直接算并验证
    5. 记录根因 下次新会话重说一遍 AI 用 memory 自动存储 → 下次自动回忆 "I2C 丢数据是因为 SCL 高电平时间不够"

    全景位置¶

    CLAUDE.md (2-3) ──→ 自定义 skill (2-4) ──→ MCP 插件 (2-5) ──→ Hooks (2-6)
                          │                       ↑
                          │                  你现在在这里
                          ↓
                   skill 定义"流程怎么做"
                                          MCP 提供"做流程需要的信息和能力"

    类比：skill 是厨房的菜谱（流程），MCP 是厨房的设备（刀具、炉灶、冰箱）。菜谱告诉你先切菜再炒——但如果没有刀（codegraph，无法分析代码）和冰箱里的食材（context7，没有最新文档），菜谱本身做不出菜。

    怎么配置 MCP¶

    配置文件位置¶

    MCP Server 在 mcp.json 中注册。有两个级别：

    文件 位置 作用范围
    用户级 ~/.claude/mcp.json 你的所有项目
    项目级 项目根目录 .mcp.json 当前项目（可提交 git 共享给团队）

    完整配置示例¶

    这是一个 AE 工程师的典型 MCP 配置：

    {
  "mcpServers": {
    "codegraph": {
      "type": "stdio",
      "command": "codegraph",
      "args": ["server"]
    },
    "context7": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@context7/mcp-server"]
    },
    "memory": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-memory"]
    },
    "node_repl": {
      "type": "stdio",
      "command": "node",
      "args": ["-e", "require('@anthropic/mcp-server-node-repl').start()"]
    },
    "sequential-thinking": {
      "type": "stdio", 
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-sequential-thinking"]
    }
  }
}

    ⚠️ 注意：以上是示例路径。实际配置中的 command 和 args 取决于你安装 MCP Server 的方式（全局安装 vs npx）。AI 可以帮助你生成正确的配置——说 "帮我配置 codegraph MCP" 即可。

    怎么安装 MCP Server¶

    MCP Server 本质上就是一个普通程序——它运行在你的电脑上，Claude Code 启动它并和它通信。安装方式取决于这个 Server 是用什么语言写的、发布在什么平台上。

    三种安装方式¶

    方式 命令示例 优点 缺点
    npx 即时运行（推荐入门） npx -y @xxx/mcp-server 不占磁盘、自动最新版 每次启动要下载（几秒）、内网可能连不上 npm
    npm 全局安装 npm install -g @xxx/mcp-server 启动快（已下载）、离线可用 占磁盘空间、需要手动更新
    pip 安装（Python 写的 Server） pip install xxx-mcp-server Python 生态丰富 需要 Python 环境、可能有依赖冲突
    二进制下载（Go/Rust 写的 Server） 下载可执行文件到 PATH 单文件、无依赖、启动最快 需要手动下载和更新

    AE 工程师推荐：先用 npx 快速验证（不折腾），确认好用了再考虑 npm install -g 全局安装（启动更快）。

    逐一手把手安装（6 个 MCP）¶

    1. codegraph¶

    codegraph 是一个独立的命令行工具，需要先安装再配置。

    步骤：

    # Windows（PowerShell，管理员权限）
irm https://codegraph.sh/install.ps1 | iex

# macOS / Linux
curl -fsSL https://codegraph.sh/install.sh | sh

    验证安装： 

    codegraph --version
# 输出：codegraph v2.x.x

    索引项目（必须先索引才能用）： 

    cd 你的项目根目录
codegraph init     # 初始化索引

    配置 mcp.json（安装成功后）： 

    {
  "mcpServers": {
    "codegraph": {
      "type": "stdio",
      "command": "codegraph",
      "args": ["server"]
    }
  }
}

    ⚠️ 关键：codegraph 必须先 init 生成 .codegraph/ 索引目录，否则 Claude Code 里查不到任何符号。

    2. context7¶

    context7 通过 npm 分发，推荐用 npx 运行（零安装）。

    无需手动安装——在 mcp.json 中配置 npx 即可自动下载运行：

    {
  "mcpServers": {
    "context7": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@context7/mcp-server"]
    }
  }
}

    -y 参数表示"如果本地没有就自动下载，不询问"。首次启动会下载 ~20MB，后续启动从缓存加载（秒级）。

    如果想全局安装（启动更快，但需手动更新）： 

    npm install -g @context7/mcp-server
    安装后 mcp.json 中把 command 改为 "context7-mcp-server"（去掉 npx -y）。

    3. memory¶

    同样通过 npm 分发。推荐 npx 运行：

    {
  "mcpServers": {
    "memory": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-memory"]
    }
  }
}

    Memory 文件存储在哪里？ - 用户级：~/.claude/memory/ - 项目级：项目 .claude/memory/

    不需要手动创建目录——memory MCP Server 首次运行时会自动初始化。

    4. node_repl¶

    node_repl 是一个 Node.js 脚本，需要 npm 全局安装：

    npm install -g @anthropic/mcp-server-node-repl

    配置 mcp.json： 

    {
  "mcpServers": {
    "node_repl": {
      "type": "stdio",
      "command": "node",
      "args": ["-e", "require('@anthropic/mcp-server-node-repl').start()"]
    }
  }
}

    ⚠️ 注意：node_repl 不能用 npx（因为它需要 node -e 方式启动）。必须先 npm install -g。

    验证安装： 

    node -e "require('@anthropic/mcp-server-node-repl')"
# 无报错 = 安装成功

    5. sequential-thinking¶

    npm 分发，推荐 npx：

    {
  "mcpServers": {
    "sequential-thinking": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-sequential-thinking"]
    }
  }
}

    零配置——首次使用时自动下载。

    6. ide¶

    IDE MCP 通常由 Claude Code 的 VS Code/JetBrains 扩展自动管理——不需要手动安装。只要你在 VS Code 或 JetBrains 中安装了 Claude Code 扩展，IDE MCP 会自动注册。

    验证：在 Claude Code 终端中问 列出当前可用的 MCP 工具，看是否有 mcp__ide__* 开头的工具。如果没有 → 确认 IDE 扩展已安装且项目已在 IDE 中打开。

    安装后：一键验证所有 MCP¶

    安装并配置完毕后，重启 Claude Code，然后问：

    列出当前可用的 MCP 工具，按 MCP Server 分组

    AI 会列出所有已连接的 MCP 及其工具。你应该看到类似这样的输出：

    codegraph:
  - codegraph_search
  - codegraph_context
  - codegraph_trace
  - codegraph_callers
  - codegraph_impact

context7:
  - resolve-library-id
  - query-docs

memory:
  - add_observations
  - search_nodes

node_repl:
  - js
  - js_add_node_module_dir

sequential-thinking:
  - sequentialthinking

    如果某个 MCP 没出现在列表中 → 跳到下面的 常见安装卡点。

    常见安装卡点¶

    现象 原因 解决方案
    npx 命令不存在 Node.js 未安装 安装 Node.js 18+：nodejs.org
    npm install -g 权限不足 Windows 需要管理员权限 / macOS 需要 sudo Windows：用管理员 PowerShell。macOS：sudo npm install -g 或配置 npm prefix
    npx 下载超时 内网环境连不上 npm registry 配置 npm 镜像：npm config set registry https://registry.npmmirror.com
    codegraph 命令不存在 安装脚本没加到 PATH 重启终端——安装脚本可能加了 PATH 但当前终端没刷新
    codegraph init 报错 项目目录权限问题 确认你对项目目录有写权限——codegraph init 需要创建 .codegraph/ 目录
    node -e "require(...)" 报 MODULE_NOT_FOUND npm 全局包没装在 Node 能找到的路径 npm list -g --depth=0 检查是否已安装。如果已安装但找不到 → 检查 NODE_PATH 环境变量
    MCP 配置正确但仍然连不上 配置文件 JSON 格式错误 用 python -m json.tool ~/.claude/mcp.json 验证 JSON 格式。常见错误：多余逗号、缺少引号

    6 个 MCP 详解¶

    1. codegraph —— 代码知识图谱¶

    解决什么问题：在大型代码库中快速定位函数、查调用关系、评估改动影响。

    为什么需要它：grep 搜函数名会返回大量噪音（注释、字符串、相似命名）。codegraph 解析了代码结构，知道每个符号的精确位置和关系。

    什么时候自动触发：AI 需要查代码结构时自动调用——不需要你手动指定。

    实际对话示例：排查 I2C 驱动 bug 的完整过程。

    你：调试时发现 I2C1 的 EV 中断有时候不触发，帮我排查一下。

AI（自动调用 codegraph_search）：
  搜索 "I2C1_EV_IRQHandler" → 找到在 i2c_driver.c:234

AI（自动调用 codegraph_context）：
  分析 I2C1_EV_IRQHandler 的上下文 → 发现它调用了 I2C_ReadByte 和 I2C_ErrorHandler

AI（自动调用 codegraph_trace）：
  从 I2C1_EV_IRQHandler 追溯完整调用路径：
  I2C1_EV_IRQHandler [i2c_driver.c:234]
    → I2C_ReadByte [i2c_driver.c:156]
    → BufferPush [ringbuf.c:89]
    → ApplicationCallback [main.c:312]

AI：看到调用链了。问题可能在三个地方：
  1. I2C_ReadByte 中的超时判断（i2c_driver.c:178）
  2. ringbuf 满了导致丢数据（ringbuf.c:89）
  3. 中断优先级配置（stm32f4xx_it.c:45）

你：查一下还有哪些地方会修改 I2C1 的 CR2 寄存器。
AI（自动调用 codegraph_impact）：
  修改 I2C1->CR2 的位置：
  - i2c_driver.c:112 (I2C_Init 中设置)
  - i2c_driver.c:201 (I2C_ErrorHandler 中清除中断标志)
  - power_mgr.c:67 (低功耗模式中关闭外设)  ← ⚠️ 这个可能在高低温测试中被意外触发

    关键价值：AI 发现了 power_mgr.c 在低功耗模式中会修改 I2C CR2 寄存器——这是手动 grep 很难发现的跨模块副作用。

    常用查询速查：

    你想知道 对 AI 说
    某个函数在哪、干什么 "查一下 I2C_ReadByte 的实现"
    谁调用了这个函数 "哪些地方调用了 I2C_ReadByte"
    这个函数调用了谁 "I2C_ReadByte 内部调用了哪些函数"
    从 IRQ 到回调的完整路径 "从 I2C1_EV_IRQHandler 到应用层回调的完整调用链"
    改了 A 影响谁 "如果改了 I2C_ReadByte 的返回值类型，会影响哪些文件"

    2. context7 —— 实时官方文档¶

    解决什么问题：AI 的训练数据可能包含旧版 API。context7 让 AI 实时拉取最新官方文档，确保 API 签名、参数、返回值、注意事项都是最新版本。

    为什么需要它：ST 的 HAL 库从 v1.6 到 v1.8，HAL_I2C_Master_Transmit 的行为变了（超时参数单位从 ms 变成了 ticks）——训练数据可能给的是旧版。

    什么时候自动触发：当你提到具体的第三方库名（"FreeRTOS"、"STM32 HAL"、"Zephyr"）时，AI 会自动拉取对应文档。

    实际对话示例：确认 API 的正确用法。

    你：我在用 STM32F4 HAL v1.8，HAL_I2C_Master_Transmit 的超时参数单位是什么？

AI（自动调用 context7 query-docs）：
  根据 HAL v1.8 的官方文档：

  HAL_StatusTypeDef HAL_I2C_Master_Transmit(
      I2C_HandleTypeDef *hi2c,    // I2C 句柄
      uint16_t DevAddress,        // 从设备地址（7bit，左对齐）
      uint8_t *pData,             // 发送数据缓冲区
      uint16_t Size,              // 数据长度（字节）
      uint32_t Timeout            // ⚠️ 超时时间，单位：毫秒（不是 tick）
  );

  返回值：
  - HAL_OK：传输成功
  - HAL_BUSY：总线忙
  - HAL_TIMEOUT：超时（从设备未应答）

你：DevAddress 是 7bit 左对齐——如果我的器件地址是 0x50，要传 0xA0？

AI（继续查 context7）：
  正确。HAL 要求地址左对齐一位：
    7bit 地址 0x50 → HAL 参数传 (0x50 << 1) = 0xA0
  这是 HAL 的设计约定——因为 I2C 协议的地址字段是 7bit+1bit R/W，
  HAL 把移位操作留给你来做。

  另注意：HAL v1.8 修复了一个 bug：
    在 v1.7 中，如果 Timeout = 0 会被当作"不超时"而不是"立即返回"。
    v1.8 修正了这个行为——Timeout = 0 现在会立即返回 HAL_TIMEOUT。

    关键价值：AI 不仅回答了 API 签名，还指出了 v1.7 到 v1.8 的 breaking change——如果你用的是 v1.7 的项目正准备升级到 v1.8，这个信息至关重要。

    什么时候 context7 不会触发： - 你用的库不在 context7 的索引范围内（小众/私有库） - 内网环境无法访问外部 API - 此时 AI 会正常回答（基于训练数据），但准确性会降低——搭配 source-driven-development skill 查本地文档

    3. node_repl —— Node.js 运行时¶

    解决什么问题：AI 需要一个"计算器"来验证数据、计算寄存器值、解析日志、做统计分析。

    实际对话示例：计算定时器参数。

    你：用 TIM2 生成 1kHz 的 PWM，时钟 84MHz，帮我算 PSC 和 ARR。

AI（自动调用 node_repl js）：
  // 计算公式：PWM频率 = 时钟频率 / ((PSC + 1) * (ARR + 1))
  // 要求 1kHz = 84MHz / ((PSC + 1) * (ARR + 1))
  // 需要选择 PSC 使 ARR 在合理范围（0~65535 for 16-bit timer）

  方案 1：PSC = 83, ARR = 999
    → 84MHz / (84 * 1000) = 1kHz ✓
    → ARR = 999（16-bit 范围，分辨率足够）

  方案 2：PSC = 839, ARR = 99  
    → 84MHz / (840 * 100) = 1kHz ✓
    → ARR = 99（分辨率低，PWM 占空比只有 100 级）

  推荐方案 1，因为 ARR = 999 提供了 1000 级占空比分辨率。

你：验证一下方案 1 的实际频率误差。
AI（自动调用 node_repl）：
  f = 84000000 / ((83 + 1) * (999 + 1)) = 84000000 / 84000 = 1000.0000 Hz
  误差 = 0%（整数除尽）

你：再帮我算一下如果时钟是内部 HSI 16MHz 而不是外部 HSE 84MHz。
AI（自动调用 node_repl）：
  f = 16000000 / ((83 + 1) * (999 + 1)) = 16000000 / 84000 ≈ 190.48 Hz
  ❌ 偏离目标 1kHz 很大——HSI 下需要重新计算 PSC/ARR

    关键价值：AI 不仅算出了参数，还主动做了灵敏度分析——HSI 变化对 PWM 频率的影响。这是人工手算容易忽略的。

    4. memory —— 持久化知识图谱¶

    解决什么问题：每次新会话 AI 都是"白纸一张"。memory 让 AI 自动记录和回忆项目知识。

    和 CLAUDE.md 的区别：

    CLAUDE.md memory
    创建者 你手动写 AI 自动创建
    内容类型 规则约束（"必须这样"） 动态知识（"上次发现..."）
    更新频率 项目结构变化时 每次重要讨论后
    示例 "这个项目用 STM32F407，HAL v1.8，FreeRTOS" "2024-03 排查发现：I2C 上拉电阻从 4.7k 换成 2.2k 解决了 400kHz 噪声"

    实际对话示例：

    你（会话 1）：分析一下为什么 I2C 在 400kHz Fast Mode 下数据偶尔出错。

AI：经过排查，根因是总线上拉电阻 4.7kΩ 太大——
    400kHz 下信号上升时间不够。建议换成 2.2kΩ。

---（几天后，新会话）---

你（会话 2）：I2C 又出问题了，帮我看看。
AI（自动从 memory 检索）：
  根据之前记录的 memory：
    - I2C 上拉电阻已从 4.7kΩ 改为 2.2kΩ
    - 之前的问题（上升时间不够）已解决

  这次的问题可能是新的。请描述当前现象。

    关键价值：AI 自动回忆了上次的修复历史，避免了重复排查同一个根因。

    5. sequential-thinking —— 结构化多步推理¶

    解决什么问题：复杂决策（架构选型、多因素权衡）需要分步推理、动态调整思路——一次性输出容易遗漏关键因素。

    实际对话示例：RTOS 任务优先级设计。

    你：帮我设计这个项目的 FreeRTOS 任务优先级。有三个任务：
    1. 传感器采集（SPI，100Hz，不能丢数据）
    2. I2C 通信（处理外部 EEPROM 读写）
    3. 串口日志输出

AI（自动使用 sequential-thinking）：
  Step 1 - 分析实时性需求：
    传感器采集 100Hz = 10ms 周期 → 最高优先级任务
    I2C EEPROM → 偶尔触发，每次 5-10ms → 中等优先级
    串口日志 → 纯输出，允许延迟 → 最低优先级

  Step 2 - 检查冲突：
    传感器采集用 SPI1，I2C 用 I2C1 → 无外设冲突 ✓
    但两个任务都用了 ringbuf → 需要互斥保护

  Step 3 - 优先级赋值：
    传感器采集：优先级 3（最高）
    I2C 通信：优先级 2
    串口日志：优先级 1（最低，idle 时运行）

  Step 4 - 验证：
    传感器采集 10ms 周期 ≈ 100Hz → 如果 I2C 任务占用超过 10ms，
    传感器任务会丢数据。建议 I2C 任务拆分：大块读写用 DMA + 回调，
    不在任务中阻塞等待。

  Step 5 - 修正方案：
    传感器采集：优先级 3，10ms 周期，使用 SPI DMA
    I2C 通信：优先级 2，DMA 模式，不在 ISR 中处理
    串口日志：优先级 1，环形缓冲，满了就丢旧数据

    关键价值：AI 在推理过程中发现了 I2C 任务可能阻塞传感器任务的隐患（Step 4），并主动修正了方案。如果没有分步推理，这个隐患可能被忽略。

    6. ide —— IDE 诊断集成¶

    解决什么问题：AI 能直接读取 IDE 的编译错误和警告——不用你手动复制粘贴。

    实际对话示例：

    你：我刚改了 i2c_driver.c，编译一下看看有没有错误。

AI（自动调用 ide getDiagnostics）：
  i2c_driver.c:178 - Error: 'HAL_I2C_STATE_READY' undeclared
    → 缺少头文件 #include "stm32f4xx_hal.h"
  i2c_driver.c:201 - Warning: variable 'timeout_val' set but not used

  修复建议：
  1. 在 i2c_driver.c 顶部加上 #include "stm32f4xx_hal.h"
  2. 删除未使用的 timeout_val 变量（或在超时逻辑中使用它）

    培训演示场景¶

    演示 1：codegraph 排查调用链（3 分钟）¶

    讲师操作（屏幕共享）：
1. 打开一个包含多个文件的嵌入式项目
2. 问："用 codegraph 从 main 函数查到 HAL_I2C_Init 的完整调用路径"
3. AI 展示 codegraph_trace 的结果 → 讲师指着调用链解释：
   "看到了吗——main → I2C_Config → HAL_I2C_Init，中间经过了 3 层。
   如果你用 grep 查，可能需要手动追溯 10 分钟"
4. 问："改了 I2C_Config 会影响哪些文件？"
5. AI 展示 codegraph_impact → 讲师指出一个意料之外的依赖

    演示 2：context7 对比旧版 vs 新版 API（2 分钟）¶

    讲师操作：
1. 问："HAL_I2C_Master_Transmit 在不同 HAL 版本中的超时参数单位有什么变化？"
2. AI 通过 context7 分别拉取 v1.7 和 v1.8 的文档 → 对比差异
3. 讲师强调："这是 training data 里不会有的——版本之间的 breaking change"

    学员实操（5 分钟）¶

    在自己项目中完成以下三件事： 1. 用 codegraph 找出 main 函数的完整调用链 2. 用 context7 查一个你项目中用到的第三方 API 的最新签名 3. 问 AI "当前有哪些 MCP 工具可用" → 确认所有 MCP 都正常工作

    常见卡点¶

    现象 自救方案
    codegraph 查不到符号 ① 确认项目已被索引（有 .codegraph/ 目录）② 确认在项目根目录启动 Claude Code ③ 如果仍不行，联系管理员运行 codegraph init
    context7 查不到某个库 context7 索引范围有限——不包含小众/私有库。此时改用 source-driven-development skill 查本地文档
    context7 返回的 API 和实际不一致 确认你用的库版本和 context7 索引的版本是否一致——在查询时明确版本号："查 STM32 HAL v1.8 的..."
    memory 记了但不回忆 memory 靠语义检索——你的问题要和记录时的用词相关。太宽泛的问题（"有什么要注意的？"）不如精确问题（"I2C 总线有什么已知问题？"）
    node_repl 计算超时 复杂计算默认 30s 超时——拆分步骤或先在小数据上验证逻辑
    sequential-thinking 推理太慢 推理步骤超过 ~10 步时会变慢——对 AI 说"压缩到 5 步以内，只保留关键决策"
    公司内网环境 MCP 无法连接外网 codegraph、memory、node_repl、sequential-thinking 都是本地运行，不需要外网。只有 context7 需要外网——内网环境用 source-driven-development 替代
