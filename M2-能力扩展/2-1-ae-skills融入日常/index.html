    ae-skills 融入日常¶

    核心价值¶

    Before：你每天用 Claude Code 聊天式提问——"帮我看看这个 bug"、"审查一下这段代码"。每次 AI 都是从零理解你的问题，输出质量不稳定。你花很多时间描述上下文。

    After：你在正确的时机触发正确的 skill——AI 按标准化流程推进，输出结构化的结果。你从"操作 AI"变成"审查 AI 的产出"。每天省下 1-2 小时的重复性脑力劳动。

    类比：裸用 Claude Code 像用示波器的手动模式（每次都要调时间基准、电压档位、触发条件）。触发 skill 像用示波器的自动设置（AutoSet）——一键让仪器自己配好参数，你只需要看波形、判断对错。

    全景位置¶

    这是 AE 工程师的日常使用手册。如果你是 SE，可以扫一眼了解 ae-skills 的存在，重点看 2-2。

    AE 工程师的一天¶

    以 AE 工程师的真实工作日为时间线，标注每个环节可以触发什么 skill。

    09:00 早上来 — 看昨晚的 CI 结果¶

    场景：打开电脑，看昨晚自动构建和测试的结果。有一个 I2C 驱动的测试用例挂了。

    触发：/ae-skills:debugging-and-error-recovery

    为什么：你有一个明确的失败用例。debugging skill 会按标准化流程推进——复现描述 → 搜索相关代码 → 定位根因 → 生成修复 → 验证——你不需要手动组织这些步骤。

    不触发 skill：你手动 grep I2C，找到 50 个文件，一个个看 while 循环，在中断处理函数里找问题……30 分钟过去了。

    触发 skill：AI 自动搜索 I2C 相关代码 → 列出所有等待硬件状态位的循环 → 标注哪些缺超时/NACK 检查 → 给出修复建议。你审查修复，5 分钟确认。

    10:00 — 同事提交了 PR，需要你审查¶

    场景：同事发了一个 HAL 驱动模块的 PR，300 行新增代码。你需要审查。

    触发：/ae-skills:code-review-and-quality

    为什么：code-review skill 从五个维度（正确性/可读性/架构/安全/性能）系统审查，每个发现标注 file:line + 维度 + 修复建议。比人眼逐行扫全面得多。

    不触发 skill：你逐行读代码，重点看逻辑对不对。你注意到变量命名不太规范、有一个边界条件可能漏了，但没发现中断优先级配置有个潜在的反转风险。审查花了 20 分钟。

    触发 skill：AI 5 分钟输出一份结构化的审查报告——正确性问题（2 个，含中断优先级风险）、可读性问题（3 个）、安全建议（1 个）。你拿着报告和同事讨论，聚焦在真正有风险的点。

    💡 你的角色：skill 不是替你下结论。它帮你找出你可能漏掉的问题。你仍然需要判断——"这个建议对吗？需要改吗？"——但你现在是在做决策，不是在做搜索。

    11:00 — PM 提了一个模糊需求¶

    场景：PM 在群里说"这次 OTA 升级功能要支持断点续传"。你不知道这到底是什么意思——断在什么粒度？续传怎么校验？异常怎么处理？

    触发：/ae-skills:interview-me

    为什么：interview-me skill 会像有经验的面试官一样，通过一问一答帮你把模糊需求逐步澄清。它不会直接给你方案，而是用问题逼你把"你想做什么"说清楚。

    不触发 skill：你按自己的理解开始写设计文档，花了一下午。第二天和 PM 对齐时发现你理解错了"断点"的粒度（你以为是固件 block 级，PM 想要的是字节级）。重写设计文档又花了一上午。

    触发 skill：interview-me 问了你 4 个问题——断点粒度？校验方式？存储介质？异常回滚？——每个问题逼你把模糊地带澄清。15 分钟后你有了一个清晰的 one-pager，发给 PM 确认，一次过。

    14:00 — 拿到澄清后的需求，开始设计¶

    场景：OTA 断点续传的需求明确了。你需要做概要设计，决定模块划分、接口、升级流程。

    触发：/ae-skills:spec-driven-development

    为什么：spec-driven skill 引导你先写规格——验收条件、边界约束、不做什么——再开始实现。避免"边写边想，最后发现方向错了"。

    不触发 skill：你开始写代码，写到一半发现 Flash 分区的设计与升级流程冲突，重构了一半代码。下午 5 点你还在改分区逻辑。

    触发 skill：AI 先引导你定义规格——写入粒度、校验算法（CRC32）、异常恢复的 3 种路径、Not Doing（不支持压缩、不支持差分升级）。规格写了 30 分钟，PM 和技术负责人各看一眼确认，然后开始写代码。实现阶段没有返工。

    14:30 — 开始实现¶

    场景：规格确认了，开始写 OTA 模块代码。这是一个中大型模块，预计 500+ 行。

    触发：/ae-skills:incremental-implementation

    为什么：incremental skill 按薄垂直切片实现——先做一个最小可工作的版本（Slice 1：能接收一个 block 并写入 Flash），每个切片编译通过 + 测试通过，再做下一个切片（Slice 2：支持多 block + 断点记录）。

    不触发 skill：你把所有函数框架搭好，然后一个个填充。填充到第三个函数时发现接口设计不合理——前面的函数都要改签名。改完后编译，8 个编译错误分布在 4 个文件里。

    触发 skill：Slice 1（Flash 写入 + 校验）→ 编译通过 → 测试通过 → Slice 2（多 block 传输 + 断点记录）→ 编译通过 → 测试通过 → Slice 3（异常恢复）→ 完成。每个切片只要 15-20 分钟，每个节点你都知道"当前状态是对的"。

    15:30 — 写完第一个版本，需要写测试¶

    场景：OTA 模块的基本功能实现了。你需要写单元测试验证 Flash 写入和校验逻辑。

    触发：/ae-skills:test-driven-development

    为什么：TDD skill 帮你先写失败测试，再让代码通过。对于 OTA 这种"写错了可能变砖"的模块，测试覆盖率是安全底线。

    不触发 skill：你手动写了几个测试用例——正常写入、写入后校验。没覆盖到：写入中途断电的 Flash 状态、Flash 写保护扇区的错误返回、多 block 的地址不连续场景。这些边缘情况在产线跑了 3 个月后才暴露。

    触发 skill：AI 帮你识别测试边界——正常路径（写入→校验→通过）、异常路径（写入失败、校验失败、Flash 写保护）、边界条件（block size 为 0、地址对齐、扇区边界）。你审查测试用例列表，补充了一个"Flash 寿命末期写入变慢"的场景。测试覆盖率从一开始的 40% 升到 85%。

    16:30 — 实现完成后，需要自查质量¶

    场景：OTA 模块的代码和测试都好了。在提交 PR 之前，你想自己做一轮审查——确认没有遗漏的错误处理、没有并发问题（OTA 过程中如果有其他中断打断了 Flash 写入怎么办）。

    触发：/ae-skills:doubt-driven-development

    为什么：doubt skill 用"对抗视角"审查你的代码——它专门找问题，不找优点。在你已经"觉得自己写完了"的时候，这是最后一道防线。

    不触发 skill：你觉得代码写得不错，测试也过了，提交 PR。Code review 时同事指出：OTA 写入过程中如果发生 SysTick 中断，Flash 写入时序可能被打断。这个问题你的测试没覆盖到。

    触发 skill：doubt 审查指出 3 个问题——Flash 写入期间应关全局中断（或使用临界区保护）、异常恢复后没有重新初始化 DMA 通道、固件版本号比较逻辑在版本回退场景下有 bug。你在提交 PR 前修复了这 3 个问题。

    17:00 — 提交 PR，更新文档¶

    场景：代码准备好了。提交 PR，更新模块文档和 CHANGELOG。

    触发：/ae-skills:git-workflow-and-versioning + /ae-skills:documentation-and-adrs

    为什么：git-workflow skill 帮你拆成原子提交（每个 commit 做一件事），写清晰的 commit message。documentation skill 帮你记录设计决策——"为什么选 CRC32 而不是 SHA256"（Flash 空间限制）。

    效率对比¶

    环节 不触发 skill 触发 skill 省时
    调试 bug 手动 grep + 逐行分析，~30min AI 搜索 + 定位 + 建议，~5min 审查 ~25min
    代码审查 逐行阅读 300 行，~20min AI 五轴审查报告，~5min 审查 ~15min
    需求澄清 自己理解 → 写文档 → 对齐失败 → 重写，~4h interview-me 15min 澄清 → PM 一次确认 节省返工
    模块实现 全量写完再编译，返工多 增量 3-4 切片，每片 15-20min 减少返工 >50%
    测试覆盖 手动写几个 happy path 用例 AI 识别边界 + 异常 + 边界条件 覆盖率 ↑40%+
    发布前自查 自信提交 doubt 审查找到 3 个隐藏问题 避免回归 bug

    总结：不是每个环节都必须触发 skill。但高频环节（调试/审查/测试）+ 高风险环节（设计/发布前自查）触发 skill 的投入回报比最高。
