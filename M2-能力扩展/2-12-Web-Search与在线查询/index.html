    Web Search & Fetch — 在线查询¶

    核心价值¶

    没有 Web Search 有 Web Search
    排查 I2C 时序问题 → 切到浏览器 → 搜 STM32 参考手册 → 找到 PDF → 翻到 I2C 章节 → 复制时序图 → 回到终端 → 粘贴给 AI 直接在终端："搜索 STM32F407 I2C 时序参数，特别是 SCL 最小高电平时间" → AI 搜索并引用来源
    AI 说某个 API 的用法 → 你怀疑是旧版本 → 切浏览器验证 → 打断心流 "用 WebSearch 确认 STM32 HAL I2C_Master_Transmit 在 HAL 1.8 中的签名" → 实时验证
    选型："STM32F4 还是 ESP32 更适合这个低功耗项目？" → 需要手动收集资料对比 "搜索比较 STM32F407 和 ESP32-S3 的低功耗模式功耗数据" → AI 搜索+对比+引用

    一句话：Web Search 让你在对话中直接查询芯片 datasheet、errata、参考设计——不用切窗口打断心流。

    全景位置¶

    context7（查库 API 签名）        WebSearch（查一切在线信息）
       ↓                                    ↓
   已知库的 API 文档               datasheet / errata / 论坛 / 技术文章

    区别： - context7：结构化查询——已知库的 API 签名，快速精确 - WebSearch：开放式搜索——芯片参数、应用笔记、选型对比、网上讨论

    怎么触发¶

    基础用法¶

    搜索一下 STM32F407 的 I2C 时序参数

    帮我查一下 AT24C02 EEPROM 的最大页写时间

    深度研究模式¶

    当需要多源交叉验证时，用 /deep-research skill：

    deep-research: STM32F4 vs ESP32-S3 低功耗方案选型
               对比 deep sleep 功耗、唤醒时间、外设可用性

    这个模式会：搜索多个来源 → 交叉验证 → 输出带引用的报告。

    拉取具体文档¶

    帮我把这个页面的内容拉下来：
https://www.st.com/resource/en/reference_manual/rm0090-stm32f4.pdf

    ⚠️ PDF 限制：单次最多拉取 20 页。对于大型参考手册，先搜索再指定页码范围。

    在 CLAUDE.md 中配置自动触发¶

    ## Web Search 规则
- 任何涉及 "选型" 或 "对比" 的问题 → 自动搜索
- 引用芯片参数（频率、功耗、电压）→ 搜索确认后再回答
- 使用不熟悉的第三方库 → 搜索确认 API 签名

    做对了的标准¶

    成功 失败
    AI 给出的参数带来源引用——"根据 STM32F407 参考手册 RM0090 第 28.4.3 节..." AI 给了一个数字但没有来源——你不知道是训练数据还是刚搜的
    搜索结果与问题直接相关——返回的是芯片 datasheet 段落，不是通用介绍 搜索结果是一般性的文章——没有芯片特有的信息
    你用 /deep-research 做选型对比 → 得到了多源交叉验证的带引用报告 只搜了一个来源——结论可能偏颇

    常见误区¶

    误区 为什么错 正确做法
    "AI 训练数据里有 datasheet，不需要搜索" 训练数据可能过时、不完整、或者是错误的二手信息 涉及具体参数（时序、电压、电流）时，始终搜索确认
    "搜出来的结果是官网的就一定对" 官网文档也有版本差异——旧版 errata 可能已被修正 注意搜索结果的文档版本号和发布日���——优先用最新版
    "让 AI 自己判断要不要搜" AI 有时会"自信地编造"——不搜也能给出看起来正确的回答 配置 CLAUDE.md 规则，强制特定场景搜索
    "内网环境不能用" WebSearch 需要外网——但内网也可以搜一部分缓存内容 确认网络环境后再决定是否依赖此功能

    培训演示场景¶

    演示：查芯片参数¶

    讲师操作（屏幕共享，2 分钟）：

    1. "搜索 STM32F407 I2C 在 Fast Mode 下的最大 SCL 频率和最小高电平保持时间"
2. AI 返回搜索结果，带来源引用
3. 追问："这个参数和实际测试的值对不上——可能是什么原因？"
4. AI 搜索 STM32F407 I2C errata → 发现一个已知的硬件限制

    学员实操（3 分钟）：搜一个自己项目中用到的芯片参数，确认来源可靠。

    演示：选型对比¶

    "deep-research: 我这个电池供电的数据采集器（STM32F103），
 要不要升级到 STM32L4？对比功耗、唤醒时间、外设兼容性"

    常见卡点¶

    现象 自救方案
    "内网环境，搜索不了" 如果大部分工作在内网，提前用 WebFetch 下载常用 datasheet 的 PDF → 存到项目目录 → AI 可以直接读本地文件
    搜索结果不相关（搜到 Arduino 教程而不是芯片文档） 搜索词要包含精确型号：不是 "I2C 超时"，而是 "STM32F407 I2C timeout erratum"
    /deep-research 耗时太长 deep-research 会搜索多个源——如果只是查一个参数，用普通 WebSearch，不要用 deep-research
    AI 搜了但不引用来源 追问："这个数据来自哪个文档？引用具体的章节号"
