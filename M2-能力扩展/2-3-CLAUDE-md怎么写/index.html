    CLAUDE.md 怎么写¶

    核心价值¶

    Before：你打开 Claude Code，每次对话都要告诉 AI——"我是 STM32F407 的项目，用 FreeRTOS，HAL 库，构建命令是 make，别用动态内存分配……"——然后 AI 还是会给你 malloc 的代码。你每次都得重复解释。

    After：项目根目录放一个 CLAUDE.md，AI 每次启动自动读取。你再也不用解释项目背景——AI 一上来就知道芯片型号、RTOS、构建命令、代码规范、不能用什么。AI 的输出从"通用代码"变成"你的项目的代码"。

    类比：CLAUDE.md 是给 AI 的项目数据手册（datasheet）。就像你写代码前要先看芯片参考手册——AI 也需要知道你的项目的"规格参数"。没有 CLAUDE.md 的 AI 像不看 datasheet 就写寄存器操作的工程师——大概率会用错 API、写错地址、违反约束。

    效果对比（真实案例——来自 1-6 的 I2C NACK 调试）：

    维度 没有 CLAUDE.md 有 CLAUDE.md
    AI 使用的 API 可能发明 I2C_ResetBus()（不存在的函数） 只用 HAL 库函数（HAL_I2C_DeInit/HAL_I2C_Init）
    中断上下文 可能建议在 ISR 里加 delay_us() 知道 ISR 不放阻塞延迟，用非阻塞计数器
    寄存器操作 可能写 I2C1->SR1 = 0（错误清零） 写 I2C1->SR1 &= ~I2C_SR1_AF（正确按位清除）
    构建命令 可能用 cmake（你的项目用 make） 直接用 make

    全景位置¶

    skill 全景 (2-0) ──→ 怎么触发 skill (2-1/2-2) ──→ CLAUDE.md (2-3) ──→ 自定义 skill (2-4)
                                                      ↑
                                                 你现在在这里

    CLAUDE.md 是所有后续高级功能（自定义 skill、MCP、Hooks、并行代理）的基础——它决定了 AI 对项目的理解深度。先写好 CLAUDE.md，再配置其他功能。

    怎么触发¶

    CLAUDE.md 是自动加载的——你不需要手动触发。AI 启动时自动查找项目根目录下的 CLAUDE.md，读入上下文。但写完 CLAUDE.md 后你最好主动说一句：

    这是我项目的 CLAUDE.md，以后请按这里的约束工作

    让 AI 明确确认它已读取并理解了你的规则。

    详细说明¶

    CLAUDE.md 放哪里¶

    你的项目/
├── CLAUDE.md        ← 放在这里（项目根目录）
├── src/
├── inc/
└── Makefile

    AI 从当前工作目录往上查找 CLAUDE.md（和 .claude/ 目录一样的行为）。一个项目一个 CLAUDE.md。

    模板：嵌入式项目的 CLAUDE.md¶

    以下是可复制的模板。每个段落上面有 <!-- 说明 --> 解释为什么需要它（在真实的 CLAUDE.md 中删掉这些注释——AI 不需要读元说明）。

    # CLAUDE.md

This file provides guidance to Claude Code when working with code in this repository.

## Project overview

<!-- 告诉 AI：芯片型号、RTOS、库、外设 —— 一次说清硬件平台 -->
- MCU: STM32F407VGT6 (Cortex-M4, 168MHz, 1MB Flash, 192KB SRAM)
- RTOS: FreeRTOS v10.4.3
- HAL: STM32F4 HAL Driver (stm32f4xx_hal)
- Toolchain: arm-none-eabi-gcc 10.3
- Debugger: J-Link / ST-Link v2

## Build commands

<!-- 让 AI 知道怎么编译、烧录、测试。精确到命令。 -->
```bash
# 编译
make clean && make -j8

# 烧录
make flash

# 单元测试
make test

# 静态分析
make lint

    Repository structure¶

    ├── src/              ← 业务逻辑 + 驱动模块
├── inc/              ← 公共头文件
├── drivers/          ← HAL 库 + CMSIS
├── freertos/         ← FreeRTOS 内核 + 配置
├── tests/            ← 单元测试（Unity + CMock）
├── tools/            ← 构建脚本 + 烧录工具
└── Makefile

    Coding conventions¶

    Naming¶

    • 函数: ModuleName_FunctionName（PascalCase）
    • 变量: snake_case
    • 宏/常量: UPPER_SNAKE_CASE
    • 全局变量: g_ 前缀（如 g_i2c1_error）
    • 静态变量: s_ 前缀

    ISR rules (MANDATORY)¶

    • 中断处理函数 禁止 使用阻塞延迟（delay_us、HAL_Delay）
    • 中断处理函数 禁止 使用 printf（阻塞 + 栈溢出风险）
    • 中断处理函数应 < 50 行，长逻辑提取到独立函数
    • 超时场景使用非阻塞计数器，不是忙等循环

    Register access¶

    • 清除寄存器标志位用 REG &= ~FLAG，不要写 REG = 0
    • 只使用 HAL 库中的真实 API，不要发明不存在的函数名
    • 外设寄存器地址通过 HAL 宏访问，不硬编码地址

    Hardware constraints¶

    • Flash: 可用 768KB（前 256KB 保留给 bootloader）
    • SRAM: 可用 160KB（系统栈 8KB, FreeRTOS 任务栈另计）
    • 堆: 不使用动态内存分配（malloc/free 禁用）——所有内存静态分配
    • I2C1: 用于 EEPROM（地址 0xA0）
    • USART2: 调试日志输出（115200 bps）
    • 看门狗: 独立看门狗 IWDG（超时 4 秒）

    Prohibited patterns¶

    • ❌ 不要在 ISR 中使用阻塞操作
    • ❌ 不要使用动态内存分配
    • ❌ 不要发明不存在的 HAL API
    • ❌ 不要使用递归（栈溢出风险）
    • ❌ 不要硬编码外设地址 
      > ⚠️ **注意**：上面的 `<!-- 说明 -->` 注释是给你看的。在你真实的 CLAUDE.md 中删掉它们——AI 不需要"元说明"，只需要规则本身。

### 逐段说明

以下是每个段落为什么重要——帮你在给自己项目写 CLAUDE.md 时知道哪些该填、哪些可以跳过。

#### Project overview（必填）

这是 CLADE.md 最重要的段落。AI 的很多错误来自"不知道你用什么芯片/RTOS/库"。比如：
- 不知道你用 FreeRTOS → 可能建议裸机方案
- 不知道芯片 Flash 大小 → 可能生成超出容量的代码
- 不知道 HAL 版本 → 可能使用不存在的函数

**最少要写**：MCU 型号 + RTOS（或裸机） + HAL 库 + 工具链。

#### Build commands（必填）

每次 AI 帮你写代码后说"编译验证一下"——它需要知道你的构建命令。额外可以加：
- 烧录命令（`make flash`）
- 测试命令（`make test`）
- 格式化命令（`make format`）

#### Repository structure（推荐）

有了目录结构，AI 用 Glob/Grep 搜索时会更精准——它知道去哪里找什么。

#### Coding conventions（推荐）

如果你有代码规范文档，提取关键部分放在这里。最关键的：
- **命名规则**——AI 生成的变量名/函数名会符合你的规范
- **ISR 规则**——这是最危险的部分，必须明确标注

#### Hardware constraints（推荐）

内存限制、外设分配、特殊约束。没有这部分，AI 可能建议使用你已经分配给其他功能的外设。

#### Prohibited patterns（必填）

这相当于"AI 的红线"。AI 可能不知道嵌入式项目的隐含约束——你必须明确说出来。

### 嵌入式项目 CLAUDE.md 示例（完整项目）

以下是一个更完整的示例——假设你的项目叫 "SmartSensor"，是一个基于 STM32L4 的低功耗传感器节点。

```markdown
# CLAUDE.md

This file provides guidance to Claude Code when working with code in this repository.

## Project overview

SmartSensor — 低功耗无线传感器节点。
- MCU: STM32L475VGT6 (Cortex-M4, 80MHz, 1MB Flash, 128KB SRAM)
- RTOS: Bare-metal (SuperLoop 架构)
- HAL: STM32L4 HAL Driver (stm32l4xx_hal)
- 无线: LoRa SX1278 (SPI2)
- 传感器: BMI160 IMU (I2C1, 地址 0x68)
- 存储: SPI Flash W25Q64 (SPI1)
- 工具链: arm-none-eabi-gcc 12.2

## Build commands

```bash
# Debug 构建
make clean && make -j8 DEBUG=1

# Release 构建（优化 size）
make clean && make -j8 DEBUG=0

# 烧录到 ST-Link
make flash

# 查看串口日志
make monitor

    Repository structure¶

    ├── app/              ← 应用层（传感器采集、LoRa 上报、电源管理）
├── bsp/              ← 板级支持包（外设驱动、引脚配置）
├── lib/              ← 第三方库（LoRa MAC、传感器驱动、环缓冲区）
├── config/           ← 项目配置头文件
├── tests/            ← 单元测试（Unity test framework）
├── scripts/          ← 辅助脚本（功耗分析、数据解析）
├── startu
└── Makefile

    Power management (CRITICAL)¶

    • 常态: STOP 2 mode（功耗 < 5µA）
    • 唤醒源: RTC 闹钟（每 10 分钟）或 GPIO 中断（传感器触发）
    • STOP 模式下：HSE/HIS 关闭，SRAM 保持，GPIO 保持
    • 唤醒后：恢复 HSE → 恢复外设时钟 → 采集 → 发送 → 进入 STOP
    • 不要在 STOP 进入前遗漏外设 DeInit——未关闭的外设会增加待机电流

    Coding conventions¶

    • 函数: bsp_ / app_ 前缀（如 bsp_i2c_write）
    • 变量: snake_case
    • 宏: UPPER_SNAKE_CASE
    • ISR: 只设置标志位，不处理逻辑（逻辑在主循环中处理）
    • 功耗敏感代码: 避免不必要的 GPIO 翻转和 printf

    Memory constraints¶

    • Flash: 最大 800KB（200KB 保留给 OTA）
    • SRAM: 最大 100KB（28KB 保留给栈 + 日志缓冲）
    • 不使用 heap（无 malloc）

    Prohibited patterns¶

    • ❌ ISR 中不放任何 > 5 行的逻辑
    • ❌ 不使用动态内存
    • ❌ 不使用 printf 在 ISR 中
    • ❌ 不阻塞主循环超过 1ms
    • ❌ 不进 STOP 模式前不关闭外设 ```

    常见问题¶

    Q: CLAUDE.md 能写多长？

    建议 50-150 行。太短（< 30 行）AI 缺少关键上下文；太长（> 300 行）会挤占对话上下文窗口。保持精炼——只写 AI 需要知道但不应该每次对话重复解释的东西。

    Q: 公司有多个项目，每个都要 CLAUDE.md 吗？

    每个项目根目录放一个。如果项目共享平台（如都是 STM32F4 + FreeRTOS），可以维护一个"基础模板"然后每项目微调。

    Q: CLAUDE.md 和 .claude/settings.json 有什么区别？

    CLAUDE.md 是给 AI 读的项目背景知识（用自然语言）。settings.json 是给 Claude Code 的配置（权限、环境变量、hook 定义——用 JSON）。两者互补：CLAUDE.md 告诉 AI 你的项目是什么，settings.json 告诉 Claude Code 允许做什么。

    Q: AI 会严格遵守 CLAUDE.md 吗？

    大部分时候会，但不是 100%。AI 可能在某些情况下偏离约束（特别是在长对话后上下文被压缩时）。验证 AI 的输出是否遵守 CLAUDE.md 是你的责任——这就是为什么你需要看懂 AI 生成的代码，而不是直接信任。
