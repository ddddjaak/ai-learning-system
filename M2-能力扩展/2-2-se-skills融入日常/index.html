    se-skills 融入日常¶

    核心价值¶

    Before：SE 面对客户需求，从需求文档到架构设计到审查追溯，每个环节都是手写——写需求文档、画架构图、写审查报告、手工维护追溯矩阵。一周有 3 天在写文档。

    After：每个 SE 工作流阶段有对应的 skill——需求分解用 requirements-decompose、架构设计用 architecture-design、审查用 design-review、追溯用 traceability-matrix。你从"写文档"变成"审查生成结果 + 填入领域判断"。

    类比：裸写 SE 文档像用手画 PCB 布线（每根线都要手动拉）。触发 se-skills 像用自动布线器（auto-router）——工具完成 80% 的布线工作，你审查并调整剩下的 20% 关键路径。

    全景位置¶

    ae-skills 26个 ──→ 2-1 AE 工程师的日常使用

se-skills 16个 ──→ 这篇告诉 SE "你的工作流怎么串起来"

    如果你不是 SE，扫一眼知道 SE 也有对应的 skill 体系即可，重点看 2-1 和 2-3。SE 路径在 M5 中深度展开。

    16 个 SE Skill 全景¶

    se-skills 包含 16 个 skill + 1 个 meta-skill，按 Define → Design → Document → Verify → Validate 五个阶段组织。每个 skill 是一个结构化工作流（不是散文），有步骤、检查点和退出标准。

    全部 Skill 列表¶

    阶段 Skill 一句话
    Define requirements-decompose 原始输入 → 结构化系统需求（6 步：收集→分类→解决冲突→派生→归属→验证）
    Design architecture-design 系统级模块分解、跨域接口、约束冲突分析、ADR
    Design software-architecture-design 固件线程模型、IPC 设计、内存预算、数据流
    Design hardware-architecture-design 引脚分配、电源域、PCB 约束、信号完整性
    Document spec-authoring 架构+需求 → SOD、HW-SW IF Spec、Test Plan
    Document software-detailed-design 函数签名、状态机、数据结构、错误处理矩阵
    Document hardware-detailed-design 原理图约束、PCB 规则、PDN 设计、热分析
    Document algorithm-design 信号处理、控制回路、标定程序、滤波器设计
    Verify design-review 四视角（HW/SW/Test/System）对抗式审查
    Verify requirements-review 需求文档清单审查——完整性、一致性、可测试性
    Verify code-static-review 编码规范审查（命名/注释/MISRA-C 合规）
    Verify test-plan-review 测试方案完整性审查——覆盖所有需求？可执行？
    Verify test-report-review 测试报告审查——通过率、覆盖率、未解决阻断项
    Verify release-review 发布包就绪审查——版本一致性、已知问题、回滚方案
    Validate traceability-matrix 需求→设计→测试追溯、覆盖率缺口检测
    Meta using-se-skills 管道引导器——自动检测当前阶段，路由到正确的 skill

    两种工作模式¶

    se-skills 支持 Pipeline Mode（引导式） 和 Goal Mode（自主式） 两种模式：

    Pipeline Mode Goal Mode
    怎么触发 说"帮我做需求分解" 说"端到端走完全流程" 或 /se-goal
    阶段切换 AI 展示 2-4 个选项，你选 AI 自动决定下一步
    验证失败 报告给你 AI 自查自修（最多重试 3 次）
    跟你互动 每阶段都问 只在真卡住时或完成时
    适用场景 你想一步步把控 你信任流程，想省时间

    Pipeline Mode 是默认模式——你说"帮我做架构设计"，AI 检测你当前在哪个阶段（检查 docs/ 目录），展示选项让你选。

    Goal Mode 是全自动模式——你给一个目标（如"完成温度传感器方案的 SE 全流程"），AI 自动跑完 Define→Design→Document→Verify→Validate，自查自修，只在真卡住时才找你。

    6 个 Slash 命令入口¶

    命令 路由到 模式
    /se-goal 全自动 SE 流程 自主式
    /se-requirements requirements-decompose 引导式
    /se-architecture architecture-design → SW/HW 架构 引导式
    /se-spec spec-authoring → 详细设计 引导式
    /se-review 6 种审查 skill 之一 引导式
    /se-traceability traceability-matrix 引导式

    5 个 Agent 角色¶

    se-skills 包含 5 个预配置的审查角色，用于 design-review 的多视角对抗审查：

    Agent 角色 视角
    system-architect 资深系统架构师 系统一致性、约束满足、跨域集成
    hw-domain-expert 硬件域专家 引脚分配、电源域、信号完整性
    fw-domain-expert 固件域专家 驱动接口、RTOS 集成、内存映射
    verification-engineer 验证工程师 可测试性、覆盖率、追溯完整性
    compliance-reviewer 合规审查员 法规合规、功能安全、行业标准

    5 个角色可以通过 /se-review 并行扇出，对同一份产物进行五视角对抗审查，合并生成分类问题报告。

    SE 的典型工作流¶

    SE 的工作按"需求输入 → 架构设计 → 规格撰写 → 多阶段审查 → 追溯验证"的流程组织。每个阶段有对应的 skill。

    客户需求 ──→ Define ──→ Design ──→ Document ──→ Verify ──→ Validate
              (需求)     (架构)     (规格)      (审查)      (追溯)

    注意这不是一次性的瀑布流——一个需求变更可能触发多轮循环。skill 的价值就在于让每一轮的产出标准化、可追溯。

    阶段 1：Define（需求定义）——"客户到底要什么"¶

    场景：客户发来一份 20 页的需求文档，包含功能需求、性能指标、接口约束。你需要把它分解为可管理、可验证的需求项。

    触发：/se-requirements 或说"帮我做需求分解"

    不触发 skill：你逐段读客户文档，手动提取需求。写完一份需求文档花了 2 天。评审时同事发现 3 个需求有歧义、2 个需求不可测试、1 个需求和另一个需求冲突。返工又花了半天。

    触发 skill： - AI 按 6 步流程（收集→分类→解决冲突→派生→归属→验证）自动处理 - 每条需求有唯一 ID（REQ-XXX）、优先级、验收条件、归属（HW/SW/System） - 标注歧义项、不可测试项、冲突项 - 你拿着结构化输出去和客户/PM 对齐，而不是从零开始

    阶段 2：Design（架构设计）——"怎么满足这些需求"¶

    场景：需求冻结了。你需要设计系统架构，然后分解到软件架构、硬件架构。

    触发：/se-architecture 或说"帮我做架构设计"

    三个专业方向：

    Skill 输入 输出
    architecture-design 系统需求 模块划分 + 接口定义 + 数据流 + ADR
    software-architecture-design 系统架构 固件线程模型 + IPC + 内存预算
    hardware-architecture-design 系统架构 引脚分配 + 电源域 + PCB 约束

    不触发 skill：你打开 Visio 画架构图。设计过程在脑子里，没留下结构化记录。3 个月后有人问"为什么选这个 MCU"，你只记得"当时觉得合适"。

    触发 skill： - 系统架构 → 软件架构 / 硬件架构逐层分解 - 每个设计决策自动生成 ADR（Architecture Decision Record） - 接口定义精确到信号级（引脚、电压、时序）

    阶段 3：Document（规格与详细设计）——"把设计写成正式文档"¶

    场景：架构确认了。你需要写正式规格文档（SOD、HW-SW IF Spec、Test Plan）和各模块的详细设计。

    触发：/se-spec 或说"帮我写规格"

    四个方向：

    Skill 输入 输出
    spec-authoring 架构 + 需求 SOD + HW-SW IF Spec + Test Plan
    software-detailed-design SW 架构 函数签名 + 状态机 + 错误处理矩阵
    hardware-detailed-design HW 架构 原理图约束 + PDN 设计 + 热分析
    algorithm-design 算法需求 信号处理 + 控制回路 + 滤波器设计

    不触发 skill：你打开 Word 从零开始写规格。写了 3 天，评审时发现"接口定义那章全是 TBD"。

    触发 skill： - AI 根据架构文档自动生成规格框架，你填充领域判断 - 每个 TBD 必须有 owner + due date——裸 TBD 是验证失败 - 详细设计精确到函数签名、数据结构、状态转换图

    阶段 4：Verify（审查）——"设计是对的吗"¶

    场景：规格和详细设计完成了。需要做正式审查——不同视角检查不同的问题。

    六种审查 skill：

    Skill 审查对象 审查角度
    design-review 架构/规格文档 HW/SW/Test/System 四视角对抗
    requirements-review 需求文档 完整性、一致性、可测试性
    code-static-review 源代码 编码规范（MISRA-C、命名、注释）
    test-plan-review 测试方案 覆盖率、可执行性、合规性
    test-report-review 测试报告 通过率、覆盖率、未解决阻断项
    release-review 发布包 版本一致性、已知问题、回滚方案

    不触发 skill：召集 4 个人开评审会。每人提前翻了翻文档，开会时靠经验和记忆提问题。开了 2 小时，记录了一页会议纪要。一周后发现"需求 3.2 和架构图里的模块 B 对不上"——开会时没人注意到。

    触发 skill： - design-review 从 5 个 agent 角色自动审查，输出结构化审查报告 - 每个发现标注对应需求编号 + 设计章节 + 严重程度 + 修复建议 - 评审会从"逐章翻文档找问题"变成"讨论 AI 审查报告中的发现"

    阶段 5：Validate（追溯）——"从需求到测试能不能连起来"¶

    场景：设计完成、实现开始、测试准备中。每个需求都有对应的设计模块吗？每个设计模块都有测试覆盖吗？

    触发：/se-traceability 或说"帮我做追溯"

    不触发 skill：你打开 Excel，维护需求→设计→测试的追溯矩阵。两周后发现 5 个"幽灵链接"（需求改了编号但矩阵没更新）和 3 个"孤儿测试"（测试用例找不到对应需求）。

    触发 skill： - AI 自动扫描需求文档 + 设计文档 + 测试计划 → 生成追溯矩阵 - 标注缺失链接（需求无对应设计 / 设计无对应测试） - 输出覆盖率报告，标注覆盖缺口 - 每个产物产出后立即验证——晚期发现的缺口修复成本呈指数增长

    效率对比¶

    SE 阶段 不触发 skill 触发 skill 关键差异
    需求分解 手动提取 + 手写需求条目，~2 天 AI 6 步流程自动生成，~2 小时 从"写"变成"审查"
    架构设计 画图 + 设计在脑子里，无结构化记录 AI 生成架构方案 + 自动 ADR 设计决策可追溯
    规格撰写 从零写 Word，~3 天，大量 TBD AI 生成框架 + 你填领域判断，~1 天 无裸 TBD
    设计审查 4 人开会 2 小时，靠经验和记忆 AI 5 视角自动审查 → 开会讨论发现 不遗漏视角
    追溯矩阵 手工 Excel，易出错，难维护 AI 自动生成 + 更新，实时覆盖率 从"维护"变成"审查"
    发布审查 靠 checklist 逐条核对，容易漏 AI 自动检查 + 标注未通过项 全面性 + 一致性
