    并行代理¶

    核心价值¶

    Before：你让 AI 审查一个 4 文件、800 行的 PR。AI 逐文件审查——每个文件 3 分钟，总共 12 分钟。你在旁边等着。

    After：AI 同时派 4 个子代理——每个代理审查一个文件，3 分钟后 4 个审查报告同时返回。一个代理负责合并报告，汇总给你。总耗时 4 分钟（3 分钟审查 + 1 分钟合并），而不是 12 分钟。

    类比：单个 AI 是单核 MCU（一次做一件事）。并行代理是多核 MCU 或 DMA（多个任务同时进行）。不是所有任务都需要并行——但审查、搜索、多维度分析这类"独立子任务"天生适合。

    全景位置¶

    Hooks (2-6) ──→ 并行代理 (2-7) ──→ 编程方式运行 (2-8)
                    ↑
               你现在在这里

    Hooks 让 AI 自动触发。并行代理让多个 AI 同时工作。两者结合——Hooks 在构建后自动触发多个并行审查代理——就是一个基础的 CI 检查流水线。

    三种并行模式¶

    模式 工作方式 适合场景 类比
    Subagent（子代理） 主 AI 派一个独立代理做一件事，等它完成 "帮我在这个文件里找一个 bug" 派一个人去查一个模块
    Agent Team（代理团队） 多个代理同时工作，各自独立，结果汇总 "同时审查 4 个文件的代码质量" 4 个人各审一个模块，组长汇总
    Workflow（工作流） 多阶段流水线——每个代理的输出是下一个的输入 "搜索 → 审查 → 验证 → 修复" 工厂流水线——每道工序产出交给下一道

    模式 1：Subagent（子代理）¶

    最简单的并行方式——主 AI 发现一个子任务可以独立完成时，自动派一个子代理去做，自己继续处理其他事。

    你不需要手动触发¶

    当你对 AI 说"审查这个 PR"，AI 可能会自动决定：

    主 AI: PR 改了 4 个文件。让我同时审查——
  → 子代理 1: 审查 src/i2c_eeprom.c（120 行）
  → 子代理 2: 审查 src/i2c_eeprom.h（35 行）
  → 子代理 3: 审查 tests/test_i2c.c（200 行）
  → 子代理 4: 审查 src/main.c（50 行）← 改动很小

4 个审查报告同时返回
  → 主 AI: 合并报告，去重，输出统一审查报告

    你也可以主动要求¶

    帮我同时检查 i2c_eeprom.c、spi_flash.c、uart_debug.c 三个文件里有没有阻塞的中断处理

    模式 2：Workflow（工作流）¶

    多阶段流水线——这是最强大的并行模式。一个 Workflow 有多个阶段（phase），每个阶段可以有多个并行代理。

    嵌入式场景：代码审查工作流¶

    场景：PR 改了 4 个文件，需要从 5 个维度审查 + 验证发现的每个问题。

    阶段 1: Review（并行审查）
  → 审查维度 1: 正确性（并发问题、竞态条件、边界条件）
  → 审查维度 2: 安全性（缓冲区溢出、注入风险、权限检查）
  → 审查维度 3: 性能（ISR 耗时、内存使用、Flash 寿命）
  → 审查维度 4: 可读性（命名、注释、函数长度）
  → 审查维度 5: 架构（模块耦合、接口设计、依赖方向）
  5 个代理同时审查，~3 分钟全部完成

阶段 2: Verify（逐个验证）
  对阶段 1 的每个发现 → 派独立验证代理确认"这真的是个问题吗" + 给修复建议
  验证和审查可以重叠——Review 维度 1 产出的发现立即进入 Verify，不等人齐

最终输出: 一份包含"确认的问题 + 修复建议 + 严重程度"的结构化审查报告

    核心概念¶

    pipeline(items, stage1, stage2, stage3)

items = [file_a, file_b, file_c, file_d]

pipeline 的执行方式:
  file_a → stage1(审查) → stage2(验证)
  file_b → stage1(审查) → stage2(验证)
  ...

关键是: file_a 在 stage2 时，file_b 可能还在 stage1
        没有跨 item 的屏障——每个 item 独立流经所有阶段
        只有跨 item 需要汇总时（如去重），才需要显式 await 做屏障

    Workflow 脚本示例（可复制）¶

    以下是一个完整的 Workflow 脚本——代码审查工作流。你可以把它保存到 .claude/workflows/code-review.js：

    export const meta = {
  name: 'code-review-workflow',
  description: 'Multi-dimensional code review with verification',
  phases: [
    { title: 'Review', detail: '5-dimension parallel review' },
    { title: 'Verify', detail: 'Adversarial verification of each finding' },
  ],
};

// 结构化输出定义 —— 确保 agent 返回可解析的数据而非自由文本
const FINDING_SCHEMA = {
  type: 'object',
  properties: {
    findings: {
      type: 'array',
      items: {
        type: 'object',
        properties: {
          file: { type: 'string' },
          description: { type: 'string' },
          severity: { type: 'string', enum: ['high', 'medium', 'low'] },
          fix: { type: 'string' },
        },
        required: ['file', 'description', 'severity'],
      },
    },
  },
  required: ['findings'],
};

const VERDICT_SCHEMA = {
  type: 'object',
  properties: { isReal: { type: 'boolean' }, reason: { type: 'string' } },
  required: ['isReal'],
};

// 从外部获取输入文件列表（通过 workflow 的 args 参数传入）
const files = args?.files || ['src/i2c_eeprom.c', 'src/spi_flash.c'];

// 阶段 1: 对每个文件做 5 维度并行审查
// 每个文件独立流经 5 维度审查，然后产出结构化发现列表
phase('Review');
const allFindings = await pipeline(
  files,
  // 阶段 1 回调: 5 维度并行审查单个文件
  async (file) => {
    const dims = ['correctness', 'security', 'performance', 'readability', 'architecture'];
    const results = await parallel(
      dims.map(dim => () =>
        agent(
          `Review ${file} from the ${dim} perspective. Find issues with file:line + severity + fix suggestion.`,
          { label: `review:${dim}:${file}`, phase: 'Review', schema: FINDING_SCHEMA }
        )
      )
    );
    // 合并 5 个维度的发现
    return results.filter(Boolean).flatMap(r => r.findings || []);
  },
  // 阶段 2 回调: 对单个文件的发现做对抗式验证（与阶段 1 重叠执行）
  async (findings, file, index) => {
    if (!findings || findings.length === 0) return [];
    const verified = await parallel(
      findings.map(f => () =>
        parallel(
          // 3 个独立验证者 —— 每人从不同角度判断
          [
            () => agent(
              `Adversarially verify this finding. Try to refute it. "${f.description}" in ${f.file}. Return {"isReal": true/false, "reason": "..."}.`,
              { label: `verify:${f.file}`, phase: 'Verify', schema: VERDICT_SCHEMA }
            ),
            () => agent(
              `Verify from a correctness perspective: "${f.description}" in ${f.file}. Is the claimed issue actually incorrect behavior? Return {"isReal": true/false, "reason": "..."}.`,
              { label: `verify2:${f.file}`, phase: 'Verify', schema: VERDICT_SCHEMA }
            ),
            () => agent(
              `Verify from a security/safety perspective: "${f.description}" in ${f.file}. Could this cause real harm? Return {"isReal": true/false, "reason": "..."}.`,
              { label: `verify3:${f.file}`, phase: 'Verify', schema: VERDICT_SCHEMA }
            ),
          ]
        ).then(votes => {
          // ≥2 个验证者确认 isReal → 确认问题
          const realCount = votes.filter(Boolean).filter(v => v.isReal).length;
          return realCount >= 2 ? { ...f, confirmed: true } : null;
        })
      )
    );
    return verified.filter(Boolean);
  }
);

// 输出: 仅返回被多方验证确认的问题，按严重程度排序
const confirmed = allFindings.flat().filter(Boolean);
confirmed.sort((a, b) => {
  const order = { high: 0, medium: 1, low: 2 };
  return order[a.severity] - order[b.severity];
});
return confirmed;

    💡 这个脚本的关键设计： - 结构化输出（schema）：每个 agent() 调用都有 schema 参数——确保 AI 返回可解析的对象（含 file、description、severity、isReal 字段），而不是自由文本。没有 schema 的 agent 返回的是字符串，后续代码无法从中提取字段。 - 对抗式验证：每个发现被 3 个独立验证者从不同角度审查（refute / correctness / security），≥2 个确认才通过——大幅降低误报。 - Pipeline 流式执行：阶段 1（审查）和阶段 2（验证）对每个文件独立流动——file_a 在验证时，file_b 还在审查。不需要等所有文件审查完才开始验证。 - 输入通过 args 传入：args.files 获取外部输入的文件列表，而非引用不存在的全局变量。

    什么时候不该用并行¶

    反例 为什么不该用 正确做法
    审查 5 行改动 并行派代理的开销 > 审查本身的时间 单个 AI 直接审查
    两个子任务有依赖（"先理解代码结构，再改代码"） 后一个依赖前一个的输出 用 Workflow 串行阶段，不要并行
    快速问答（"这个函数做什么"） 单次查询，不需要拆分 单个 AI 直接回答
    子任务共享大量上下文 每个子代理需要加载相同上下文，浪费 token 合并为一个大一点的子任务

    判断标准：如果一个子任务 ≥ 2 分钟且和其他子任务没有依赖 → 适合并行。如果 < 1 分钟或有依赖 → 不适合。
