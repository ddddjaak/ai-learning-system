    环境配置详解¶

    概述¶

    本页是 AE 工程师的 MCP + Skill 工具箱参考。当你遇到以下情况时，回来看这页：

    • 某个 skill 不生效，不知道怎么触发
    • MCP 工具调用了但没反应
    • 不确定某个场景该用什么 skill 组合
    • 新项目需要检查环境是否就绪

    类比：本页是你的工具箱清单——不是让你从头读到尾，而是螺丝刀找不到了、扳手卡住了的时候，回来查"这东西在哪"和"怎么修"。

    快速自查（30 秒）¶

    在深入排查之前，先确认这四件事：

    □ Claude Code 能正常启动           → 终端输入 claude --version
□ ae-skills 插件已安装             → 终端输入 claude skill list，看有无 ae-skills
□ codegraph 能索引你的项目         → 问 AI："用 codegraph 查一下 main 函数在哪个文件"
□ CLAUDE.md 在项目根目录           → 问 AI："这个项目的 MCU 是什么"——它应该能回答

    四项都通过？跳过排查，直接看下面的 常用工具组合 或 客户方案配置。

    MCP 插件清单¶

    已安装的 MCP¶

    MCP Server 用途 触发方式 关键工具
    codegraph 代码知识图谱——符号搜索/调用链/影响分析 问 AI "查一下 XX 函数的调用链" 时自动触发 codegraph_search, codegraph_trace, codegraph_callers, codegraph_impact
    context7 实时拉取官方库文档（STM32 HAL / FreeRTOS / Zephyr 等） 问 "XXX API 的参数是什么" 时触发 resolve-library-id, query-docs
    node_repl Node.js 运行时——数据处理、寄存器值计算 需要计算/脚本验证时手动触发 js
    memory 持久化知识图谱——跨会话记住项目决策 AI 自动记录关键决策和规范 add_observations, search_nodes
    sequential-thinking 结构化多步推理 架构选型、复杂 bug 分析时自动或手动触发 sequentialthinking
    ide IDE 诊断信息（编译错误/警告） 代码有编译错误时自动获取 getDiagnostics

    各 MCP 常见卡点¶

    MCP 现象 自救方案
    codegraph AI 说"codegraph 不可用"或查不到符号 ① 确认项目根目录有 .codegraph/ ② 在项目根目录启动 Claude Code ③ 如果仍然不行，项目可能未被索引——联系管理员运行 codegraph init
    context7 AI 给的 API 签名和实际对不上 ① context7 需要外网——确认网络能访问 https://context7.com ② 部分芯片厂商的库不在 context7 索引范围内——此时改用 source-driven-development skill 查本地文档
    node_repl js 调用超时 ① 复杂计算默认 30s 超时——如果数据量大，拆分计算步骤 ② 检查是否有死循环——先在小数据上验证逻辑
    memory AI 不记得上次的决策 ① memory 记录的是结构化知识节点，不是聊天记录——确保关键决策用了明确陈述（"记录一下：XX 项目用 FreeRTOS 而不是裸机，因为…"）② 问 AI "回忆一下这个项目的 memory 中有哪些关键决策"
    sequential-thinking 推理过程被截断或结果不完整 ① 问题太复杂时，拆成多个子问题分步推理 ② 总步数超过限制时，先做关键分支再深入
    ide 看不到编译错误 ① 确认 IDE 已打开且项目已加载 ② Claude Code 需要和 IDE 在同一工作区

    MCP 通用排查¶

    如果所有 MCP 都不生效（比单个 MCP 失效更常见）：

    步骤 检查项 操作
    1 配置文件是否正确 检查 .claude/mcp.json —— server 名称和配置格式是否正确
    2 MCP server 进程是否启动 重启 Claude Code，观察启动日志中的 MCP 连接状态
    3 工具名是否带正确前缀 完整格式: mcp__<server>__<tool>（如 mcp__codegraph__codegraph_search）
    4 权限是否被拦截 检查 .claude/settings.json 中是否有 deny 规则阻止了 MCP 调用

    💡 终极自助排查：对 AI 说 列出当前可用的 MCP 工具——如果某个 MCP 的工具不在列表中，说明连接失败。

    Skill 不生效的排查步骤¶

    现象 自救方案
    AI 说"找不到这个 skill" ① 检查 ~/.claude/skills/ 或项目 .claude/skills/ 目录——skill 是否真的安装了 ② skill 名称对大小写敏感——确认拼写 ③ 重启 Claude Code（skill 在启动时加载）
    AI 没有自动触发某个 skill ① 显式触发："请使用 XX skill" ② 检查 CLAUDE.md 中是否有规则屏蔽了该 skill ③ 部分 skill 只在特定场景自动触发（如 source-driven 需要引用第三方 API 时）
    Skill 触发了但行为不符合预期 ① 检查 CLAUDE.md 中的项目规则是否与 skill 默认行为冲突 ② 在触发时明确指定约束条件 ③ 用 doubt-driven 审查 AI 的输出
    ae-skills 全部不生效 ① 确认插件已安装：claude skill list 查看是否有 ae-skills 条目 ② 检查 .claude/settings.json 中 enabledPlugins 是否包含 ae-skills ③ 重新安装插件

    常用工具组合¶

    以下组合覆盖 AE 工程师最常见的场景。每个组合按工作流顺序排列，后面跟建议的 MCP 配合。

    场景 Skill 链 MCP 配合 预计耗时
    客户问题 → 理解本质 interview-me — 10-20 分钟
    理解问题 → 可选方案 interview-me → idea-refine sequential-thinking 30-45 分钟
    方案方向 → 可验收规格 idea-refine → spec sequential-thinking 45-60 分钟
    新功能开发（含代码） interview-me → spec → plan → build → test → doubt codegraph + context7 1-3 小时
    修 bug debugging → review → doubt codegraph（查调用链） 30-90 分钟
    HardFault / 硬件异常 embedded-debugging → debugging codegraph 1-4 小时
    外设驱动开发 peripheral-driver-design → spec → build → test → review codegraph + context7 2-6 小时
    RTOS 设计 rtos-and-concurrency → spec → build → review codegraph 2-4 小时
    代码审查 review → doubt（对 HIGH 发现做对抗验证） codegraph（查影响面） 15-30 分钟
    接手陌生代码 context-engineering → codegraph 探索 → review codegraph（核心） 30-60 分钟
    规格 + 方案 → 客户文档 spec → documentation-and-adrs — 20-30 分钟
    完整客户方案（端到端） interview-me → idea-refine → spec → documentation → shipping sequential-thinking 2-4 小时
    构建系统配置 embedded-build-and-toolchain codegraph 30-60 分钟

    环境就绪检查¶

    培训前或新项目开始前，逐项确认：

    □ Claude Code 能正常启动
   → 验证：终端输入 claude --version，输出版本号即可

□ CC Switch 已配置 API Key
   → 验证：问 AI "你好，你是什么模型"——能正常回复即可
   → 注意：切换供应商后需要新终端窗口才生效

□ codegraph 已配置并能索引你的项目
   → 验证：问 AI "用 codegraph 查一下 main 函数在哪个文件"
   → 卡点：项目根目录下需要 .codegraph/ 目录（由管理员初始化）

□ CLAUDE.md 已在项目根目录
   → 验证：问 AI "这个项目的 MCU 是什么"——它应该能从 CLAUDE.md 中回答
   → 如果回答不出：检查 CLAUDE.md 是否在项目根目录、是否包含硬件信息

□ ae-skills 插件已安装（V1.0+）
   → 验证：claude skill list | grep ae-skills

□ context7 可用（需要外网）
   → 验证：问 AI "用 context7 查一下 STM32 HAL_I2C_Init 的签名"

□ 至少一个嵌入式项目可用于练习
   → M3 所有实操场景需要真实项目，不是 demo 项目

    客户方案交付补充配置¶

    以下内容适用于客户方案交付场景——在通用开发环境基础上，需要额外的 skill 和配置用于生成方案文档和交付检查。

    客户方案交付 Skill¶

    Skill 用途 触发方式 对应页面
    documentation-and-adrs 生成方案文档 + 架构决策记录 "基于规格生成方案文档 + ADR" 3-12
    shipping-and-launch 交付前清单检查 "帮我做方案交付前的 shipping 检查" 3-13

    CLAUDE.md 建议¶

    在项目根目录加入客户方案场景的特定配置：

    ## 客户方案工作流程约束
- 客户描述问题 → 必须先 interview-me 澄清，再进入方案方向
- 方案方向 > 2 个时必须做 idea-refine 对比
- 方案确认后必须有 spec（含验收条件），再生成文档
- 交付前必须走 shipping 检查清单

## 文档要求
- 所有客户文档使用中文
- 技术术语首次出现时需加注中文解释
- 方案文档必须包含：概述（客户可读） + 技术方案 + 集成说明
- 关键技术决策必须有 ADR 记录

    常见卡点汇总¶

    以下是在培训中收集的最高频卡点（5 个以上），每个都有具体的自救命令。

    # 现象 自救方案
    1 Claude Code 启动后无法对话 检查 CC Switch 是否在系统托盘运行（不是关掉了）→ 确认 API Key 已配置 → 新开一个终端窗口（CC Switch 的配置只对新终端生效）
    2 AI 频繁编造不存在的 HAL 函数 CLAUDE.md 中缺少硬件平台信息 → 在 CLAUDE.md 中明确写出 MCU 型号、HAL 库版本、参考手册编号 → 用 context-engineering skill 系统性配置
    3 codegraph 查不到符号 项目未索引或不在项目根目录启动 Claude Code → 确认 pwd 在项目根目录 → 确认有 .codegraph/ 目录
    4 skill 明明装了但 AI 说找不到 skill 名称大小写敏感 → 用 claude skill list 确认确切名称 → 用别名（如 review 代替 code-review-and-quality）
    5 interview-me 自动触发后只问了一个问题就出结论 这是最常见的误用——AI 给了总结但没深入 → 追问："继续问，我还没说清楚" → 或显式触发："请用 interview-me skill 来 interview 我"
    6 context7 给了错误的 API 签名 部分芯片厂商的 HAL 库在 context7 上的版本可能过时 → 用 source-driven-development skill 查本地文档或官网
    7 用了 CC Switch 切换供应商后 API 调用失败 CC Switch 的配置只对新终端生效 → 关闭当前终端 → 新开一个 → 重试
