    incremental-implementation¶

    这个 skill 做什么¶

    incremental-implementation 是一种拆分策略——把一个完整功能拆成薄垂直切片，每一片都是"编译通过 + 测试通过 + 可演示"的最小可用增量。不是先写完所有底层、再写所有上层（水平分层），而是每个切片从上到下走通一个完整的路径。

    类比：水平分层是"先把所有墙砌好，再铺所有电线，再刷所有漆"——直到最后一步之前，房子都不能住人。薄垂直切片是"先装好一间卧室的全部（墙+电+漆），这间卧室今晚就能住"——每一步都有可用的产出。

    什么时候用 / 什么时候不该用¶

    ✅ 应该用¶

    场景 为什么
    实现一个新模块（3+ 个函数、跨文件） 薄切片让你在每个切片完成时验证——不会在最后才发现底层假设错了
    接手一个不熟悉的代码库 小步快跑降低风险——每步改动小，出了问题容易定位
    复杂逻辑（协议栈、状态机） 分步验证每层逻辑——而不是写完 500 行再发现状态转换有 bug
    和硬件打交道的新驱动 每个切片验证一个硬件行为——不会假设硬件一定按预期工作

    ❌ 不该用¶

    场景 为什么不该用 替代方案
    改单行代码或加单个函数 不需要拆分 直接改
    纯配置变更（改 CubeMX 重新生成） 不是增量实现——是自动生成 生成后对照 diff 检查
    重构（不改外部行为） 重构有自己的技能 code-simplification

    怎么触发¶

    触发方式¶

    用增量方式实现 EEPROM 驱动模块。先拆成 3-4 个薄切片，每个切片编译通过 + 测试通过后再进下一片。

    薄切片 vs 水平分层¶

    水平分层（❌ 反例）：

    Slice 1: 写完所有头文件声明（i2c_eeprom.h 完整 API）
Slice 2: 写完所有实现（i2c_eeprom.c 全部函数体）
Slice 3: 写完所有测试
Slice 4: 集成调试 → 发现底层 I2C 初始化假设错了，大量返工

    薄垂直切片（✅ 推荐）：

    Slice 1: 能检测 EEPROM 是否存在（IsDeviceReady）
  → eeprom.h:     只声明 IsDeviceReady()
  → eeprom.c:     只实现 IsDeviceReady()
  → test_eeprom.c: 只测试 IsDeviceReady()
  → 编译通过 → 测试通过 → 在硬件上验证 I2C 能通信 ✅

Slice 2: 能读单字节（ReadByte）
  → eeprom.h:     加 ReadByte() 声明
  → eeprom.c:     加 ReadByte() 实现（复用 Slice 1 的 I2C 句柄）
  → test_eeprom.c: 加 ReadByte() 测试
  → 编译通过 → 测试通过 → 硬件上读一下 EEPROM 的已知值验证 ✅

Slice 3: 能写单字节 + 回读验证（WriteByte + ReadByte 组合）
  → eeprom.h:     加 WriteByte() 声明
  → eeprom.c:     加 WriteByte() 实现
  → test_eeprom.c: 加 WriteByte→ReadByte 回环测试
  → 编译通过 → 测试通过 → 硬件上写读验证 ✅

Slice 4: 批量读写 + 错误处理（BurstRead/BurstWrite + 超时+重试+错误恢复）
  → eeprom.h:     加 BurstRead/BurstWrite 声明
  → eeprom.c:     加实现（含错误恢复逻辑）
  → test_eeprom.c: 加边界测试（超大数据、跨页、错误注入）
  → 编译通过 → 测试通过 → 硬件上边界测试 ✅

    每个切片产出的是一个可用的功能，不只是"代码写了一部分"。

    嵌入式实例：I2C EEPROM 驱动¶

    以下是 AI 在 incremental-implementation 模式下的完整工作流：

    Slice 1: 设备检测¶

    你: 增量实现 EEPROM 驱动。Slice 1：先实现设备检测功能。

AI: 好。Slice 1 只做一件事——确认 EEPROM 在 I2C 总线上存在且响应。

    [生成 bsp/eeprom.h]
    typedef enum { EEPROM_OK = 0, EEPROM_ERROR_NO_RESPONSE } eeprom_status_t;
    eeprom_status_t EEPROM_IsDeviceReady(void);

    [生成 bsp/eeprom.c]
    eeprom_status_t EEPROM_IsDeviceReady(void) {
        if (HAL_I2C_IsDeviceReady(&hi2c1, EEPROM_I2C_ADDR, 3, 100) != HAL_OK)
            return EEPROM_ERROR_NO_RESPONSE;
        return EEPROM_OK;
    }

    [生成 tests/test_eeprom.c]
    void test_device_detected_when_present(void) {
        mock_hal_i2c_set_device_ready(HAL_OK); /* Mock: 设备在线 */
        TEST_ASSERT_EQUAL(EEPROM_OK, EEPROM_IsDeviceReady());
    }
    void test_device_not_detected_when_absent(void) {
        mock_hal_i2c_set_device_ready(HAL_ERROR); /* Mock: 设备不在 */
        TEST_ASSERT_EQUAL(EEPROM_ERROR_NO_RESPONSE, EEPROM_IsDeviceReady());
    }

    编译通过? → Yes
    测试通过? → 2 passed ✅

    Slice 1 完成。要我继续 Slice 2 吗？

    Slice 2: 单字节读¶

    你: 继续 Slice 2——单字节读。

AI: Slice 2 在 Slice 1 基础上加 ReadByte。
    [追加到 eeprom.h] eeprom_status_t EEPROM_ReadByte(uint8_t addr, uint8_t *data);
    [追加到 eeprom.c] 实现 ReadByte——用 HAL_I2C_Mem_Read, I2C_MEMADD_SIZE_8BIT
    [追加测试] 读成功/读失败/读 NULL 指针（边界）

    编译通过? → Yes
    测试通过? → 5 passed (2 old + 3 new) ✅
    已有功能未退化? → Slice 1 的两个测试仍然通过 ✅

    Slice 2 完成。

    Slice 3: 单字节写 + 回读验证¶

    你: Slice 3——单字节写。

AI: Slice 3 加 WriteByte。关键：EEPROM 写入后进入内部写入周期（tWR ≤ 5ms），
    期间 NACK 所有通信。测试策略：WriteByte 后用 IsDeviceReady 轮询确认写入完成，
    再 ReadByte 回读验证数据一致。
    [追加到 eeprom.h] eeprom_status_t EEPROM_WriteByte(uint8_t addr, uint8_t data);
    [追加到 eeprom.c] 实现 WriteByte——HAL_I2C_Mem_Write + IsDeviceReady 轮询
    [追加测试] 写读一致性 / 写入无效地址 / 写入后立即读（验证 ACK polling 有效）

    编译通过? → Yes
    测试通过? → 8 passed ✅
    Slice 1+2 测试仍通过 ✅

    Slice 4: 批量读写 + 错误恢复¶

    你: Slice 4——批量操作加错误恢复。

AI: Slice 4 加 BurstRead/BurstWrite + 错误恢复。
    [追加声明] BurstRead(addr, *buf, len), BurstWrite(addr, *buf, len)
    [追加实现] 循环调用 HAL_I2C_Mem_Read/Write——
    含错误恢复（NACK：重试+延迟 / BERR：SCL脉冲释放总线 / TIMEOUT：重试+IsDeviceReady验证）
    注意跨页写入边界：AT24C02 的 8 字节页边界内会自动回绕，
    BurstWrite 需按页边界切分写入
    [追加测试] 跨页读写 / len=0 边界 / NACK 恢复 / 超大数据

    编译通过? → Yes
    测试通过? → 14 passed ✅
    全部已有测试仍通过 ✅

    四个切片全部完成。

    怎么判断切片粒度¶

    切片太小 切片刚好 切片太大
    一个函数声明算一片
    → 太多切片，管理开销大    		 一个功能点算一片
    → 每个切片有独立可用价值    		 全部读写操作算一片
    → 失去增量验证的价值
    举例：.h 中加一个声明就交一片 举例：能读单字节（头+实现+测试） 举例：整个驱动写完才算一片

    判断标准：这个切片完成后，你能做一件独立的事吗？如果不能——切片太小。如果这个切片包含多个独立的事——切片太大。

    做对了的标准是什么¶

    ✅ 成功输出¶

    标准 说明
    每个切片编译通过 不是"代码框架写好了等最后一起编译"——每片都独立编译
    每个切片测试通过 已有测试 + 新增测试全部通过
    已有功能未退化 Slice 3 不会破坏 Slice 1 和 Slice 2 的功能
    每个切片有独立价值 如果项目在 Slice 2 完成时终止，你至少有"能读 EEPROM"的能力
    集成问题早发现 硬件假设、接口设计问题在 Slice 1/2 就被发现，而不是在 Slice 4

    ❌ 失败输出¶

    失败表现 为什么不行 怎么办
    每个切片只加了声明/注释/框架代码 这种切片没有独立可用价值——你只是在拆分工作量，没有验证任何假设 重新设计切片：第一片必须是端到端可用的功能
    最后一个切片才在硬件上测试 你失去了增量实现的核心价值——硬件假设应当最早验证 确保 Slice 1 就在硬件上（或 mock 中）验证
    Slice 4 时发现 Slice 1 的设计有问题需要大改 设计问题应该在 Slice 1/2 就被暴露——说明切片太大或验证不充分 回退到发现问题的切片，改设计后重新推进

    常见误区¶

    误区 1："增量实现就是分批提交代码"¶

    现象：你把一个大功能分成 3 次 commit——第一次提交 .h，第二次提交 .c，第三次提交测试。然后说"我用增量实现了"。

    正确做法：每个切片是端到端可用的——不是"代码量分几份"，而是"功能点分几个"。每个切片的产出是可以独立演示和验证的。

    误区 2："第一个切片应该先搭框架"¶

    现象：你想先把头文件、目录结构、构建配置全写好——"打好地基再盖房"。

    正确做法：你的"框架假设"可能错了——比如你假设用 I2C_MEMADD_SIZE_8BIT，但 EEPROM 的地址空间需要 16 位（AT24C32+ 需要 I2C_MEMADD_SIZE_16BIT）。如果 Slice 1 就做一个端到端的设备检测，你能在最早的时机用最少的代码验证你的假设。

    ⚠️ 嵌入式注意事项： - RTOS 线程安全：I2C 外设是共享资源——HAL_I2C_Mem_Read/Write 和 HAL_I2C_IsDeviceReady 不是线程安全的。在 FreeRTOS 等多任务环境中，需要用互斥锁保护所有对同一个 I2C_HandleTypeDef 的访问。 - ISR 上下文约束：阻塞 HAL I2C 函数（HAL_I2C_Mem_Read 等）不能在 ISR 中调用——会死等超时。ISR 中只能使用 _IT 或 _DMA 版本的非阻塞函数。

    进阶¶

    与 TDD 配合（天然配对）¶

    incremental-implementation 定义"切多薄"，TDD 定义"每片怎么验证"。组合使用：

    Slice N: 
  1. 用 TDD 写本片的测试（RED）
  2. 用 incremental-implementation 写本片的实现（GREEN）
  3. 全部已有测试仍通过（REFACTOR → 确认无退化）

    与 planning-and-task-breakdown 配合¶

    对于大功能，先用 planning-and-task-breakdown 拆成 3-5 个任务。然后每个任务再用 incremental-implementation 拆成 2-4 个薄切片。

    与 doubt-driven-development 配合¶

    每 2-3 个切片完成后，用 doubt-driven-development 做一次交叉审查——确认累计的设计质量。
