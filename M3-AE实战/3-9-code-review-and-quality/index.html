    code-review-and-quality¶

    这个 skill 做什么¶

    code-review-and-quality 是一个五维度代码审查流程——正确性、可读性、架构、安全性、性能。不是"AI 你帮我看看这段代码有没有问题"（那只能得到随机评论），而是一个结构化的审查：每个发现标注 file:line + 所属维度 + 严重程度 + 具体修复建议。

    类比：code-review 是你的"五合一仪表盘"（车上的转速表+油温表+胎压监测+燃油表+电池电压）。你不需要逐个检查每个子系统——一个面板同时显示五个维度的健康状况，异常项高亮标注。

    什么时候用 / 什么时候不该用¶

    ✅ 应该用¶

    场景 为什么需要 code-review skill
    自己的代码提交前自查 五维度系统化——不会遗漏安全性或性能维度（人工自查最容易漏这两项）
    同事的 MR/PR 需要你 review AI 先做第一轮审查，你只需要确认和补充——省掉 70% 的机械性检查
    接手了一个不熟悉的模块 审查报告帮你快速了解模块的代码质量和风险点
    重构前后的质量对比 重构后触发一次审查，对比重构前后的发现数量和严重程度

    ❌ 不该用¶

    场景 为什么不该用 替代方案
    改了一行注释 不需要五维度审查 直接提交
    紧急 hotfix（5 分钟内要上线） 五维度审查需要 2-5 分钟——紧急时先修再审查 修完后再触发审查作为事后检查
    需求/方案审查 code-review 审代码，不是审方案 SE 用 design-review
    只想知道"这段代码能跑吗" 你需要的是测试，不是审查 用 test-driven-development

    怎么触发¶

    触发方式¶

    审查 bsp/i2c_eeprom.c 的代码质量，用五维度审查

    或更具体的：

    我改动了 i2c_eeprom.c、i2c_eeprom.h、tests/test_i2c.c，帮我从正确性、安全性、可读性三个维度做提交前自查

    AI 收到后会按五维度自动审查，输出结构化的发现列表。

    五维度详解¶

    维度 检查什么 嵌入式典型问题
    正确性 逻辑有没有 bug？边界条件处理了吗？ ISR 中有没有阻塞延迟？volatile 是否正确使用？竞态条件？返回值被忽略？
    可读性 命名是否清晰？注释是否解释了"为什么"？ 寄存器操作有没有注释说明 bit 含义？Magic number 有没有 #define？ISR 触发源和副作用是否标注？
    架构 模块间耦合是否合理？接口是否稳定？ 驱动层和应用层是否分离？HAL 调用是否隔离在硬件抽象层（而非散布在业务逻辑中）？
    安全性 缓冲区溢出？输入验证？故障恢复？ 串口 DMA 缓冲区有无溢出保护？I2C/SPI 传输长度是否校验？外设故障后的恢复策略是否完整？Watchdog 策略是否正确？
    性能 ISR 耗时？内存使用？Flash 耐久性？ 中断处理中有没有耗时操作？频繁 Flash 写入的页擦除策略是否合理？

    审查输出示例¶

    场景：审查一个 I2C EEPROM 驱动模块 bsp/i2c_eeprom.c（约 120 行）

    AI 审查报告¶

    ## Code Review: bsp/i2c_eeprom.c + i2c_eeprom.h

### 正确性 (Correctness) — 发现 2 个问题

**[HIGH] i2c_eeprom.c:89 — I2C 错误后未完整复位外设**
- 问题：`HAL_I2C_GetError()` 返回 AF 后只清除了 `I2C_FLAG_AF`，没有复位 I2C 状态机。
       连续的 NACK 会导致外设状态累积，最终总线挂死。
- 修复：在 AF 后执行 `HAL_I2C_DeInit(&hi2c1)` + `HAL_I2C_Init(&hi2c1)` 完成软件复位。
- 参考：STM32F4 参考手册 RM0090 §27.6.5

**[MEDIUM] i2c_eeprom.c:45 — HAL_I2C_Mem_Read 返回值未检查**
- 问题：直接使用返回值传入后续逻辑，`HAL_TIMEOUT` 和 `HAL_ERROR` 被等同处理。
- 修复：区分 TIMEOUT 和 ERROR——TIMEOUT 可以重试，ERROR 需要复位后重试。

### 可读性 (Readability) — 发现 2 个问题

**[LOW] i2c_eeprom.c:30 — Magic number**
- 问题：`0x50` 没有 #define 为有意义的宏名。
- 修复：`#define EEPROM_I2C_ADDR 0x50` 并在所有使用处替换。

**[LOW] i2c_eeprom.c:78 — 注释描述了"做什么"，不是"为什么"**
- 问题：`// 等待 5ms` — 读代码的人已经看到 `HAL_Delay(5)`，不需要注释重复。
- 修复：改为 `// AT24C02 最大写入周期 5ms（datasheet §8.2.4），等待完成后再发起读操作`

### 架构 (Architecture) — 发现 1 个问题

**[MEDIUM] i2c_eeprom.c:15 — HAL 调用直接暴露在业务逻辑中**
- 问题：`EEPROM_Read()` 函数中直接调用 `HAL_I2C_Mem_Read()`——EEPROM 驱动和硬件抽象没有分层。
       如果换用 I2C2 或改用软件 I2C，需要修改所有 EEPROM 操作函数。
- 修复：增加一层 `I2C_ByteRead()` / `I2C_ByteWrite()` 抽象，EEPROM 驱动通过这层访问总线。
       当前开销量不大（只一个 I2C 设备），可以标注为 TODO 在下次重构时处理。

### 安全性 (Security) — 发现 1 个问题

**[HIGH] i2c_eeprom.c:102 — 写入长度无上限校验**
- 问题：`EEPROM_Write(addr, *data, len)` 的 `len` 参数没有校验——如果传入 300（AT24C02 只有 256 字节），
       会发生页边界回绕写入（datasheet 规定的页内地址回绕），导致数据损坏而没有任何报错。
- 修复：函数开头加 `if (addr + len > EEPROM_SIZE) return -1;`。

### 性能 (Performance) — 发现 1 个问题

**[MEDIUM] i2c_eeprom.c:67 — 每次写入后固定等待 5ms，未使用 ACK polling**
- 问题：EEPROM 写入后，HAL 提供了 `HAL_I2C_IsDeviceReady()` 用于 ACK polling——
       写完后轮询设备就绪，通常 ~1ms 就能完成，而不是固定等待最坏情况的 5ms。
- 修复：将 `HAL_Delay(5)` 替换为 `while (HAL_I2C_IsDeviceReady(&hi2c1, EEPROM_ADDR, 3, 10) != HAL_OK)`，
       设合理的 timeout（如 50 次 × 10ms = 500ms）。

---

### 总结

| 维度 | HIGH | MEDIUM | LOW | 总计 |
|------|------|--------|-----|------|
| 正确性 | 1 | 1 | 0 | 2 |
| 可读性 | 0 | 0 | 2 | 2 |
| 架构 | 0 | 1 | 0 | 1 |
| 安全性 | 1 | 0 | 0 | 1 |
| 性能 | 0 | 1 | 0 | 1 |
| **总计** | **2** | **3** | **2** | **7** |

### 建议处理顺序

1. ✅ 先修 2 个 HIGH（正确性-错误复位 + 安全性-长度校验）——这两个可能导致数据损坏
2. ⏰ 再修 3 个 MEDIUM（返回值检查、架构分层、ACK polling）——质量提升
3. 📝 LOW 的可以在后续改动时顺手修

    做对了的标准是什么¶

    ✅ 成功输出¶

    标准 说明
    每个发现可追溯 每个问题标注了 file:line + 所属维度——不是"代码质量一般"，而是"i2c_eeprom.c:89 正确性问题"
    严重程度分明 HIGH / MEDIUM / LOW 分级合理——HIGH 是"不改可能导致数据损坏/安全事故"，LOW 是"不影响功能但风格不好"
    修复建议可执行 不是"检查一下 I2C 配置"，而是"把 HAL_Delay(5) 替换为 HAL_I2C_IsDeviceReady() 轮询"
    五个维度都被检查 报告中有每个维度的发现，或明确标注"该维度无发现"
    有优先级建议 审查报告结尾给出了处理顺序——哪些必须马上修，哪些可以延后

    ❌ 失败输出¶

    失败表现 为什么不行 怎么办
    "代码写的不错，没什么大问题" 任何非 trivial 的代码都有改进空间——空报告说明 AI 没有认真审查 指定具体维度重新审查："从安全性维度重新审查，重点看缓冲区操作和输入验证"
    审查结果全是可读性问题 AI 只看了表面（命名、注释），跳过了正确性和安全性 "正确性和安全性维度重新审查——重点看竞态条件、边界检查和错误路径"
    严重程度标注混乱 LOW 标成 HIGH，或反之——用户无法判断"我真的该马上修吗" 对每个发现追问："这个问题不改的后果是什么？"——从实际后果判断严重程度

    常见误区¶

    误区 1："AI 审查过了，不需要人再看"¶

    现象：AI 审查通过，你直接合并了。但 AI 没有你的领域知识——它不知道你们团队"中断处理必须用寄存器操作不能用 HAL"的规则。

    正确做法：AI 做第一轮审查（机械性检查——API 用法、缓冲区边界、返回值检查），人做第二轮审查（领域知识——团队规范、硬件特性、历史坑位）。二者互补，不是替代。

    误区 2："审查发现太多了，全修完再提交"¶

    现象：AI 给了 15 个发现，你全部修完再提交——PR 拖了三天，和相邻模块产生了合并冲突。

    正确做法：按严重程度分级处理： - HIGH：必须本次修（可能是数据损坏/安全漏洞） - MEDIUM：本次修或开 issue 跟踪 - LOW：不阻塞本次提交，下次改动到同一文件时顺手修

    误区 3："AI 的审查维度不适合嵌入式"¶

    现象：嵌入式工程师觉得"安全性"维度在嵌入式场景下意义不大——代码跑在 MCU 上，没有用户输入。

    正确做法：嵌入式场景下，"安全性"维度有特定的含义： - 缓冲区溢出（串口 DMA 接收、I2C/SPI 传输） - 输入校验（外部传感器的数据范围、通信协议字段合法性） - 错误恢复（外设故障后的恢复策略） - Watchdog 策略（IWDG 是否正确喂狗？有没有可能死在 ISR 里导致看门狗复位？）

    你可以在 CLAUDE.md 中定义你们的"安全性"维度具体包含什么。

    进阶¶

    审查特定维度¶

    当代码有特定的风险偏好时，可以只审查部分维度：

    从正确性和安全性两个维度审查这个 ISR 实现——重点看有没有阻塞操作和缓冲区溢出

    与 CLAUDE.md 配合¶

    在 CLAUDE.md 中定义团队的审查规则，AI 会自动引用：

    ## Code Review Rules
- ISR 中禁止使用 HAL_Delay、printf、I2C/SPI 阻塞操作
- 所有外设操作必须检查返回值
- GPIO 配置必须使用 LL 库而非 HAL（团队规范）
- Magic number 必须使用 #define（寄存器 bit 位除外）

    与 doubt-driven-development 配合¶

    审查完成后，对 AI 发现的 HIGH 级问题用 doubt-driven-development 做对抗验证——确认这真的是个问题，不是误报。

    提交前自查清单¶

    可以把 code-review 加到 Hooks（M2 2-6）中，每次 git commit 前自动触发：

    {
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "if": { "command": "git commit" },
      "hooks": [{
        "type": "command",
        "command": "claude -p \"快速审查即将提交的改动，只输出 HIGH 和 MEDIUM 发现\" --tools \"Read\" \"Grep\" --dangerously-skip-permissions --max-turns 8"
      }]
    }]
  }
}
