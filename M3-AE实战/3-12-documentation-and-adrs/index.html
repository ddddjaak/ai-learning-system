    documentation-and-adrs（AE 场景）¶

    这个 skill 做什么¶

    documentation-and-adrs 是一个方案文档生成器——它把方案的方向、规格、技术决策整理成结构化的文档，包含架构决策记录（ADR: Architecture Decision Record）——记录"我们为什么做了这个技术选择，而不是另一个"。在 AE 场景下，这个 skill 的核心价值是把方案从技术细节翻译成客户和内部团队都能读的文档。

    类比：你装修完房子，装修队给你一张纸写着"改了 3 个插座、换了 2 根水管"。documentation-and-adrs 给你的是一本《房屋水电改造说明》——每根管为什么换、换成了什么、未来如果改要注意什么。

    AE 场景的特殊性：AE 的方案文档要同时满足两类读者： - 客户：关心"方案是什么、对我有什么影响、我需要注意什么" - 内部团队（研发/FAE/后续维护者）：关心"为什么做了这些选择、有哪些边界条件、未来改动要注意什么"

    什么时候用 / 什么时候不该用¶

    ✅ 应该用（AE 场景）¶

    场景 为什么需要 documentation
    方案规格已确认，需要生成正式方案文档发给客户 客户需要一份专业文档来内部流转和审批
    方案涉及非显而易见的技术决策（选了方案 A 而不选 B） ADR 记录"为什么"——半年后你自己或接手的人不会再纠结
    方案的维护周期长（超 6 个月），后续可能换人维护 文档降低知识流失风险
    客户要求方案文档需包含技术原理说明 客户的技术人员需要理解方案的技术依据

    ❌ 不该用¶

    场景 为什么不该用 替代方案
    方案极简单（改一个配置值） documentation 的成本 > 价值 在邮件里写两句话
    方案还在探索中，方向随时可能变 文档会频繁过时 先等规格确认（spec-driven）
    文档只给自己看 不需要正式结构 写个笔记或代码注释

    怎么触发¶

    触发方式¶

    方案规格已确认（降 SPI 速率方案），帮我生成方案文档。
需要包含：
- 方案概述（客户可读）
- 技术方案详细描述

- 对客户系统的集成说明
- 已知限制和未来扩展建议

# 方案文档：SPI 通信可靠性优化
## 方案编号: SOL-TEMP-2026-001 | 版本: v1.0 | 日期: 2026-06-17

---

## 1. 方案概述

### 1.1 问题简述

贵司温度记录仪在低温环境（-20°C）下运行约 1 小时后，
SPI 读取外部 ADC（ADS1118）开始出现通信失败（返回全 0xFF），
随后触发看门狗复位。常温（25°C）下无此问题。

### 1.2 根因分析

STM32L4 使用内部高速振荡器（HSI）作为系统时钟源。
HSI 在 -20°C 下的频率可能漂移（datasheet 全温度范围规格约 -2% 到 +1.5%），影响 MCU 内部对 MISO 数据的采样点位置。
ADS1118 的 SPI 时序由绝对纳秒参数定义（t_SCLK_min = 250ns），
当前 2 MHz SCLK（500ns 周期）远高于最低要求——但低温下
CMOS 驱动增强导致的信号边沿加速可能引发振铃/反射，
使从机误判时钟边沿。

### 1.3 推荐方案

将 SPI2 波特率从 2 MHz 降低至 1 MHz（修改预分频值），
使 SCLK 周期从 500ns 增加到 1000ns，降低信号边沿速率，
减少振铃和反射风险——在 -40°C 到 +105°C 全温度范围内
均可保证信号完整性。

## 2. 技术方案

### 2.1 改动范围

| 文件 | 改动 | 改动量 |
|------|------|--------|
| `spi.c` (CubeMX 生成) | hspi2.Init.BaudRatePrescaler: SPI_BAUDRATEPRESCALER_16 → SPI_BAUDRATEPRESCALER_32 | 1 行 |
| 无其他文件 | — | — |

### 2.2 对客户系统的影响

| 维度 | 变化 | 说明 |
|------|------|------|
| 采样率 | 不受 SPI 时钟影响（ADS1118 采样率由内部 DR 寄存器设定，1 MHz SPI 完全满足最高采样率的传输带宽需求） | 仍远高于需求（100 sps） |
| 功耗 | 无显著变化（SPI 预分频器改动不改变 APB1 总线时钟，外设功耗差异在纳安级别） | 1 MHz 通信功耗略低 |
| 硬件 | 无变化 | 不改 PCB、不加元件 |
| 其他外设 | 无变化 | SPI2 独立通信，不影响 I2C/UART |
| FreeRTOS 任务耗时 | SPI 任务每次传输约增加 16µs（32-bit 传输：2 MHz 下 16µs → 1 MHz 下 32µs） | 对调度无显著影响 |

## 3. 架构决策记录（ADR）

### ADR-001: 选择降速方案而非换晶振

- **状态**: Accepted
- **决策日期**: 2026-06-17
- **背景**: SPI 通信在低温下因 HSI 频率漂移失败，有两个可行方案：
  方案 A（换 HSE 晶振）和方案 B（降 SPI 速率）。
- **决策**: 采用方案 B（降 SPI 速率，2 MHz → 1 MHz）。
- **理由**:
  1. 零硬件成本、零 PCB 改动——客户产品已量产
  2. 1 MHz 速率对应用场景（温度记录仪，≥ 100 sps）完全足够
  3. 改动量极小（1 行配置变更），验证成本低
  4. 如果未来需要更高采样率，可以再升级到方案 A（HSE）
- **后果**:
  - ✅ SPI 通信在全温度范围内可靠
  - ✅ 不影响产品交付计划（1 天完成）
  - ⚠️ 如果客户未来需要 > 200 sps 采样率，需要升级方案 A
  - ⚠️ 长期来看，如果该产品系列需要高速 SPI（≥ 5 MHz），
    HSE 是必要的——建议在下一版 PCB 设计中预留 HSE 晶振布局

### ADR-002: 不采用温度补偿方案

- **状态**: Accepted
- **决策日期**: 2026-06-17
- **背景**: 考虑了利用 STM32L4 内部温度传感器 + HSITRIM 校准的方案
- **决策**: 不采用温度补偿方案
- **理由**:
  1. HSITRIM 校准步长约 0.25%（约 40kHz/step @ 16MHz），在特定 trim 码附近可能出现非单调跳变——需要每颗芯片标定
  2. 生产工序增加标定环节——不适用已量产产品
  3. 软件校准精度有限，可靠性不如直接降速或换晶振
- **后果**:
  - 如果未来有产品无法接受降速且不想加晶振，
    可重新评估此方案——但需先评估生产标定的成本

## 4. 集成说明

### 4.1 客户需要的准备

- 一台含当前固件的温度记录仪样机
- 可编程高低温箱（-20°C 到 +85°C）
- 串口日志接收工具

### 4.2 验证步骤（简要）

1. 烧录新固件（仅 SPI 分频值变更）
2. 常温（25°C）下运行，确认功能正常
3. 高低温循环测试（-20°C 保持 2h → +85°C 保持 2h → 循环 3 次）
4. 确认无 SPI 通信错误、无复位

## 5. 已知限制

1. 本方案解决的是"低温下 SPI 信号完整性问题导致的通信失败"——
   如果系统存在其他低温相关的问题（如电源、焊接），不在本方案范围内
2. 降速不影响 ADS1118 采样率（采样率由内部 DR 寄存器设定，可达 860 sps），但限制了
   未来升级到更高速 ADC 的可能性
3. 如果 SPI 总线上未来增加其他从机，需重新评估时序余量

## 6. 未来扩展建议

1. 下一版 PCB 设计：预留 HSE 晶振布局（即使不焊接），
   为未来高速 SPI 需求留后路
2. 使能 PVD（可编程电压检测器）监测 VDD 电压——如低温下供电异常导致电压跌落，PVD 可触发告警辅助诊断——
   如果未来出现非时钟原因的低温异常，可快速诊断
3. 在固件中添加 SPI 错误计数器和统计，方便远程诊断

- ADS1118 数据手册（SBAS457F）§7.6 Timing Requirements: Serial Interface
- 方案规格书 v2.0（SOL-TEMP-2026-001-spec-v2.0.md）

    AE 文档的关键元素¶

    ADR（架构决策记录）¶

    ADR 是"为什么做这个选择"的记录——比方案本身更重要：

    ### ADR-[序号]: [决策标题]

- **状态**: Proposed | Accepted | Deprecated | Superseded
- **背景**: 为什么需要做这个决策
- **决策**: 我们选择做什么
- **理由**: 为什么这个选择优于备选方案
- **后果**: 正面影响 + 负面影响 + 风险

    ADR 的价值在决策被质疑或需要反悔时体现——你可以看到当时的上下文和理由，而不是"不知道当初为什么选了这个"。

    客户可读 vs 内部详细¶

    AE 场景下，建议生成两个版本：

    版本 目标读者 内容重点
    客户版（精简） 客户项目经理、客户技术负责人 方案概述、影响、集成说明、验证方法
    内部版（完整） 内部研发、FAE、后续维护者 完整技术细节、ADR、已知限制、未来扩展

    做对了的标准是什么¶

    ✅ 成功输出¶

    标准 说明
    方案文档有完整的 6 个章节 概述 → 技术方案 → ADR → 集成说明 → 已知限制 → 未来扩展
    每个非平凡决策有 ADR "为什么选 A 不选 B"被记录——不只是"选了 A"
    ADR 包含后果（正面 + 负面） ADR 不是赞美诗——客观记录负面影响和风险
    客户可独立阅读方案概述和集成说明 不需要技术背景也能理解问题、方案、影响
    文档标注版本号和日期 未来回看时不会困惑"这是哪个版本"

    ❌ 失败输出¶

    失败表现 为什么不行 怎么办
    文档只有技术细节，没有方案概述 客户看不懂 "加一段方案概述——非技术人员能看懂的问题 + 方案 + 影响"
    ADR 只写了"选了 A"没写"为什么不选 B" ADR 的核心价值是记录为什么排除了其他选项 "对每个 ADR，补充备选方案和排除理由"
    文档没有版本信息 未来修改后无法区分 "在文档开头加版本号 + 日期"

    常见误区¶

    误区 1："ADR 要写得非常正式，每个决策都记录"¶

    现象：你给每个微小的技术选择都写了 ADR——"选了 SPI_BAUDRATEPRESCALER_32 而不是 SPI_BAUDRATEPRESCALER_64"也成了一个 ADR。文档膨胀到 30 页，没人想看。

    正确做法：ADR 只记录非平凡决策——如果两个选择之间有显著的 trade-off，或者决策在将来可能被质疑，才需要 ADR。修改一个预分频值不需要 ADR——"降速 vs 换晶振"需要 ADR。

    判断标准：如果这个决策在 6 个月后可能被新加入的同事质疑——就值得写 ADR。

    误区 2："方案文档要包含全部实现细节"¶

    现象：你把寄存器配置值、HAL 函数调用序列、测试脚本都写进方案文档——客户收到的是一份"代码仓库"，不是文档。

    正确做法：方案文档的详细程度 = "客户能据此理解方案的影响"。实现细节放在代码注释和内部开发文档中。客户不关心 SPI_BAUDRATEPRESCALER_32 的值——客户关心"这对采样率有什么影响"。

    误区 3："文档写完发给客户就完了"¶

    现象：方案文档 v1.0 发出去了。2 周后实现过程中发现了一个新约束，但你没更新文档。3 个月后客户问"为什么这行为和文档里写的不一样"。

    正确做法：文档是活的——方案实施过程中任何影响客户可见行为的变化，必须更新文档版本。在 Change Log 中记录每次变更。
