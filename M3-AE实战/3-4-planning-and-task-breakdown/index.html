    planning-and-task-breakdown¶

    这个 skill 做什么¶

    planning-and-task-breakdown 把你的中型任务拆成可执行、可验证的子任务——每个子任务有明确的依赖关系、验收条件和预估范围。拆解后的任务列表可以直接喂给 AI 逐个实现。

    类比：planning 是你的"施工图纸"。盖房子不能从"盖一栋房子"这个大任务开始——你需要一张图纸标清楚：先打地基（依赖 0）、再砌墙（依赖地基）、再走水电（依赖墙）……每个工序有明确的完成标准（"地基混凝土浇筑完毕，养护 7 天，抗压强度 ≥ 25MPa"）。planning 就是给你的代码画这种图纸。

    什么时候用 / 什么时候不该用¶

    ✅ 应该用¶

    场景 为什么
    接手一个需要 2-5 天的开发任务 值得花 30 分钟做拆解——避免做到一半发现方向错了
    任务的依赖关系不显然 显式画出依赖图能防止"先做了 B 才发现 B 依赖 A 但 A 还没做"
    多人协作（你 + AI 或你 + 同事） 拆解后的子任务可以分配——明确谁做什么、验收标准是什么
    任务需要分阶段交付 拆解成可独立交付的 checkpoints

    ❌ 不该用¶

    场景 为什么不该用 替代方案
    半天以内的小任务 拆解的时间 > 执行的时间 incremental-implementation
    紧急 bug fix 不需要计划——定位+修+验证 debugging-and-error-recovery
    纯探索性工作（"试试看这个方案行不行"） 探索的方向不确定，无法做依赖分析 idea-refine

    怎么触发¶

    触发方式¶

    帮我拆解这个任务：实现一个 I2C 命令队列模块，支持非阻塞的 I2C 多字节读写。
现有的 I2C 驱动在 bsp/i2c_eeprom.c 中，使用 HAL 库的阻塞模式。
需要改为中断模式 + 命令队列。

    AI 会输出一份包含以下要素的拆解计划： 1. 依赖图 2. 任务列表（每个任务含：描述、验收条件、预估范围、依赖项） 3. Checkpoints 4. 风险和开放问题

    拆解示例：I2C 命令队列模块¶

    依赖图¶

    Task 0: I2C 中断模式基础设施
   → 配置 I2C1 中断优先级 + 使能 NVIC
   → 实现基础的 IT 模式收发（单次）
        │
        ├──→ Task 1: 环形队列数据结构
        │      → 队列结构体 + push/pop/is_full/is_empty
        │      依赖: Task 0（需要确认 ISR 中访问队列的安全性）
        │           │
        │           ├──→ Task 2: 命令执行引擎（ISR 中逐条执行队列）
        │           │      → 中断中取队首命令 → 启动传输 → 传输完成 → 
        │           │        回调通知 → 取下一个命令
        │           │      依赖: Task 0, Task 1
        │           │           │
        │           │           ├──→ Task 3: 错误恢复 + 超时处理
        │           │           │      → NACK/TIMEOUT/BERR/ARLO 的恢复策略
        │           │           │      依赖: Task 2
        │           │           │
        │           │           └──→ Task 4: 公共 API + 集成验证
        │           │                  → i2c_queue_init / i2c_queue_push / 
        │           │                     i2c_queue_get_status
        │           │                  依赖: Task 2, Task 3
        │           │
        │           └──→ Task 5: EEPROM 驱动迁移
        │                  → 将 bsp/i2c_eeprom.c 从阻塞模式迁移到命令队列
        │                  依赖: Task 4

    任务列表¶

    ## Task 0: I2C 中断模式基础设施
**预估**: Small (1-2 文件)
**依赖**: 无

**描述**: 在现有 I2C1 初始化基础上，配置中断模式。
        实现 `HAL_I2C_Master_Transmit_IT` 和 `HAL_I2C_Master_Receive_IT` 的基本单次调用。
        验证中断能正常触发和完成。

**验收条件**:
- [ ] I2C1 事件中断和错误中断已使能，优先级设置为 4
- [ ] 单字节 IT 发送 + 接收在逻辑分析仪上波形正确
- [ ] 中断处理时间 < 50μs（GPIO toggle 测量）

**涉及文件**: Core/Src/stm32l4xx_it.c, bsp/i2c.c

---

## Task 1: 环形队列数据结构
**预估**: Small (2 文件)
**依赖**: Task 0

**描述**: 实现中断安全的环形队列——用于存储待执行的 I2C 命令。
        支持 push(命令入队)、pop(ISR 中取下一个命令)、is_full、is_empty。
        队列深度可配置（默认 8）。

**验收条件**:
- [ ] push 8 个命令后队列满，第 9 个返回 false
- [ ] pop 空队列返回 NULL
- [ ] 从 ISR 中 pop、从任务中 push——并发安全（临界区保护）
- [ ] 单元测试覆盖：满/空/半满/环回（push 超过队列尾部后绕回头部）

**涉及文件**: bsp/i2c_queue.h, bsp/i2c_queue.c, tests/test_i2c_queue.c

---

## Task 2: 命令执行引擎
**预估**: Medium (2-3 文件)
**依赖**: Task 0, Task 1

**描述**: 在 I2C 中断中执行队列中的命令。
        ISR 流程：上一个传输完成 → 调用用户回调 → 检查队列 → 
        如果有待执行命令 → 启动下一个传输。

**验收条件**:
- [ ] 单个命令入队后自动执行，完成后回调被调用且参数正确
- [ ] 连续入队 5 个命令，按 FIFO 顺序依次执行
- [ ] 执行过程中断优先级不被其他中断抢占致数据丢失（示波器验证时序）
- [ ] 无内存泄漏（静态分配，反复出入队 1000 次后内存不变）

**涉及文件**: bsp/i2c_queue.c（补充 ISR 逻辑）, Core/Src/stm32l4xx_it.c

---

## Task 3: 错误恢复 + 超时处理
**预估**: Small (1 文件)
**依赖**: Task 2

**描述**: 处理 I2C 通信错误——NACK / TIMEOUT / BERR / ARLO。
        每个错误类型有不同的恢复策略。恢复后队列继续执行下一个命令。

**验收条件**:
- [ ] NACK 后：清除标志 + DeInit + Init，3ms 内恢复，下一个命令正常执行
- [ ] TIMEOUT 后：记录超时，释放总线，下一个命令正常执行
- [ ] BERR/ARLO 后：执行总线恢复序列，记录错误
- [ ] 连续 3 次错误后：队列进入暂停状态，通知主任务（而非死循环）

**涉及文件**: bsp/i2c_queue.c

---

## Task 4: 公共 API + 集成验证
**预估**: Small (1 文件)
**依赖**: Task 2, Task 3

**描述**: 封装公共 API——`i2c_queue_init()` / `i2c_queue_push()` / 
        `i2c_queue_get_status()` / `i2c_queue_drain()`。
        集成测试验证完整链路。

**验收条件**:
- [ ] API 文档完整（每个函数的参数/返回值/调用上下文约束）
- [ ] 集成测试：EEPROM 读写 200 次无错误
- [ ] 集成测试：注入 NACK 3 次后队列暂停，drain 后恢复

**涉及文件**: bsp/i2c_queue.h（API 声明）, tests/test_i2c_queue_integration.c

---

## Task 5: EEPROM 驱动迁移
**预估**: Small (1-2 文件)
**依赖**: Task 4

**描述**: 将 `bsp/i2c_eeprom.c` 从 HAL 阻塞模式改为使用 I2C 命令队列。
        EEPROM 操作（Read/Write）提交命令到队列，通过回调获取结果。

**验收条件**:
- [ ] EEPROM_Read/EEPROM_Write 改为非阻塞版本（提交命令后立即返回）
- [ ] 原有调用方（main.c 中的 EEPROM 读写逻辑）适配新的异步接口
- [ ] EEPROM 操作在队列模式下通过所有已有测试

    Checkpoints¶

    Checkpoint 1: After Tasks 0-2
  → I2C 中断模式可工作 + 环形队列可操作 + 命令可入队执行
  → 可以用逻辑分析仪看到 I2C 波形从命令队列自动发出 ✅

Checkpoint 2: After Tasks 3-4
  → 错误恢复完整 + 公共 API 可用
  → 手动制造 NACK → 确认队列自动恢复 ✅

Checkpoint 3: After Task 5
  → EEPROM 驱动成功迁移
  → 所有已有功能在队列模式下正常 ✅

    风险和开放问题¶

    风险 影响 缓解
    I2C 中断优先级不当导致时序问题 中断被更高优先级中断延迟 > I2C 时序要求 Task 0 中用逻辑分析仪测量中断响应延迟
    队列深度 8 不够 EEPROM 大批量操作 队列满导致任务阻塞 运行时统计最大队列深度，支持配置
    中断中回调调用 FreeRTOS API 可能触发 assert 回调在 ISR 上下文 → FromISR 版本 API Task 2 中明确文档"回调在 ISR 上下文"

    做对了的标准是什么¶

    ✅ 成功输出¶

    标准 说明
    依赖图无环 不会出现"A 依赖 B、B 依赖 A"的死锁情况
    每个 task 有 ≥2 个验收条件 验收条件描述"做完的标准"，不是"做什么"
    task 粒度合理 每个 task 预估 Small-Medium（1-5 文件）
    并行机会被识别 互相没有依赖的 task 被标注为可并行（如 Task 1 和 Task 5 如果解耦可以并行）
    有 checkpoints 明确哪些 task 完成后可以做一个"阶段演示"

    ❌ 失败输出¶

    失败表现 为什么不行 怎么办
    Task 描述是"实现 XX 模块" 缺少验收条件——无法判断做完没有 给每个 task 补充 ≥2 个验收条件
    所有 task 都串行依赖 可能过度拆分——检查是否有些不必要的依赖 检查可以解耦的 task
    Task 太大了（整个命令队列实现是一个 task） 没有拆解价值——和直接做一样 拆成 4-6 个 task

    常见误区¶

    误区 1："拆解就是按文件拆分"¶

    现象：你把"I2C 命令队列"拆成了"Task 1: i2c_queue.h / Task 2: i2c_queue.c / Task 3: test_i2c_queue.c"——按文件拆，每个 task 不能独立验证。

    正确做法：按功能点拆——每个 task 是一组"做了就能验证"的功能，而不是一组文件。

    误区 2："依赖图必须完美才能开始"¶

    现象：你花 2 小时画依赖图，反复调整，迟迟不开始实现。

    正确做法：依赖图是活的——实现过程中发现的依赖可以在图中补充。初始拆解花 15-30 分钟，Task 0-1 完成后重新审视剩余 task 的依赖。

    进阶¶

    与 incremental-implementation 配合¶

    planning 拆出 Task 0-5 → 每个 Task 用 incremental-implementation 再拆成 2-3 个薄切片。两层拆解覆盖"大任务→中任务→小切片"。

    拆解作为团队沟通工具¶

    拆解产出（依赖图 + 任务列表）可以直接用作团队任务分配——标注"谁负责哪个 task"、"哪些可以并行"、"预计完成时间"。
