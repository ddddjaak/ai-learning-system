    doubt-driven-development¶

    这个 skill 做什么¶

    doubt-driven-development 是一个对抗式代码审查——它不验证你的代码"看起来对不对"，而是用全新视角试图证明你的代码有问题。它的核心动作是：提取你的代码 + 约束条件 → 派一个独立审查者（没看过你写代码的过程）→ 审查者收到对抗性提示词（"找出问题，不要验证"）→ 你作为仲裁者对发现做分类处理。

    类比：doubt-driven 是给你的代码请一个"故意找茬的同事"。正常 review 的同事会说"这里有个潜在问题，但总体不错"。doubt-driven 的审查者会说"既然是中断上下文，你确认这个 while 循环不会死锁吗？你确认这个 volatile 对编译器优化有效吗？你确认 HAL_I2C_DeInit 在 ISR 里重入安全吗？"——不是找优点，是找漏洞。

    这个 skill 的完整流程在 M2 的 2-7-并行代理 中有 Workflow 脚本示例。本章聚焦于你什么时候用、怎么用、怎么判断结果。

    什么时候用 / 什么时候不该用¶

    ✅ 应该用¶

    场景 为什么
    代码写完了、测试通过了，但你不确定有没有遗漏 测试测的是"你想到的情况"——doubt 找的是"你没想的情况"
    涉及并发/中断/多线程的代码 并发 bug 是最难在测试中发现的——doubt 的对抗视角专门找这类问题
    修复了一个 bug，想确认修复没引入新问题 doubt 审查修复代码 + 所有受影响的上游调用者
    对已有代码没信心（接手别人的代码、代码很旧） doubt 帮你快速评估代码的风险水平

    ❌ 不该用¶

    场景 为什么不该用 替代方案
    刚写完第一版草稿 doubt 是做"接近完成时的验证"——草稿阶段先让代码结构稳定 code-review-and-quality
    5 行以内的简单改动 不值得启动对抗审查 肉眼检查
    已经跑过 2 轮 doubt 且没有新发现 满足 stop condition——不需要无限循环 停止，接受当前代码

    怎么触发¶

    触发方式¶

    用 doubt-driven 审查 bsp/i2c_eeprom.c 中的错误恢复逻辑——
重点检查：在所有错误路径上恢复是否完整？恢复过程中有没有竞态条件？

    你不需要说"启动 doubt-driven skill"——你只需要指定审查范围和重点关注的方向。

    五步流程¶

    Step 1: CLAIM（声明）
  → 你声明："I2C 错误恢复逻辑在所有 NACK/TIMEOUT/BUSY 条件下都能正确恢复总线"
  → AI 把这作为待检验的假说

Step 2: EXTRACT（提取）
  → AI 提取审查对象（代码 + 约束条件），剥离你的实现推理过程
  → 只传代码和合约给审查者，不传"我是怎么想的"

Step 3: DOUBT（对抗审查）
  → AI 派一个独立审查者（或你手动派给同事）
  → 审查者的提示词是恶意的："找出问题。假设作者过于自信。"
  → 审查者不验证代码"对不对"——而是找"哪里可能出错"

Step 4: RECONCILE（仲裁）
  → 你（不是审查者）对每个发现做分类：
    · 合同误读（你的约束没写清）→ 改约束，重新审
    · 有效+可操作 → 改代码
    · 有效但值得接受（修的成本 > 接受的代价）→ 标注，不修
    · 噪音（审查者缺少上下文导致的误报）→ 丢弃

Step 5: STOP（停）
  → 满足以下之一就停：
    · 新一轮审查只返回 trivial 或已考虑过的发现
    · 已完成 3 轮（不要再跑第四轮——升给人类决定）
    · 你说"够了"

    嵌入式实例：审查 I2C 错误恢复¶

    Step 1: CLAIM¶

    你: 我声称 bsp/i2c_eeprom.c 中的 EEPROM_Write 错误恢复逻辑是正确的——
    在任何 I2C 通信错误（NACK/TIMEOUT/BERR）后，总线都能在合理时间内恢复，
    不影响后续 I2C 通信。

    Step 2: EXTRACT¶

    AI 提取相关代码段（不附带实现推理）：

    /* [审查对象] bsp/i2c_eeprom.c:120-155 */
int32_t EEPROM_Write(uint16_t addr, const uint8_t *data, uint16_t len) {
    HAL_StatusTypeDef status;
    status = HAL_I2C_Mem_Write(&hi2c1, EEPROM_ADDR, addr,
                                I2C_MEMADD_SIZE_8BIT, (uint8_t *)data, len, 100);
    if (status != HAL_OK) {
        uint32_t error = HAL_I2C_GetError(&hi2c1);
        if (error & HAL_I2C_ERROR_AF) {
            /* __HAL_I2C_CLEAR_FLAG 操作 SR1 寄存器 AF bit (bit 10) */
            /* 注意：I2C_FLAG_AF ≠ HAL_I2C_ERROR_AF ——两者值不同，不可互换 */
            HAL_I2C_DeInit(&hi2c1);
            HAL_I2C_Init(&hi2c1);
        }
        return -1;
    }
    return 0;
}

/* [约束] 
- hi2c1 在 main() 中初始化，100kHz，7-bit 地址，单主机模式
- 被 FreeRTOS 任务调用（非中断上下文）
- 可能发生的 I2C 错误类型：NACK (AF) / TIMEOUT / BERR（总线错误）
- 注意：ARLO（仲裁丢失）在单主机模式下不会发生
- ⚠️ HAL_BUSY 是函数返回值（非错误码），与 HAL_I2C_GetError() 返回的错误码不同
*/

    Step 3: DOUBT — 审查者输出¶

    ## Doubt Review Findings

### Finding 1: TIMEOUT 和 BERR 发生时没有执行任何恢复操作
- 代码只检查了 `HAL_I2C_ERROR_AF`。
- `HAL_TIMEOUT` 返回后外设可能处于中间状态（半次传输未完成），
  后续 I2C 操作可能失败或产生意外波形。
- `HAL_I2C_ERROR_BERR`（总线错误，如 SDA 被意外拉低）需要生成额外
  SCL 脉冲来释放总线——仅靠 DeInit+Init 可能不够。
- 严重程度: HIGH

### Finding 2: FreeRTOS 任务中的 HAL_I2C_DeInit+Init 非互斥
- 如果有另一个 FreeRTOS 任务同时通过 hi2c1 通信，DeInit 和 Init 之间
  I2C1 处于半初始化状态——另一个任务的操作可能失败或产生不可预期行为。
- 需要互斥锁保护所有对 hi2c1 的访问（不仅是 EEPROM_Write），
  确保包含 DeInit+Init 在内的整个 I2C 事务是互斥的。
- 注意：互斥锁需通过 xSemaphoreCreateMutex() 在调度器启动前创建；
  持有锁执行 DeInit+Init 期间（约 1-2ms），更高优先级的任务会被阻塞——
  需要考虑实时性影响（FreeRTOS 互斥锁的优先级继承只能防止中优先级
  任务抢占，不能缩短 DeInit+Init 的硬件操作时间）。
- 严重程度: HIGH

### Finding 3: const 限定符被显式丢弃
- 函数签名 `const uint8_t *data` 正确表达了"不修改输入数据"的语义
- 但 `(uint8_t *)data` 显式丢弃了 const，仅为了绕过 HAL API 不接收
  const 指针的限制
- 这是 C 代码的常见坏味道——应该通过 const-safe wrapper 或接受
  这个 HAL API 设计缺陷（标注原因）
- 严重程度: LOW

### Finding 4: DeInit+Init 后没有验证设备是否恢复
- DeInit+Init 重置了 I2C 外设，但没有确认目标 EEPROM 是否可通信
- 建议在恢复后调用一次 HAL_I2C_IsDeviceReady() 确认设备响应再返回
- 严重程度: MEDIUM

### Finding 2: FreeRTOS 任务中的 HAL_I2C_DeInit+Init 非原子操作
- 如果有另一个 FreeRTOS 任务同时尝试访问 I2C1，DeInit 和 Init 之间
  I2C1 处于半初始化状态——另一个任务可能操作未初始化的外设。
- 需要互斥锁保护整个恢复序列。
- 严重程度: HIGH

### Finding 3: HAL_I2C_DeInit 后 hi2c1 的 State 成员变为 HAL_I2C_STATE_RESET
- HAL_I2C_Init 会重置 State。但如果 Init 本身失败（如 I2C 时钟未使能），
  State 会留在 HAL_I2C_STATE_RESET 而非 HAL_I2C_STATE_READY。
  后续调用 HAL_I2C_Mem_Write 会因状态检查失败直接返回 HAL_BUSY。
- 严重程度: MEDIUM

    Step 4: RECONCILE — 你的分类¶

    # 发现 分类 处理
    1 TIMEOUT/BERR 无恢复 Valid + Actionable 改：错误处理覆盖所有 HAL 错误码，对 BERR 加 SCL 脉冲恢复
    2 I2C 访问非互斥 Valid + Actionable 改：创建互斥锁保护所有 hi2c1 访问路径，标注 DeInit+Init 期间的实时性影响
    3 const 限定符丢弃 Valid Trade-off 不修：HAL API 本身不接收 const 指针，接受这个限制。在代码中加注释标注原因
    4 恢复后未验证设备 Valid Trade-off 不修：当前 NACK 场景中设备仍在总线上（只是暂时忙）；如果后续加入 BERR 恢复则需补设备验证

    💡 注意：如果你发现一个发现是"合同误读"（你给的约束不清楚导致审查者误判）——先修正约束描述，再重新审查。如果是"噪音"（审查者缺少领域知识导致的误报）——记录原因后丢弃。本示例中的 4 个发现均不属于这两类，所以表中只展示了 Actionable 和 Trade-off 两类。

    做对了的标准是什么¶

    ✅ 成功输出¶

    标准 说明
    审查者找到了你没考虑的问题 doubt 的价值就体现在这里——找到了"你对代码有信心"但实际存在问题的点
    你对发现做了分类 不是全盘接受审查者的发现，而是做了仲裁（actionable / trade-off / noise）
    有 stop condition 不是无限审查——1-2 轮后无新发现就停止
    每个 actionable 发现被修复后重新验证 不是"标了 actionable 就忘了"——实际改了代码并确认

    ❌ 失败输出¶

    失败表现 为什么不行 怎么办
    审查者报告"代码看起来不错" 审查者没有进入对抗模式——它在验证而不是质疑 换一个审查提示词，或换一个审查者（cross-model）
    所有发现都是噪音 要么代码确实好，要么约束不够清晰（审查者因缺少上下文而误报） 补充约束，重新审查
    审查了 4 轮还有新发现 审查对象太大或设计本身有问题 分解审查对象，或重新评估设计

    常见误区¶

    误区 1："审查者说有问题就是有问题"¶

    现象：审查者给了 10 个发现，你全部当 bug 改。但实际上有 3 个是审查者缺少上下文导致的误报（噪音），改了反而引入问题。

    正确做法：你才是仲裁者。Step 4 RECONCILE 是整个流程的核心——审查者提供信息，你来判断哪些是真的问题。审查者不知道"你们团队故意在这个文件里不检查 HAL 返回值因为历史原因"——这个判断只有你能做。

    误区 2："没有新发现 = 代码完美"¶

    现象：doubt 审查没找到新问题，你觉得代码完美了。

    正确做法：doubt 审查的覆盖率取决于审查者的能力 + 约束的清晰度 + 审查范围的大小。零发现可能意味着： - ✅ 代码确实好（最理想） - ❌ 约束不够清晰（审查者不知道该从哪找问题） - ❌ 审查对象太小（没有足够的复杂度暴露问题）

    当出现零发现时，检查后面两种可能。

    误区 3："doubt-driven 和 code-review 是一回事"¶

    现象：你觉得 doubt 就是"更严格的 code-review"。

    正确做法：两者有本质区别：

    code-review doubt-driven
    目的 全面评估（好+坏） 只找问题
    提示词 "从五个维度审查" "找出问题，假设作者过于自信"
    审查者 同一个 AI 上下文 全新上下文（独立审查者）
    输出 平衡报告（好+坏+建议） 问题列表
    时机 提交前、PR review 做完后验证、对已有代码没信心时

    进阶¶

    审查结果的 Workflow 自动化¶

    在 M2 2-7-并行代理 中有完整的 Workflow 脚本——5 维度审查 + 3 验证者对抗投票。把这个 Workflow 改为纯 doubt 模式（只怀疑、不验证）就是一个自动化的 doubt-driven 流水线。

    与 code-review 互补使用¶

    先 code-review（全面评估）→ 发现"有些地方不太对但说不清" → 对那几段代码跑 doubt（针对性对抗审查）。code-review 扫描面，doubt 深挖点。
