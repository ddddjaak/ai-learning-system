    AE 深化 Skill 路径入口¶

    概述¶

    M1 让你会用 Claude Code。M2 让你知道有哪些 skill 和扩展能力。M3 基于 ae-skills 插件，把这套能力翻译成 AE 工程师的日常工作语言——每个 skill 不再是抽象概念，而是你每天打开终端时会遇到的真实场景。

    类比：M2 给了你一张超市地图（42 个 skill 在哪）。M3 是你的工作流导航——"从理解需求到交付方案，按 AE-SKILLS 的六个阶段依次走：Define → Plan → Context → Implement → Verify → Deliver"。

    AE 工程师的一天¶

    AE 工程师的工作覆盖从"理解客户问题"到"写代码解决"到"交付方案"的完整链路。下面是一个典型的工作日——按时间线走一遍，每个节点对应一个 skill。

    08:30 客户邮件："你们芯片的 I2C 在 105°C 下通信不稳定"
      → interview-me [3-1] ← 把模糊描述翻译成可验证的技术问题

09:30 问题澄清了，需要确定方案方向
      → idea-refine [3-2] ← 发散多个方案方向 + 对比优劣 + 给出推荐

10:30 方向定了，写方案规格书
      → spec-driven-development [3-3] ← 含验收条件的可执行规格

11:30 规格确认，拆成可执行的任务
      → planning-and-task-breakdown [3-4] ← 任务拆解 + 依赖排序

13:00 开始写代码之前，先让 AI 理解项目
      → context-engineering [3-5] ← 配置 CLAUDE.md，AI 不再编造 API

14:00 写代码：测试先行 + 增量实现 + 文档验证
      → test-driven-development [3-6] ← 红→绿→重构
      → incremental-implementation [3-7] ← 薄垂直切片，每片编译+测试通过
      → source-driven-development [3-8] ← 用官方文档验证 API 正确性

16:00 代码写完，质量检查
      → code-review-and-quality [3-9] ← 五维度系统化审查
      → doubt-driven-development [3-10] ← 对抗审查，全新视角找深层问题

17:00 bug 修复 / 线上问题排查
      → debugging-and-error-recovery [3-11] ← 从现象到根因

次日：方案交付
      → documentation-and-adrs [3-12] ← 方案文档 + 架构决策记录
      → shipping-and-launch [3-13] ← 交付前检查清单 + 回滚方案

    关键洞察：AE 工程师的核心风险不是"代码写错了"，而是理解偏差——做了一堆客户没要的东西。所以工作流从 interview-me（澄清需求）开始，而不是从写代码开始。

    按工作流阶段：完整学习路径¶

    阶段一：Define（理解问题）—— 一切从"客户/需求方要什么"开始¶

    这三个 skill 帮你把模糊的需求变成清晰的规格。

    顺序 Skill 解决什么问题 如果没有它
    1 interview-me "需求太模糊，通过一问一答逐步澄清" 你脑补了需求——做了一堆客户没要的东西
    2 idea-refine "精炼模糊想法为可执行的方案方向" 只有一个方向，没对比过——可能选了次优方案
    3 spec-driven-development "先写规格（含验收条件）再实现" AI 自由发挥——写出来的和你要的不一样

    阶段二：Plan（规划）—— 把规格拆成可执行的任务¶

    顺序 Skill 解决什么问题 前置依赖
    4 planning-and-task-breakdown "拆解中型任务为可执行的小块，每块有验收条件" 3-3（spec 已完成）

    阶段三：Context（上下文）—— 让 AI 理解你的项目¶

    顺序 Skill 解决什么问题 前置依赖
    5 context-engineering "配置 CLAUDE.md 和项目规则——让 AI 不编造不存在的函数和寄存器" M2 2-3（CLAUDE.md）

    为什么在 Implement 之前？ 在开始写任何代码之前，先让 AI 理解你的项目结构、代码风格、硬件平台。否则 AI 会发明不存在的 HAL 函数和寄存器地址。

    阶段四：Implement（实现）—— 写代码¶

    顺序 Skill 解决什么问题 前置依赖
    6 test-driven-development "测试先失败再通过——确保代码行为被验证" 3-5
    7 incremental-implementation "大功能拆成薄切片，每片编译通过+测试通过" 3-6
    8 source-driven-development "用官方文档验证 AI 写的 API 是否正确" M2 2-5（context7）

    阶段五：Verify（验证）—— 代码对吗？¶

    顺序 Skill 解决什么问题 什么时候用
    9 code-review-and-quality "提交前/审查别人代码时，AI 从 5 个维度帮你找问题" 每次提交前、审查 PR 时
    10 doubt-driven-development "对抗审查你的代码——用全新视角找深层问题" 代码写完、准备合并时
    11 debugging-and-error-recovery "从现象到根因的系统化调试" 代码挂了、测试失败、硬件异常时

    阶段六：Deliver（交付）—— 方案交付¶

    顺序 页面 解决什么问题 前置依赖
    12 documentation-and-adrs "生成方案文档 + 架构决策记录" 3-3（spec 已完成）
    13 shipping-and-launch "交付前检查清单 + 回滚方案" 3-12

    前置依赖¶

    进入 M3 前，确保你已完成：

    • [ ] M1 全部（至少 1-6 桥梁篇已实操——能独立用 Claude Code 解决一个真实问题）
    • [ ] M2 关键页：2-0（skill 全景图）、2-1（ae-skills 日常融入）、2-3（CLAUDE.md 怎么写）
    • [ ] M2 推荐页：2-5（MCP 插件——至少 codegraph 能用）
    • [ ] ae-skills 插件已安装（V1.0+）
    • [ ] 有一个自己的嵌入式项目用于实操
    • [ ] 项目的 CLAUDE.md 已写好（M2 2-3 的产出）

    自学路径建议¶

    如果你时间有限¶

    1. 最少必读：3-0（本页）→ 3-1（interview-me）→ 3-2（idea-refine）→ 3-3（spec-driven）
    2. 加上推荐：+ 3-5（context-engineering）+ 3-9（code-review）
    3. 剩余按需：遇到对应场景时再查对应的 skill 页面

    如果你想走完整流程¶

    1. 快速上手：3-0 → 3-1（interview-me）→ 3-2（idea-refine）→ 3-3（spec-driven）→ 3-12（documentation）
    2. 完整流程：+ 3-4（planning）→ 3-5（context-engineering）→ 3-6（TDD）→ 3-7（incremental）→ 3-9（code-review）→ 3-11（debugging）→ 3-13（shipping）
