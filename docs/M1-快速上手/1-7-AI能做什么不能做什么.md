# AI 能做什么、不能做什么

> **这是一篇附录。** 不影响主线——你可以在需要时随时回来看。

## AI 擅长的事

| 任务类型 | 示例 | 可靠性 |
|---------|------|--------|
| 解释代码逻辑 | "这段 I2C 中断处理在做什么？" | ⭐⭐⭐⭐⭐ 很高 |
| 找常见 bug 模式 | "有没有未初始化的变量？有没有竞态条件？" | ⭐⭐⭐⭐ 高 |
| 生成代码框架 | "写一个 GPIO 驱动的 .c 和 .h 框架" | ⭐⭐⭐⭐ 高 |
| 写测试用例 | "为这个函数生成边界测试" | ⭐⭐⭐⭐ 高 |
| 重构/简化代码 | "把这个 200 行的函数拆成 3 个" | ⭐⭐⭐ 中高 |
| 翻译代码 / 写文档 | "把这段驱动的注释翻成中文" | ⭐⭐⭐⭐⭐ 很高 |
| 搜索和分析 | "项目中所有调用 HAL_I2C_* 的地方" | ⭐⭐⭐⭐⭐ 很高 |

## AI 容易出错的事

| 问题类型 | 表现 | 为什么 |
|---------|------|--------|
| **编造寄存器地址** | AI 给出一个看似合理的地址（如 `0x40005400`），但查数据手册发现不对 | AI 不知道特定芯片的寄存器映射，它在"猜"一个看起来像寄存器地址的数字 |
| **发明不存在的 API** | AI 写 `HAL_GPIO_InitEx()` 但 HAL 库里根本没有这个函数 | AI 知道类似的 API 命名模式，会推理出一个看起来合理的函数名——但它不存在 |
| **自信地给错误答案** | 你问"这段代码会不会导致死锁"，AI 很自信地说"不会"，但其实就是死锁了 | AI 的"自信度"不等于"正确度"。它对复杂并发逻辑的分析可能遗漏关键路径 |
| **指令集混淆** | AI 给出 ARM 指令但你用的芯片是 RISC-V | AI 训练数据里 ARM 代码最多，默认偏向 ARM |
| **忽略编译器和平台差异** | AI 给的代码在 GCC 上跑但你的项目用 IAR/Keil | 不同编译器的 `__attribute__`、`#pragma`、内置函数不完全兼容 |

## 你怎么验证 AI 的输出

> **核心原则：不信任 AI 的输出，直到你验证过。**

| AI 给的 | 你怎么确认 |
|---------|-----------|
| 代码修改 | 编译通过 → 跑测试 → 手动检查逻辑 |
| 寄存器地址 / 外设配置 | 查数据手册（datasheet），以手册为准 |
| bug 分析结论 | 看它指出的代码行，确认它的推理路径是对的 |
| API 函数名 | 搜索项目中这个函数是否真的存在（grep / IDE 跳转） |
| 性能优化建议 | 先测量再优化——AI 说的瓶颈可能不是真的瓶颈 |

## 三条底线

1. **AI 没有芯片的 datasheet。** 寄存器地址、时序参数、外设配置——这些以数据手册为准，AI 只是在猜。
2. **AI 是你项目中"最聪明但最不了解你代码的人"。** 它理解 C 语言、理解嵌入式模式，但它没接触过你的具体项目——CLAUDE.md 和 project context 是弥补这个差距的关键。
3. **你永远是决策者。** AI 给你建议和代码，你决定要不要采纳。AI 的"自信"不构成采纳理由——验证结果才是。
