    路径入口：SE 如何用 Claude Code¶

    这个页面做什么¶

    这是 M4（SE 实战） 的路径入口。读完这一页，你会知道： 1. SE（系统工程师）的典型工作流中，哪些环节有对应的 skill 可以加速 2. 推荐的阅读顺序和学习路径 3. 和 M3（AE实战）的技能重叠与差异 4. 不知道用哪个 skill 时怎么办——using-se-skills 元技能

    SE 和 AE 的关键区别：AE 面向客户（方案沟通、需求澄清、交付文档），SE 面向内部工程系统（需求分解、架构设计、规格撰写、设计审查、追溯性、发布质量门禁）。SE 是"工程质量守门员"——确保系统在纸面上就是对的，而不只是代码能跑。

    按频率分组 + 推荐学习顺序¶

    高频（几乎每个项目都要用）¶

    序号 页面 Skill 为什么高频
    4-1 需求分解 requirements-decompose 每个新项目/新功能都从需求分解开始——SE 的日常入口
    4-2 架构设计 architecture-design / software-architecture-design / hardware-architecture-design 需求分解完就进入架构设计——SE 的核心产出

    中频（每个阶段节点用）¶

    序号 页面 Skill 为什么中频
    4-3 规格与详细设计 spec-authoring / software-detailed-design / hardware-detailed-design / algorithm-design 架构确认后写正式规格——项目关键交付物
    4-4 设计审查 design-review / requirements-review 规格完成后审查——项目关键质量门禁
    4-5 编码规范审查 code-static-review 代码提交后审查编码规范——持续但按批次触发
    4-6 测试审查 test-plan-review / test-report-review 测试方案和报告的正式审查——验证阶段门禁

    低频（里程碑节点用）¶

    序号 页面 Skill 为什么低频
    4-7 追溯矩阵 traceability-matrix 需求→设计→测试追溯——评审节点或交付前触发
    4-8 发布审查 release-review 发布前最终质量门禁——每个 release 一次
    4-9 SE Goal Mode /se-goal（Goal Mode） 全自动 SE 流程——信任流程时使用

    using-se-skills：不知道用哪个 skill 时的入口¶

    using-se-skills 是 se-skills 的元技能（管道引导器）。它不是用来做具体工作的——它是用来帮你找到该用哪个 skill。

    什么时候触发¶

    场景 为什么需要
    你刚开始用 se-skills，不知道 Define→Design→Document→Verify 每个阶段该用什么 元技能自动检测你当前处于哪个阶段，推荐下一步该触发的 skill
    你手头有一个任务但不确定该用哪个 skill 描述你的任务，元技能会路由到正确的 skill
    你在 Pipeline 中间卡住了不知道下一步 元技能帮你查看当前进度，推荐接下来的选项

    怎么触发¶

    直接说你的意图即可，不需要记忆 skill 名称：

    我不知道该用哪个 SE skill——我有一个客户需求，需要分解成系统需求

    AI 会自动检测你的阶段（Define）并路由到 requirements-decompose。

    或者直接说：

    用 using-se-skills 帮我看看接下来该做什么

    元技能 vs 直接触发¶

    直接触发具体 skill 通过 using-se-skills
    适用 你明确知道要做什么 你不确定该用哪个 skill
    速度 快（一步到位） 多一步（AI 先判断再路由）
    推荐度 熟练后用 刚上手时用

    原则：你明确知道要做什么 → 直接触发。你不知道有哪些 skill 或不确定该用哪个 → 用 using-se-skills。

    两种工作模式¶

    se-skills 支持两种模式：

    Pipeline Mode（引导式） Goal Mode（自主式）
    怎么触发 说"帮我做需求分解" 说"端到端走完全流程" 或 /se-goal
    阶段切换 AI 展示选项，你选 AI 自动决定下一步
    验证失败 报告给你 AI 自查自修（最多重试 3 次）
    跟你互动 每阶段都问 只在真卡住时或完成时
    适用场景 你想一步步把控 你信任流程，想省时间

    详见 4-9 SE Goal Mode。

    本模块的运行案例¶

    整个 M4 使用一个统一的案例贯穿所有页面，方便你看到每个 skill 在同一项目中的上下游关系：

    案例：为某 BLDC 电机控制器（基于 STM32G4）增加过温保护（OTP: Over-Temperature Protection）子系统。

    背景：现有产品只有简单的 ADC 温度读取 + 软件阈值比较。客户要求满足 IEC 60730 Class B 功能安全标准，需要一套完整的三级保护机制（预警 → 降功率 → 紧急关断），包含硬件冗余检测路径和故障记录。

    SE 的工作：接收这个高层级需求 → 分解为可执行的系统需求 → 设计 HW/SW 架构 → 撰写规格 → 审查设计 → 建立追溯矩阵 → 发布审查。

    每个页面都从同一个 OTP 案例出发，但聚焦于该 skill 在该案例中的具体应用。

    与 M3（AE实战）的关系¶

    维度 M3 AE实战 M4 SE实战
    视角 "这段代码怎么写" / "客户要什么方案" "这个系统怎么设计才正确"
    典型输入 Bug 报告、功能需求、客户模糊问题 产品需求 / 客户需求文档
    典型输出 修复代码、方案文档、交付物 需求规格 + 架构设计 + 审查报告 + 追溯矩阵
    核心插件 ae-skills se-skills
    前置依赖 M1 + M2 M1 + M2（不需要先学 M3）

    学习建议¶

    • 如果这是你第一次接触 Claude Code：先完成 M1（安装 + 基础对话），再回来看 M4
    • 如果你是嵌入式/AE 想了解 SE 工作流：建议先学 M3（AE实战）建立日常使用习惯，再学 M4 建立系统级视角
    • 如果你已经是 SE：按页面顺序阅读（4-1 → 4-2 → ... → 4-10），每个页面 15-20 分钟。案例是统一的 OTP 子系统
    • 如果你想快速了解某个 skill：直接跳到对应页面——每页都是独立可读的
    • 如果你想一次性跑完全流程：看 4-9 SE Goal Mode
