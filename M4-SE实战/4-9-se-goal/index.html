    SE Goal Mode：全自动 SE 流程¶

    这个页面做什么¶

    这一页介绍 se-skills 的 Goal Mode（自主模式）——通过 /se-goal 命令或自然语言触发，让 AI 自动跑完 Define→Design→Document→Verify→Validate 全流程。与 Pipeline Mode（每阶段展示选项让你选）不同，Goal Mode 全自动执行，只在真卡住时才找你。

    Pipeline Mode vs Goal Mode¶

    Pipeline Mode（引导式） Goal Mode（自主式）
    怎么触发 说"帮我做需求分解" 说"端到端走完全流程" 或 /se-goal
    阶段切换 AI 展示 2-4 个选项，你选 AI 自动决定下一步
    验证失败 报告给你 AI 自查自修（最多重试 3 次）
    跟你互动 每阶段都问 只在真卡住时或完成时
    适用场景 你想一步步把控 你信任流程，想省时间

    怎么选？¶

    • 第一次用 se-skills → 用 Pipeline Mode——理解每个阶段在做什么
    • 流程已经跑过几次 → 用 Goal Mode——省时间
    • 需求有歧义或约束复杂 → 用 Pipeline Mode——需要你的领域判断
    • 需求清晰、约束明确 → 用 Goal Mode——让 AI 自动处理

    怎么触发 Goal Mode¶

    方式 1：Slash 命令¶

    /se-goal 完成 OTP 子系统的 SE 全流程

    方式 2：自然语言¶

    端到端做完 OTP 子系统的 SE 流程——从需求到追溯矩阵

    从 PRD 到追溯矩阵全部自动做，不用问我

    自动完成温度传感器方案的 SE 全链路

    触发条件¶

    Goal Mode 在以下条件下激活：

    触发 示例
    /se-goal 命令 /se-goal 完成温度传感器方案的SE全流程
    明确的完成意图 "端到端做完"、"走完全流程"、"自动完成"
    清晰的终态声明 "从PRD到追溯矩阵全部自动做"
    Pipeline Mode 后说"开干" 展示第一个选项后用户说"直接开始，不用问我"

    不确定时默认 Pipeline Mode。只有意图明确时才进入 Goal Mode。

    自动 Skill 选择规则¶

    Goal Mode 不展示选项——根据当前阶段和已有产物自动选择：

    阶段 条件 自动选择的 Skill
    Define 始终 requirements-decompose
    Design 无架构 architecture-design（系统级）
    Design 系统架构存在，用户提到固件 software-architecture-design
    Design 系统架构存在，用户提到硬件 hardware-architecture-design
    Document 系统架构存在 spec-authoring
    Document SW 架构存在 software-detailed-design
    Document HW 架构存在 hardware-detailed-design
    Document 有算法需求 algorithm-design
    Verify 有规格文档 design-review（四视角）
    Verify 只有需求文档 requirements-review
    Verify 有源代码 code-static-review
    Verify 有测试方案 test-plan-review
    Verify 有测试报告 test-report-review
    Verify 有发布包 release-review
    Validate 有审查报告 traceability-matrix

    OTP 案例：Goal Mode 完整执行¶

    目标：完成 BLDC 电机控制器 OTP 子系统的 SE 全流程

    执行过程¶

    ✅ Define  完成 — 23 条需求（REQ-001 ~ REQ-023），4 个类别
✅ Design  完成 — 5 个模块（MOD-01 ~ MOD-05），12 个接口，3 个 ADR
✅ Document 完成 — SOD v1.0 + HW-SW IF Spec v1.0 + Test Plan v1.0
✅ Verify  完成 — design-review: 10 条发现（4 HIGH / 4 MEDIUM / 2 LOW）
                 — code-static-review: 8 条违规（1 个真实 bug）
                 — test-plan-review: 4 条发现
✅ Validate 完成 — 追溯矩阵: 需求覆盖率 100%, 设计覆盖率 95%, 测试覆盖率 87%
                 — 2 个覆盖缺口（已标注修复计划）

    自动修复示例¶

    Verify 阶段 design-review 发现 4 个 HIGH：

    1. FEA-1: AT24C02 容量不够存完整故障记录 → AI 自动修改设计：压缩记录格式，从 512B 压缩到 256B
    2. FEA-2: Class B 自检细节缺失 → AI 自动补充：CPU 寄存器自检的伪代码
    3. FEA-3: NTC 开路检测缺失 → AI 自动补充：ADC 原始值阈值检测逻辑
    4. FEA-4: PWM 控制任务临界区过长 → AI 自动修改：将临界区从 50µs 压缩到 10µs

    第二轮 Verify：4 个 HIGH 全部关闭，新增 1 个 MEDIUM（压缩后的故障记录丢失了部分诊断信息）。

    5 个 Agent 角色¶

    se-skills 包含 5 个预配置的审查角色，用于 design-review 的多视角对抗审查：

    Agent 角色 视角
    system-architect 资深系统架构师 系统一致性、约束满足、跨域集成、风险暴露
    hw-domain-expert 硬件域专家 引脚分配、电源域、时钟树、信号完整性、电气合规
    fw-domain-expert 固件域专家 驱动接口、RTOS 集成、内存映射、启动流程、并发模型
    verification-engineer 验证工程师 可测试性、测试覆盖率、追溯完整性、验证方法论
    compliance-reviewer 合规审查员 法规合规、功能安全、安全控制、隐私、行业标准

    在 Goal Mode 中，design-review 自动扇出 5 个 agent 并行审查同一份产物，合并生成分类问题报告。

    进度报告¶

    Goal Mode 每个阶段完成后报告一行状态：

    ✅ Define  完成 — 23 条需求（REQ-001 ~ REQ-023）
✅ Design  完成 — 5 个模块（MOD-01 ~ MOD-05）
⏳ Document 进行中 — 正在生成 SOD...

    不会在阶段之间问"准备好了吗？"——直接继续。

    停止条件¶

    条件 行为
    traceability-matrix 通过且零缺口 成功——报告完成摘要
    同一阶段连续 3 次重试失败 升级——报告卡住原因和建议选项
    总 skill 执行次数超过 20 停止——报告当前进度，询问是否继续
    用户说"停止"/"暂停" 暂停——报告当前阶段、已完成产物、下一步
    发现矛盾的需求或约束 升级——指出矛盾的具体 REQ-XXX ID

    常见问题¶

    Q1: Goal Mode 会不会做出我不想要的决策？¶

    Goal Mode 在以下情况下会升级给你：矛盾的需求、物理上不可实现的约束、3 次重试仍失败。对于明确的需求和约束，它会自动做出合理决策——但你可以随时用"暂停"打断。

    Q2: Goal Mode 的结果和 Pipeline Mode 有什么区别？¶

    输出产物完全一致——只是过程不同。Pipeline Mode 每阶段展示选项让你选；Goal Mode 自动选择并执行。最终的 SOD、架构文档、审查报告、追溯矩阵的质量应该相同。

    Q3: Goal Mode 失败了怎么办？¶

    如果 Goal Mode 升级给你——说明自动流程遇到了它无法解决的问题。你可以： 1. 回答它的问题，让它继续 2. 切换到 Pipeline Mode，手动控制后续流程 3. 修复问题后重新触发 Goal Mode（它会从上次完成的阶段继续）
