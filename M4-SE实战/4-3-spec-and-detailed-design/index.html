    spec-authoring + detailed-design（SE 场景）¶

    这些 skill 做什么¶

    这一页覆盖 SE 工作流中 Document 阶段的 4 个 skill——从架构设计到正式规格文档和详细设计：

    Skill 做什么 输出
    spec-authoring 架构+需求 → 系统级规格文档 SOD（软件概要设计）、HW-SW IF Spec（软硬件接口规格）、Test Plan（测试方案）
    software-detailed-design SW 架构 → 模块级详细设计 函数签名、状态机、数据结构、错误处理矩阵、资源量化预算
    hardware-detailed-design HW 架构 → 电路级详细设计 原理图约束、PCB 规则、PDN 设计（Z_target 计算）、热分析（T_j 计算）
    algorithm-design 算法需求 → 算法详细设计 信号处理、控制回路、标定程序、滤波器设计（含数学模型和量化位宽分析）

    类比：架构设计是画平面图——"卧室在哪、水管怎么走"。详细设计是出施工图——"水管用 PPR 还是 PVC、管径 20mm 还是 25mm、弯头用 45° 还是 90°"。前者定骨架，后者定每个细节。

    触发方式：/se-spec 或说"帮我写规格"/"帮我做详细设计"

    什么时候用 / 什么时候不该用¶

    ✅ 应该用¶

    场景 为什么需要
    架构设计完成，需要正式规格文档 架构是骨架——规格文档是交付物。客户/评审/认证机构需要看到正式文档
    某个固件模块需要详细设计后再编码 复杂模块（如状态机、ISR、DMA 链路）直接写代码容易返工——先设计再实现
    硬件原理图即将开始——需要详细的电路约束 NTC 分压精度、LPF 截止频率、PDN 阻抗目标——这些需要数学推导，不是拍脑袋
    算法需要独立于固件进行设计 FOC 控制算法、卡尔曼滤波、PID 参数——需要先在数学层面验证再嵌入固件

    ❌ 不该用¶

    场景 为什么不该用 替代方案
    架构还没确认 基于不稳定的架构写规格——架构改一次规格全废 先完成 architecture-design
    改动很小（改一个阈值参数） 不需要详细设计——直接改代码 直接在代码层面修改
    需求还有歧义 规格文档的需求来源不清晰——写出来也是错的 先完成 requirements-decompose

    spec-authoring：系统级规格¶

    spec-authoring 从已确认的架构和需求生成三类系统级规格文档：

    SOD（Software Outline Design / 软件概要设计）¶

    软件系统的顶层设计文档——模块划分、任务模型、数据流、资源预算。

    OTP 案例：基于 4-2 的架构设计，生成 OTP 子系统的 SOD： - 4 个 FreeRTOS 任务的职责、优先级、栈大小 - 温度环形缓冲区的数据结构和访问协议 - 三级保护状态机的状态转换表 - Flash/RAM 预算分配

    HW-SW IF Spec（软硬件接口规格）¶

    硬件和软件之间的精确接口定义——寄存器映射、信号时序、中断优先级、DMA 通道分配。

    OTP 案例： - ADC1 的通道分配、采样时间、DMA 缓冲区地址 - COMP1 的输入/输出极性、内部互连到 TIM1 BRK 的配置寄存器 - I2C1 的 EEPROM 读写时序和页写入约束（AT24C02 页大小 = 8 字节） - TIM1 的刹车触发源（软件 BG 位 vs 硬件 BRK 输入）和恢复序列

    Test Plan（测试方案）¶

    验证所有需求的测试策略——每个需求对应哪些测试用例、测试方法、通过标准。

    OTP 案例： - FR-2.3（紧急关断）→ TC-5.1：注入 125°C 温度信号，验证 PWM 关断延迟 ≤ 50ms - FR-3.1（硬件冗余）→ TC-6.1：模拟 MCU 程序跑飞（禁用软件），验证 COMP1 能独立触发刹车 - NFR-1（时序）→ TC-7.1：示波器测量温度超标到 PWM 关断的实际延迟

    software-detailed-design：固件模块详细设计¶

    software-detailed-design 对单个固件模块进行详细设计——精确到函数签名、数据结构、状态机伪代码、错误处理矩阵。

    OTP 案例——otpEngineTask 模块的详细设计：

    函数签名¶

    // 温度阈值比较——返回触发的保护级别
otp_prot_level_t otp_check_thresholds(const temp_sample_t *sample);

// 状态机主循环——由 FreeRTOS task 调用
void otp_engine_task(void *params);

// 执行保护响应动作
void otp_execute_response(otp_prot_level_t level, uint8_t channel, int16_t temp);

// 写故障记录到 EEPROM（紧急关断时调用）
HAL_StatusTypeDef otp_write_fault_log(const otp_fault_record_t *record);

    状态机伪代码¶

    otp_prot_level_t otp_check_thresholds(const temp_sample_t *sample) {
    for (int ch = 0; ch < 3; ch++) {
        int16_t temp = sample->channel[ch];
        if (temp >= g_otp_config.shutdown_thresh) return OTP_PROT_SHUTDOWN;
        if (temp >= g_otp_config.derating_thresh) return OTP_PROT_DERATING;
        if (temp >= g_otp_config.warning_thresh) return OTP_PROT_WARNING;
    }
    return OTP_PROT_NORMAL;
}

    错误处理矩阵¶

    错误场景 检测方式 响应动作 恢复策略
    NTC 开路（读数 = -40°C） ADC 原始值 < 阈值 上报 SENSOR_FAULT，忽略该通道 下次采样重新检测
    NTC 短路（读数 = +150°C） ADC 原始值 > 阈值 上报 SENSOR_FAULT，忽略该通道 下次采样重新检测
    ADC 采样超时（DMA 未完成） HAL_ADC 超时返回 上报 ADC_TIMEOUT，使用上次有效值 连续 3 次超时 → 切换到降功率
    EEPROM 写入失败 I2C NACK 重试 3 次，仍失败则上报 不阻塞关断流程——故障记录丢失可接受

    hardware-detailed-design：电路详细设计¶

    hardware-detailed-design 对硬件模块进行详细设计——原理图约束、PCB 规则、PDN 设计、热分析。

    OTP 案例——NTC 信号调理电路的详细设计：

    NTC 分压电阻精度分析¶

    V_out = VREF+ × R_NTC / (R_pullup + R_NTC)
    • R_pullup = 10kΩ ± 1%（金属膜电阻）
    • NTC（25°C 时 10kΩ）在 130°C 时约 680Ω
    • 分压比在 130°C：680 / (10000 + 680) = 0.064 → V_out = 0.211V
    • 分压比在 25°C：10000 / (10000 + 10000) = 0.5 → V_out = 1.65V
    • ADC 分辨率：3.3V / 4096 = 0.806mV/LSB
    • 130°C 时 NTC 每°C 电阻变化约 20Ω → 电压变化约 0.6mV → 接近 1 LSB
    • 结论：高温段 ADC 精度不足——需要过采样（4× → 2 LSB 有效分辨率）或使用更高分辨率 ADC

    LPF 截止频率计算¶

    f_c = 1 / (2π × R × C) = 1 / (2π × 100Ω × 100nF) = 15.9 kHz
    • PWM 频率 = 20kHz → 需要衰减 PWM 开关噪声
    • 15.9kHz 截止频率对 20kHz 衰减不足（仅 -1.2dB）
    • 修正：将 C_filter 改为 1µF → f_c = 1.59kHz → 对 20kHz 衰减 -22dB
    • 但需注意：1µF + 100Ω 的时间常数 τ = 100µs → ADC 采样时间需 > 5τ = 500µs

    热分析¶

    T_J = T_A + P × θ_JA
    • OTP 子系统本身功耗 < 5mW（ADC + COMP + DAC）
    • θ_JA（LQFP-100）= 45°C/W
    • T_J = 85°C + 0.005W × 45°C/W = 85.2°C → 安全（远低于 T_Jmax = 150°C）

    algorithm-design：算法详细设计¶

    algorithm-design 对算法模块进行独立设计——数学模型、量化位宽、时序预算。

    OTP 案例——NTC 温度转换算法的详细设计：

    查表法 vs Steinhart-Hart 方程¶

    方法 精度 计算量 Flash 占用 适用场景
    查表法（256 点） ±2°C（插值） 1 次查表 + 1 次线性插值 512B CPU 资源紧张、精度要求不高
    Steinhart-Hart（定点数） ±0.5°C 1 次 ln + 2 次乘 + 1 次除 200B 需要高精度、CPU 有 FPU 或定点数库

    OTP 选择：查表法——因为 CON-2 约束"不使用浮点运算"，且 ±3°C 精度目标查表法足够。

    量化位宽分析¶

    • ADC 原始值：12 位（0-4095）
    • 查表索引：8 位（0-255）→ 需要将 12 位 ADC 值右移 4 位
    • 温度输出：int16_t，单位 0.1°C（定点数，1 位小数）→ 范围 -400 ~ +1500（-40°C ~ 150°C）
    • 中间计算：int32_t（防止乘法溢出）

    做对了的标准是什么¶

    ✅ 成功输出¶

    标准 说明
    规格文档覆盖所有需求 每条 FR/NFR/IR/CON 在 SOD 中有对应的设计描述
    HW-SW 接口精确到寄存器级 不是"使用 ADC1"——是"ADC1_IN1, 采样时间 47.5 cycles, DMA1_CH1 循环模式"
    详细设计有函数签名和数据结构 不是"温度采集模块"——是 int16_t ntc_convert(uint16_t adc_raw, uint8_t ch)
    每个 TBD 有 owner 和 due date 裸 TBD（"待定"）是验证失败——说明设计不完整
    算法有量化位宽分析 不是"用查表法"——是"8 位索引、int32_t 中间计算、int16_t 输出"

    ❌ 失败输出¶

    失败表现 为什么不行 怎么办
    规格文档的章节全是"参见架构设计" 规格应该比架构更详细——如果一样详细就不需要规格 每个章节补充实现细节
    HW-SW 接口只写了"UART2" 不够——需要波特率、帧格式、超时、错误处理 补充完整的接口规格
    详细设计没有错误处理矩阵 嵌入式系统的 80% 代码是错误处理——不设计就不会实现 逐个模块补充错误场景和响应

    常见误区¶

    误区 1："规格文档就是架构设计加长版"¶

    现象：你让 AI 写规格文档，它输出的内容和架构设计几乎一样——只是多了一些段落。

    正确做法：规格文档比架构设计更具体。架构说"使用 ADC1 采集温度"。规格说"ADC1_IN1, 规则组扫描, 采样时间 47.5 cycles, DMA1_CH1 循环模式, 缓冲区地址 0x2000_0100, 100ms 触发一次"。

    误区 2："详细设计是写代码前的最后一步——可以直接跳过"¶

    现象：你觉得架构设计已经够详细了，直接开始写代码。写到一半发现状态机有死锁——两个状态互相等待对方的信号。

    正确做法：复杂模块（状态机、ISR、DMA 链路、算法）必须先做详细设计再编码。详细设计的成本 < 返工成本的 1/10。

    误区 3："硬件详细设计不需要数学推导"¶

    现象：你画了 NTC 分压电路，选了 10kΩ 上拉电阻——觉得"差不多"就行。生产后发现高温段精度不足，需要改板。

    正确做法：硬件详细设计的每个参数都需要数学验证。NTC 分压精度、LPF 截止频率、PDN 阻抗目标——这些都有公式。AI 可以帮你做计算，但你必须确认公式和输入参数的正确性。

    进阶¶

    与 design-review 配合¶

    详细设计完成 → 用 design-review 对详细设计做审查——重点看可行性视角（参数计算是否正确、资源预算是否超限）。

    与 code-static-review 配合¶

    详细设计中的函数签名和数据结构 → 直接作为编码的规范——编码规范审查时对照检查实现是否偏离设计。

    与 traceability-matrix 配合¶

    详细设计中的每个模块、函数、接口 → 都应追溯到需求 ID。见 traceability-matrix。
