    release-review（SE 场景）¶

    这个 skill 做什么¶

    release-review 是一个发布就绪审查引擎——它在发布之前检查所有工程交付物是否完整、所有质量门禁是否通过、所有已知风险是否有应对措施。在 SE 场景下，这个 skill 的核心价值是在按下"发布"按钮之前做最后一次系统级检查——防止"有条件的通过"积累成发布事故。

    类比：飞机起飞前的检查清单（Pre-flight Checklist）——引擎正常 ✅、油量足够 ✅、航路许可 ✅、舱门关闭 ✅。飞行员不会凭"感觉差不多"起飞——每一项都要确认。release-review 是 SE 的 Pre-release Checklist。

    和 design-review 的区别： - design-review：审查设计质量——"设计有没有问题？" - release-review：审查发布就绪度——"所有东西都做完了吗？能发布吗？"

    相关 SE skill： - release-review：发布审查——发布前的最终质量门禁 - se-review：SE 综合审查——可以包含发布审查作为一个环节 - test-plan-review：测试计划审查 - test-report-review：测试报告审查

    什么时候用 / 什么时候不该用¶

    ✅ 应该用（SE 场景）¶

    场景 为什么需要 release-review
    软件版本准备发布给客户或生产 "差不多好了"不是发布标准——需要逐项确认
    功能安全相关系统的每次发布 IEC 60730 / ISO 26262 要求有记录的发布审查
    项目里程碑节点——如 β 版发布、RC 版发布 每个发布节点有不同级别的审查要求
    上次发布后出现了回归问题——这次需要更严格的发布门禁 从上次问题中学习——把缺失的检查项加入本次审查清单

    ❌ 不该用¶

    场景 为什么不该用 替代方案
    内部开发调试版本（非正式发布） 调试版本不需要完整审查——快速迭代 > 形式完整 轻量自检清单
    只改了文档（README、注释） 文档变更不涉及功能——不需要完整发布审查 diff review
    项目还在开发中期——离发布还很远 release-review 是最终门禁——不是中期检查 design-review 或进度审查

    怎么触发¶

    触发方式¶

    OTP 子系统固件 v1.0 准备发布给客户进行集成测试。
帮我做 release-review。需要检查：
- 所有需求是否已实现（对照 traceability-matrix）
- 所有测试是否已通过（测试报告）
- 所有 HIGH 以上设计审查发现是否已关闭
- 所有编码规范违规是否已修复或豁免
- 已知问题（Known Issues）和限制是否已记录
- 版本号和 Change Log 是否正确

    发布审查清单的通用框架¶

    每个项目的 release-review 清单可以根据以下框架定制：

    必须检查（每次发布）¶

    1. 需求实现完整性：所有 MUST 需求已实现并测试
    2. 测试完整性：测试通过率 = 100%（或已知失败有记录和计划）
    3. 高严重度发现关闭：所有 HIGH/CRITICAL 审查发现已关闭
    4. 编码规范合规：无未豁免的 Mandatory 违规
    5. 版本信息正确：版本号 + Change Log + 构建信息完整

    建议检查（根据项目风险级别）¶

    1. 文档完整性：需求/设计/测试/API 文档版本一致
    2. 已知问题记录：所有已知问题有描述 + 影响评估 + 缓解措施
    3. 回滚计划：如果发布失败，退回到哪个版本

    功能安全项目额外检查¶

    1. 安全分析：FMEA / FTA 已更新至当前版本
    2. 认证证据包：追溯矩阵 + 测试报告 + 审查记录已打包
    3. 独立审查：非开发团队的独立人员完成发布审查

    做对了的标准是什么¶

    ✅ 成功输出¶

    标准 说明
    每个检查项有三态结果 通过 / 有条件通过 / 不通过——不是全部"通过"
    每个"不通过"和"有条件通过"附带了阻塞条件 明确"什么情况下可以通过"——不是模糊的"差不多"
    已知问题被显式记录 不是藏在文档角落里——发布审查报告的前面部分就应该列出来
    审查结论明确 APPROVED / CONDITIONALLY APPROVED / REJECTED——不是"建议再改改"

    ❌ 失败输出¶

    失败表现 为什么不行 怎么办
    所有检查项都"通过"但没有任何条件或建议 审查不够深入——任何非平凡系统都应该有已知问题或待改进建议 追加："如果你是这个系统的维护者，你希望知道什么？"
    审查发现了 2 个 HIGH 未关闭——但结论仍然是 APPROVED 自相矛盾——有未关闭的 HIGH 发现不能直接 APPROVED 改为 CONDITIONALLY APPROVED——标注修复条件
    已知问题列了 15 条但没有优先级或影响评估 决策者无法判断哪些问题真的影响发布 每条已知问题标注严重度 + 影响范围

    常见误区¶

    误区 1："release-review 就是再跑一遍测试"¶

    现象：你觉得 release-review 等于"确认测试通过了"。测试确实通过了——但发布审查的关注点是工程流程的完整性，不只是测试结果。

    正确做法：release-review 验证的是"流程是否完整"——需求是否被实现了、设计是否被审查了、编码规范是否被遵守了、文档是否齐了。测试是流程中的一个环节——不是全部。

    误区 2："AI 可以完全自动做 release-review"¶

    现象：AI 给出了 APPROVED 的审查结论——你直接签名发布了。

    正确做法：AI 的 release-review 是建议——不是决定。SE 必须亲自确认 AI 无法确认的事项：客户是否真的准备好了接收这个版本？团队是否有未记录的担忧？实际硬件测试中是否有"偶发性"问题在测试报告中被略过了？

    误区 3："有条件通过"就是没通过——等于项目失败¶

    现象：审查结果是 CONDITIONALLY APPROVED——你觉得这是负面结论。

    正确做法：绝大多数非平凡系统的发布审查结果是"有条件通过"——有 1-3 个条件（如"修复 HIGH 发现 #5"、"补充测试用例 TC-12.3"）。这不是失败——这是工程严谨性的体现。APPROVED 没有任何条件通常只见于极简单的 bug fix 发布。

    与 CI/CD 配合¶

    将发布审查的部分检查项自动化（CI 中自动验证：版本号递增、编译无警告、测试全部通过）——让 AI 专注于 CI 无法自动检查的项（文档完整性、已知问题记录、回滚计划）。
