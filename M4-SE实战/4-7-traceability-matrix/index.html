    traceability-matrix（SE 场景）¶

    这个 skill 做什么¶

    traceability-matrix 是一个可追溯性矩阵生成引擎——它读取需求文档、设计文档、测试用例，生成一个矩阵表显示每条需求是否被设计覆盖、是否被测试验证。在 SE 场景下，这个 skill 的核心价值是在评审或交付之前，一眼看出哪些需求是"孤岛"——有需求但没设计、有设计但没测试。

    类比：你有一个 30 项的装修要求清单和一份装修图纸。traceability-matrix 是那个拿着清单逐条对照图纸的人——"第 7 项"厨房防水"在图纸上没看到对应的设计方案"、"第 12 项"网络布线"有设计方案但没有对应的验收测试"。在功能安全认证（IEC 60730 / ISO 26262 / IEC 61508）中，需求→设计→测试的可追溯性不是"最佳实践"——是强制要求。

    相关 SE skill： - traceability-matrix：可追溯性矩阵——需求→设计→测试的双向追溯 - se-traceability：SE 可追溯性——更广泛的可追溯性视图（包含风险、决策、变更）

    什么时候用 / 什么时候不该用¶

    ✅ 应该用（SE 场景）¶

    场景 为什么需要 traceability-matrix
    功能安全认证审计——需要证明每条需求被实现和测试 审计员会随机抽查需求的追溯链——追溯缺失 = 认证失败
    设计审查节点——需要确认所有需求都有设计方案 未覆盖的需求 = 设计遗漏——在审查中发现比在测试中发现便宜
    测试完成后——需要确认所有需求都被测试了 未测试的需求 = 验证盲区——功能安全标准不可接受
    需求变更后——需要评估变更影响范围 追溯矩阵告诉你"改了这条需求会影响哪些设计和测试"

    ❌ 不该用¶

    场景 为什么不该用 替代方案
    项目只有 3 条需求且极其简单 建立追溯矩阵的成本 > 手工对照的成本 在需求文档中手写交叉引用
    需求还在频繁变更中 矩阵需要频繁重建——维护成本高 等需求基线冻结后再建矩阵
    非安全相关的内部工具项目 追溯矩阵的主要驱动因素是合规——非安全项目可选做 简化的需求→设计映射表

    怎么触发¶

    触发方式¶

    基于 OTP 子系统的以下文档，帮我生成需求→设计→测试的可追溯性矩阵：

- 需求文档：OTP_SRS_v0.1（FR-1 ~ FR-4, NFR-1 ~ NFR-5, IR-1 ~ IR-5）
- 架构设计文档：OTP_ARCH_v0.1（包含模块定义、数据流、硬件设计）
- 测试用例文档：OTP_TEST_v0.1（28 个测试用例）

需要标识：
- 每条需求是否有对应的设计元素
- 每条需求是否有对应的测试用例
- 每个设计元素是否追溯到至少一条需求（避免过度设计）
- 是否存在"孤儿测试"（测试了设计中没有的功能）

| NFR-4: OTP 子系统增加功耗 ≤ 5mW | ❌ 无对应设计元素 | ❌ 无测试用例 |

**问题**：功耗需求没有对应的设计分析（哪个模块消耗多少功率、总功率如何计算），也没有对应的测试（如何在系统中测量 OTP 子系统单独增加的功耗）。

    可追溯性矩阵的核心价值¶

    不只是"打勾表"¶

    certification 审计中，审计员会随机抽取 3-5 条需求，要求你展示"从需求到代码到测试的完整链条"。如果任何一环断裂——认证失败。追溯矩阵的价值：

    追溯方向 回答的问题 断裂意味着
    需求 → 设计 "每条需求有没有对应的设计方案？" 有需求没做——功能遗漏
    设计 → 测试 "每个设计元素有没有被测试？" 做了没测——验证盲区
    需求 → 测试 "每条需求有没有对应的测试用例？" 需求没被验证——不可接受
    测试 → 需求 "每个测试用例对应哪条需求？" 测了不需要的东西——浪费 + 混淆
    设计 → 需求 "每个设计元素对应哪条需求？" 过度设计——增加了不必要的复杂度

    做对了的标准是什么¶

    ✅ 成功输出¶

    标准 说明
    每个需求行有三个状态之一 覆盖 / 部分覆盖 / 未覆盖——不是全部"已覆盖"
    覆盖缺口被量化并分类 红色（必须修）/ 黄色（建议修）/ 白色（信息缺口）
    有反向追溯 不只是需求→设计→测试的正向追溯——也检查了测试→需求和设计→需求的逆向
    统计总览 一眼可以看到覆盖率和缺口数——审查者和审计员需要这个摘要

    ❌ 失败输出¶

    失败表现 为什么不行 怎么办
    矩阵显示 100% 覆盖——但实际有遗漏 矩阵的覆盖判断过于宽松——把"涉及"当成了"覆盖" 收紧追溯标准：设计必须显式解决需求所述的问题，不只是"相关"
    缺口只标注了"缺失"但不给修复建议 团队知道缺什么但不知道怎么补 每个缺口附带"怎么修复"的具体建议
    矩阵是静态文本表格——不可筛选、不可排序 23 条需求能一眼看完——230 条呢？ 使用 CSV/Markdown 表格——可以导入 Excel 做筛选和透视

    常见误区¶

    误区 1："可追溯性就是在一张表里打勾"¶

    现象：你花了 30 分钟把所有表格填上 ✅——看起来很完整。但实际内容是"需求 FR-1.1 → 设计 D-ADC → 测试 TC-1.1"——这三者之间的语义关联是虚假的。D-ADC 是一个 ADC 配置块——它和 FR-1.1（3 路温度采集）的关系是"有关联"但不等于"解决了需求"。

    正确做法：每个追溯链接必须有语义验证——不只是"这两个 ID 在同一个子系统里"。AI 可以帮助做这个验证——给 AI 需求文本 + 设计描述 + 测试目的描述——问 AI 这个追溯链是否成立。

    误区 2："需求覆盖率不到 100% 就是失败"¶

    现象：你发现 NFR-4（功耗）没有测试——觉得项目失败了。

    正确做法：不是所有需求都可以被直接测试（如约束条件 CON-1"不可增加外部 IC"）。对于不可直接测试的需求——用设计审查或分析来代替测试。在矩阵中标注"通过设计审查/分析验证"而非"无测试"。

    误区 3："追溯矩阵只对认证审计有用"¶

    现象：你觉得如果项目不需要 IEC 60730 认证，追溯矩阵没有价值。

    正确做法：追溯矩阵也是变更影响分析的基础工具。当需求变更时——矩阵告诉你哪些设计和测试会受影响。当测试失败时——矩阵告诉你这对应哪条需求。对于任何超过 10 条需求的系统，追溯矩阵都会在被问到"这个改动影响什么"时节省大量时间。

    进阶¶

    与 se-traceability 配合¶

    traceability-matrix 是需求→设计→测试的追溯。se-traceability 扩展到更广的维度——需求→风险→决策→变更——适合更复杂的安全关键系统。

    追溯矩阵的版本管理¶

    每次需求/设计/测试变更后，追溯矩阵应更新版本：

    OTP_Traceability_Matrix_v1.0.xlsx  ← 初版
OTP_Traceability_Matrix_v1.1.xlsx  ← 补充 FR-4.2 测试后的更新
OTP_Traceability_Matrix_v2.0.xlsx  ← 认证审计基准版本

    与 release-review 配合¶

    追溯矩阵的完整度（红色缺口 = 0）是发布审查的准入条件——见 release-review。
