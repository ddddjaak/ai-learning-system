    requirements-decompose（SE 场景）¶

    这个 skill 做什么¶

    requirements-decompose 是一个系统需求分解引擎——你给它一个高层级的客户需求或产品需求，它帮你逐层分解为可验证的系统需求：功能需求、非功能需求、接口需求、约束条件。在 SE 场景下，这个 skill 的核心价值是把一句"我要 XXX"变成一组"系统必须满足以下 N 条可验证条件"。

    类比：客户说"我要一栋舒服的房子"——建筑师不会直接画图。他先分解："舒服"= 温度 18-26°C（暖通需求）+ 噪音 < 35dB（隔音需求）+ 日照 ≥ 4h/天（朝向需求）+ ...。requirements-decompose 就是那个帮你从"舒服"分解出暖通/隔音/朝向规格的建筑师。

    触发方式：/se-requirements 或说"帮我做需求分解"

    相关 SE skill： - requirements-decompose：主需求分解引擎——输入高层需求，输出结构化需求集（本页） - requirements-review：需求审查——检查需求集的完整性、一致性、可验证性（见 4-4 设计审查） - spec-authoring：规格撰写——把需求写成正式的系统规格文档（见 4-3 规格与详细设计）

    什么时候用 / 什么时候不该用¶

    ✅ 应该用（SE 场景）¶

    场景 为什么需要 se-requirements
    拿到一份高层级的产品需求文档（PRD）或客户需求 PRD 通常是"想要什么效果"——不是"系统必须满足什么条件"
    需求涉及多个子系统（硬件 + 软件 + 机械） 需要分解到每个子系统的边界内——哪个子系统负责什么
    需求隐含功能安全或合规要求 "电机不能过热"隐含了 IEC 60730 / UL 60730 的 Class B 要求——需要显式化
    开发团队说"需求不清楚，没法做" requirements-decompose 把模糊需求变成可执行的工程规格

    ❌ 不该用¶

    场景 为什么不该用 替代方案
    需求已经非常明确且单一（改一个 GPIO 引脚） 需求分解的成本 > 价值 直接写规格：[spec-authoring]
    需求是纯软件功能，不涉及系统级分解 requirements-decompose 偏系统级——纯软件需求用 interview-me 足够 M3 interview-me
    还在探索阶段——需求本身还没定 分解基于不稳定的输入——每改一次输入就要重新分解 先和客户/PM 确认需求方向

    怎么触发¶

    触发方式¶

    我有一个新的系统需求——为 BLDC 电机控制器增加过温保护（OTP）子系统。

高层需求：
- 产品需要满足 IEC 60730 Class B 功能安全标准中的温度保护要求
- 需要监测 3 路温度（电机绕组 ×1、MOSFET 桥 ×1、MCU 内部温度 ×1）
- 需要三级响应：预警（LED 指示）→ 降功率（PWM 限幅）→ 紧急关断（PWM 输出强制关闭）
- 紧急关断后需要记录故障信息到 EEPROM

帮我分解为系统需求——覆盖功能需求、非功能需求、接口需求、约束条件。

AI: 好的。以下是从高层需求分解出的系统需求集：

# 系统需求规格：过温保护（OTP）子系统

## 文档信息
- 系统：BLDC 电机控制器
- 子系统：过温保护（OTP）
- MCU：STM32G474（Arm Cortex-M4, 170MHz）
- 安全标准：IEC 60730 Class B
- 版本：v0.1（初稿）

──────────────────────────────────────

## 功能需求（Functional Requirements）

### FR-1: 多通道温度采集
**FR-1.1** 系统应同时采集 3 路温度信号：
  - CH1：电机绕组温度（外部 NTC，通过 ADC1_IN1）
  - CH2：MOSFET 桥温度（外部 NTC，通过 ADC1_IN2）
  - CH3：MCU 内部温度（STM32G4 内部温度传感器，ADC1_IN16）
    （注意：内部传感器量程为 -40°C ~ 125°C，超出此范围的读数不符合规格）

**FR-1.2** 温度采集周期 ≤ 100ms（所有 3 通道在 100ms 内完成一轮采集）。

**FR-1.3** 外部 NTC 通道（CH1/CH2）应进行 NTC 线性化转换（查表法或 Steinhart-Hart 方程），
  CH3（内部温度传感器）应使用 STM32 出厂校准值（TS_CAL1/TS_CAL2）做线性插值转换。
  输出单位为 °C，精度目标 ±3°C（外部 NTC，0°C ~ 150°C），
  CH3 内部传感器精度约 ±5°C（受限于出厂两点校准的固有精度）。

### FR-2: 三级保护响应

**FR-2.1 预警级（Warning）**
  - 触发条件：任一通道温度 ≥ 预警阈值（默认 85°C，可配置）
  - 响应动作：
    a. 红色 LED 以 2 Hz 频率闪烁
    b. 通过 UART 上报"OTP WARNING"事件（含通道号 + 当前温度值）
    c. 电机继续正常运行（无功率限制）
  - 解除条件：所有通道温度 < 预警回滞阈值（默认 80°C，可配置）
  - 响应延迟：从温度超标到 LED 闪烁 ≤ 200ms

**FR-2.2 降功率级（Derating）**
  - 触发条件：任一通道温度 ≥ 降功率阈值（默认 105°C，可配置）
  - 响应动作：
    a. PWM 占空比限制为最大值的 50%（通过 TIM1 CCR 寄存器限制）
    b. 红色 LED 常亮
    c. 通过 UART 上报"OTP DERATING"事件（含通道号 + 当前温度 + 当前 PWM 限幅比例）
  - 解除条件：所有通道温度 < 预警阈值（默认 85°C）
  - 响应延迟：从温度超标到 PWM 限幅 ≤ 100ms

**FR-2.3 紧急关断级（Emergency Shutdown）**
  - 触发条件：任一通道温度 ≥ 关断阈值（默认 125°C，可配置）
  - 响应动作：
    a. PWM 输出强制关闭（TIM1 软件刹车——通过 TIM1_EGR 的 BG 位触发）
    b. 故障信息写入 EEPROM（含时间戳、触发通道、触发温度、关断前 1s 内温度变化曲线）
    c. 红色 LED 以 5 Hz 频率快速闪烁
    d. 通过 UART 上报"OTP SHUTDOWN"事件
  - 解除条件：手动复位或断电重启（不可自动恢复）
  - 响应延迟：从温度超标到 PWM 关断 ≤ 50ms

### FR-3: 硬件冗余检测路径

**FR-3.1** 除 MCU ADC 软件采集路径之外，系统应提供一个独立的硬件过温检测路径：
  - 使用 STM32G4 内部比较器（COMP1）对 MOSFET 桥 NTC 分压信号进行硬件阈值比较
  - 比较器输出通过内部互连直接连接到 TIM1 BRK（刹车）输入——绕过 MCU 软件，纯硬件关断通路
  - 硬件阈值对应的温度点：130°C（高于软件关断阈值 125°C，作为软件路径失效时的最后防线；
    注意：130°C 接近 MCU Tjmax 绝对最大额定值，实际应用中需评估热设计裕量）

**FR-3.2** 硬件检测路径应独立于软件——即使 MCU 程序跑飞或 ADC 故障，
  硬件比较器仍能触发 PWM 刹车。但需注意：COMP1 + DAC1 的初始化和使能仍需软件在启动时完成配置，
  配置完成后（TIM1 BRK 使能 + COMP1 输出极性确认）方可独立运行。

### FR-4: 故障记录与诊断

**FR-4.1** 紧急关断发生后，系统应在 500ms 内将以下信息写入 EEPROM：
  - 事件编号（递增计数器，永不归零）
  - 关断时间戳（基于 RTC）
  - 触发通道编号
  - 触发时刻温度值（°C）
  - 关断前 1 秒内 3 个通道的温度值（每 100ms 一个采样点，共 30 个数据点）
  - 复位源信息（RCC_CSR 寄存器值）

**FR-4.2** 系统应在运行期间维护以下诊断计数器（存储在 SRAM 中，可通过 UART 查询）：
  - 预警触发次数（分通道）
  - 降功率触发次数（分通道）
  - 紧急关断总次数
  - ADC 采样超时次数
  - NTC 开路/短路检测次数

## 非功能需求（Non-Functional Requirements）

### NFR-1: 时序要求
- 温度采集周期：≤ 100ms
- 预警响应延迟：≤ 200ms
- 降功率响应延迟：≤ 100ms
- 紧急关断响应延迟：≤ 50ms
- 硬件冗余关断延迟：≤ 10µs（比较器传播延迟 ~16.7ns typ + 刹车生效延迟）

### NFR-2: 安全完整性
- OTP 子系统的软件部分应符合 IEC 60730 Class B 要求
- 需要实现以下 Class B 诊断措施：
  - CPU 寄存器自检（启动时）
  - 程序计数器自检（运行时）
  - 栈溢出检测
  - ADC 参考电压监控
  - 温度传感器合理性检查（3 路之间互校——任意两路温差不应超过基于热设计的预期范围，
  如电机绕组 vs MCU 芯片在正常运行中温差可达 30-50°C，交叉校验阈值需根据实际热仿真或实测数据设定）

### NFR-3: 可靠性
- OTP 子系统不应受电机 PWM 噪声干扰（EMC 要求）
- NTC 信号应进行硬件 RC 低通滤波（截止频率 ~10 Hz）
- ADC 采样应采用过采样（4 次取平均）

### NFR-4: 功耗
- OTP 子系统增加的持续功耗 ≤ 5mW（主要是 ADC 和比较器）
- 不影响电机控制的实时性能（PWM 更新周期 ~50µs，OTP 处理不能阻塞 PWM 更新）

### NFR-5: 可配置性
- 三级阈值应可通过 UART 命令修改（存储在 EEPROM 中，上电加载）
- 出厂默认值应固化在固件中（若 EEPROM 读取失败则使用默认值）

## 接口需求（Interface Requirements）

### IR-1: ADC 接口
- ADC1 通道 1（PA0）：电机绕组 NTC（分压电路：NTC + 10kΩ 上拉至 VREF+）
- ADC1 通道 2（PA1）：MOSFET 桥 NTC（分压电路：NTC + 10kΩ 上拉至 VREF+）
- ADC1 通道 16：内部温度传感器（TSENSE）
- ADC1 参考电压：VREF+ = 3.3V ± 1%

### IR-2: 比较器接口
- COMP1 正输入（PA1）：MOSFET 桥 NTC 分压信号（与 ADC1_IN2 共用引脚）
- COMP1 负输入：DAC1_CH1 输出（可配置的硬件关断阈值电压）
- COMP1 输出：内部连接至 TIM1 BRK 输入

### IR-3: 定时器接口
- TIM1：产生 3 路互补 PWM（控制 BLDC 三相桥）
- TIM1 BRK：接收 COMP1 输出（硬件关断）+ 软件触发刹车（软件关断路径）

### IR-4: 通信接口
- UART2（PA2/PA3）：OTP 事件上报 + 诊断数据查询 + 阈值配置
- I2C1（PB6/PB7）：EEPROM（AT24C02, 256 字节）——故障记录存储

### IR-5: GPIO 接口
- PB0：红色 LED（OTP 状态指示）——推挽输出，高电平有效
- TIM1 BRK 引脚（PB12）：也可接收外部硬件关断信号

## 约束条件（Constraints）

### CON-1: 硬件约束
- 使用 MCU 现有引脚——不可增加外部 IC（如专用的温度监控芯片）
- 不使用外部 ADC——所有采样使用 MCU 内部 ADC1
- EEPROM 已在板上（AT24C02），不可更换型号

### CON-2: 软件约束
- 运行在 FreeRTOS 上——OTP 任务优先级应为最高（≥ 5）
- 不使用浮点运算——NTC 转换使用整数查表法（或定点数 Steinhart-Hart）
- Flash 占用 ≤ 8KB（OTP 相关代码）

### CON-3: 合规约束
- 软件需通过 IEC 60730 Class B 自检要求（可参考 ST 提供的 X-CUBE-CLASSB 软件包实现）
- 需求需追溯到设计（每个 FR/NFR 至少对应一个设计元素）
- 需求需追溯到测试（每个 FR/NFR 至少对应一个测试用例）

这是我根据你提供的高层需求分解出的系统需求集。
共：4 条功能需求组（含子需求） + 5 条非功能需求 + 5 条接口需求 + 3 条约束。

需要重点关注的问题：
1. 硬件冗余路径（FR-3）跨软硬件边界——是 SE 的关键设计节点
2. IEC 60730 Class B 的软件自检措施（NFR-2）会影响固件启动流程
3. NTC 线性化精度（±3°C）在高温段（>120°C）可能达不到——需要确认 NTC 规格

    需求分解的关键原则¶

    原则 1：每条需求必须可验证¶

    ❌ 不可验证的"需求" ✅ 可验证的需求
    "系统应在过热时快速响应" "从温度超过关断阈值到 PWM 输出关闭的延迟 ≤ 50ms"
    "温度监测应该准确" "NTC 转换后在 0°C ~ 150°C 范围内精度 ±3°C"
    "系统应可靠" "OTP 子系统在 1000 次连续过温触发测试中无漏触发"

    规则：如果一条需求读完后你无法设计一个测试来验证它——它不是需求，是愿望。

    原则 2：需求应分类¶

    类别 回答的问题 示例
    功能需求（FR） 系统应该做什么 "触发紧急关断时关闭 PWM 输出"
    非功能需求（NFR） 系统应该做到多好 "关断响应延迟 ≤ 50ms"
    接口需求（IR） 系统怎么和外界交互 "通过 UART2 上报 OTP 事件"
    约束条件（CON） 系统不能做什么/必须用什么 "不可增加外部温度监控 IC"

    为什么分类：不同类别在后续设计阶段的处理方式不同。功能需求驱动模块划分，非功能需求驱动技术选型，接口需求驱动引脚分配，约束条件限制设计空间。

    原则 3：需求应有优先级¶

    在 SE 场景中，不是所有需求同等重要。建议标注：

    • MUST：不满足则系统不可接受（如 FR-2.3 紧急关断）
    • SHOULD：应满足，但如果有充分理由可以偏离（如 FR-4.1 故障记录时间 500ms）
    • NICE-TO-HAVE：锦上添花（如诊断计数器可通过 UART 查询）

    做对了的标准是什么¶

    ✅ 成功输出¶

    标准 说明
    每条需求可独立验证 读完后能设计出一个测试来检验是否满足——不需要补充额外信息
    需求覆盖 4 个类别 FR + NFR + IR + CON 齐全——缺一类意味着需求分析不完整
    需求无歧义 两个不同的开发人员读同一条需求→应该设计出相同的实现方案
    隐含的合规/安全要求被显式化 "电机不能过热"背后有 IEC 60730——被提取并写进 NFR
    跨子系统边界的交互被明确 FR-3（硬件冗余路径）清楚标注跨 HW/SW 边界

    ❌ 失败输出¶

    失败表现 为什么不行 怎么办
    需求全是"系统应该 XXX"但没有量化 无法验证——开发不知道做到什么程度算合格 每条 NFR 加上数值指标
    需求遗漏了接口定义 架构设计时发现不知道该用哪个引脚/外设 补 IR 部分——列出所有外部接口
    需求没有标注优先级 MVP 开发时不知道哪些需求可以延期 给每条需求标注 MUST / SHOULD / NICE-TO-HAVE
    需求隐含了实现方案 "使用 FreeRTOS 软件定时器每 100ms 采集温度"——这是设计，不是需求 需求只描述外部可观测行为，不描述内部实现

    常见误区¶

    误区 1："需求越多越完整"¶

    现象：你让 AI 分解需求，AI 给出了 50 条需求——包括"LED 应使用红色 0805 封装"、"UART 波特率 115200"。这不是需求分解——这是过度规格化。

    正确做法：需求描述系统必须满足的条件，不是实现决策。"LED 应使用红色"是需求（因为红色表示告警，是人因工程要求）。"0805 封装"是设计决策——不是需求。判断标准：改变这个条件是否影响系统的功能/安全/合规？如果否——不是需求。

    误区 2："AI 分解的需求不需要人工审查"¶

    现象：AI 给出了完整的需求分解，你直接导入需求管理系统。2 周后发现有一条需求在物理上不可能实现（"响应延迟 ≤ 10ms"——但 ADC 采样 + 滤波就需要 50ms）。

    正确做法：AI 不知道具体的硬件约束细节。每一条 AI 生成的需求必须经过人工可行性确认。特别关注时序需求——AI 容易给出物理上不可实现的数值。

    误区 3："需求分解是一次性的"¶

    现象：你做了需求分解，一个月后设计过程中发现了一个新约束需要追加需求——你觉得"需求文档已经冻结了"，硬着头皮不改。

    正确做法：需求是活的——设计过程中发现新约束意味着需求分析不完整，应该补充。但变更要记录版本和理由。这不是需求管理的失败——恰恰是需求管理的价值。
