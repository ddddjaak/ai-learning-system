    环境配置详解（SE 角色）¶

    这个页面做什么¶

    这一页是 SE（系统工程师）角色的 Claude Code 环境配置参考。读完这一页，你会知道： 1. SE 需要哪些 skill 和 MCP——和嵌入式工程师、AE 有什么不同 2. 如何安装和配置这些 skill/MCP 3. 常见环境问题的排查步骤

    和 M2 的关系：M2 的环境全景讲了 MCP 是什么、怎么配置——这一页聚焦于 SE 角色具体需要哪些 skill 和 MCP。如果你还没读过 M2 的 MCP 配置基础，建议先看一遍。

    SE 角色 skill 清单¶

    SE 专用 skill（16 个 + 1 个 meta-skill）¶

    se-skills 包含 16 个 skill，按 Define → Design → Document → Verify → Validate 五阶段组织：

    Skill 阶段 一句话 覆盖页面
    requirements-decompose Define 原始输入 → 结构化系统需求 4-1
    architecture-design Design 系统级模块分解、接口定义、ADR 4-2
    software-architecture-design Design 固件线程模型、IPC、内存预算 4-3
    hardware-architecture-design Design 引脚分配、电源域、PCB 约束 4-3
    spec-authoring Document SOD、HW-SW IF Spec、Test Plan 4-3
    software-detailed-design Document 函数签名、状态机、错误处理 4-3
    hardware-detailed-design Document 原理图约束、PDN、热分析 4-3
    algorithm-design Document 信号处理、控制回路、滤波器 4-3
    design-review Verify 4 视角对抗审查 4-4
    requirements-review Verify 需求文档清单审查 4-4
    code-static-review Verify 编码规范审查 4-5
    test-plan-review Verify 测试方案完整性审查 4-6
    test-report-review Verify 测试报告正确性审查 4-6
    release-review Verify 发布就绪审查 4-8
    traceability-matrix Validate 跨产物追溯、覆盖率缺口检测 4-7
    using-se-skills Meta 管道引导器——自动检测阶段并路由 —

    Slash 命令入口（6 个）¶

    命令 路由到 模式
    /se-goal 全自动 SE 流程 自主式（Goal Mode）
    /se-requirements requirements-decompose 引导式
    /se-architecture architecture-design → SW/HW 架构 引导式
    /se-spec spec-authoring → 详细设计 引导式
    /se-review 6 种审查 skill 之一 引导式
    /se-traceability traceability-matrix 引导式

    跨角色共享 skill（SE 也有用）¶

    这些 ae-skills 中的 skill 虽然"面向嵌入式工程师"——但 SE 在工作中也经常用到：

    Skill SE 什么时候用 频率
    interview-me 和 PM/客户沟通需求时——澄清模糊的高层需求 中频
    idea-refine 架构选型时——发散多个架构方案做对比 中频
    documentation-and-adrs 记录架构决策——和 architecture-design 的 ADR 互补 中频
    context-engineering 做 code-static-review 或 design-review 前——加载正确上下文 低频
    doubt-driven-development 对关键设计决策做对抗审查（和 design-review 互补） 低频
    code-review-and-quality 审查代码逻辑正确性（不只是编码规范） 低频

    SE 角色 MCP 清单¶

    必装 MCP¶

    MCP SE 场景下的用途 为什么对 SE 重要
    codegraph 审查时快速探索代码结构——调用链、影响分析、符号搜索 SE 审查代码时需要快速理解代码结构——不是逐文件读
    context7 查询 MCU datasheet / ARM 架构参考 / 功能安全标准文档 SE 做架构设计和可行性分析时需要查最新官方文档
    memory 存储项目知识图谱——需求/设计/决策之间的关联关系 SE 维护的是"系统知识"——不是代码片段——跨会话持久化很重要
    sequential-thinking 复杂的架构权衡决策——多维度的 trade-off 分析 SE 经常需要权衡：成本 vs 可靠性、性能 vs 功耗、灵活性 vs 简单性

    选装 MCP¶

    MCP SE 场景下的用途 何时需要
    node_repl 数据处理——如计算 NTC 转换表、生成测试数据、验证算法 当需要数值计算或数据处理时

    环境配置步骤¶

    前提¶

    已完成 M1 的 Claude Code 安装 + CC Switch 配置。如果没有——回到 M1-1-安装与启动。

    Step 1: 确认 skill 是否已安装¶

    SE skill 通过 se-skills 插件安装：

    /plugin marketplace add ddddjaak/se-skills
/plugin install se-skills@se-skills
/reload-plugins

    安装后运行"列出所有可用的 se-skills"——如果看到 requirements-decompose、architecture-design 等 skill——说明已安装。

    Step 2: 配置 CLAUDE.md¶

    在项目根目录的 CLAUDE.md 中增加 SE 相关的配置段。一个 SE 角色的 CLAUDE.md 示例：

    ## SE Role Configuration

### Default Review Standards
- All designs reviewed against: IEC 60730 Class B (for safety-related systems)
- Coding standard: MISRA-C:2012 (Mandatory + Required)
- Company internal: no dynamic allocation, function ≤ 100 lines, snake_case

### Traceability Requirements
- Every FR requires: ≥ 1 design element + ≥ 1 test case
- Every NFR requires: ≥ 1 design analysis or test case
- Backward trace: every test case traces to ≥ 1 requirement
- Gap threshold: 0 RED gaps allowed for release

### Release Gate Criteria
- All MUST requirements: implemented + tested
- All HIGH design review findings: closed
- All MISRA Mandatory violations: 0
- Traceability matrix coverage: 100% for MUST requirements

    Step 3: 配置常用 MCP¶

    codegraph 初始化：

    请帮我初始化 codegraph——索引当前项目。

    首次使用 codegraph 时需要初始化索引——它会解析所有源代码的 AST 构建知识图谱。在包含 CubeMX 生成代码的项目中，建议排除生成代码的目录（通常可以配置 .codegraphignore）。

    memory 设置：

    在 Claude Code 中配置 memory MCP 后，可以存储 SE 工作的持久化信息： - 项目架构决策的演进历史 - 需求变更的追溯记录 - 审查发现的统计趋势

    常见问题¶

    Q1: 报错"skill not found: se-requirements"¶

    原因：se-skills 插件未安装或版本过旧。

    解决方案： 1. 确认 se-skills 插件已安装：在 Claude Code 中运行 /plugin list，检查是否有 se-skills@se-skills 2. 如未安装：/plugin marketplace add ddddjaak/se-skills → /plugin install se-skills@se-skills → /reload-plugins 3. 重启 Claude Code 4. 重新尝试：使用 se-requirements 帮我分解需求

    Q2: codegraph 索引速度很慢——整个项目索引了 10 分钟¶

    原因：CubeMX 生成的 HAL 库代码量很大（数千个符号）——codegraph 在索引所有代码。

    解决方案： 1. 创建 .codegraphignore 文件，排除自动生成代码目录： 

    Drivers/STM32G4xx_HAL_Driver/
Middlewares/
    2. 重新初始化索引 3. 只索引手写代码（通常占项目总代码量的 20-30%）

    Q3: context7 查询 STM32G4 datasheet 返回的内容不够新¶

    原因：context7 拉取的是在线官方文档——如果文档最近更新，context7 的缓存可能有延迟。

    解决方案： 1. 在查询中添加 datasheet 编号："查询 STM32G474 datasheet DS12589 Rev 6" 2. 如果 context7 拉取的版本不对——手动指定 Rev 编号 3. 备选方案：直接下载 PDF 版本的 datasheet → 让 AI 读 PDF（Claude Code 支持直接阅读 PDF 文件）

    Q4: sequential-thinking 在想复杂架构决策时输出太长——淹没了关键结论¶

    原因：sequential-thinking 默认会展示多步推理的完整过程——包括被否定掉的路径。

    解决方案： 1. 在触发时加约束："只用 sequential-thinking 做关键权衡分析——最终结论不超过 5 条" 2. 或者让 AI 先做完整分析，然后追加："把上面的分析总结为 3 个要点"

    Q5: 多个 SE skill 的名字很接近——不知道该用哪个¶

    原因：SE skill 体系中确实有相近的 skill。它们各有分工： - requirements-decompose：主入口——输入高层需求，输出完整需求集 - requirements-review：审查工具——检查已有需求集的完整性、一致性 - spec-authoring：规格撰写——把需求和架构写成正式规格文档

    解决方案： 1. 不确定时使用 using-se-skills 元 skill——它会根据你的输入推荐正确的 skill 2. 或者用 /se-requirements、/se-architecture、/se-spec、/se-review、/se-traceability 这 5 个 slash 命令——它们会自动路由到正确的 skill 3. 或者直接用自然语言描述需求——AI 会根据描述自动选择最合适的 skill

    SE 日常命令速查¶

    每日操作 命令/触发 预计耗时
    上午开始工作 "帮我回顾 OTP 子系统的当前状态——需求、设计、审查发现、测试覆盖率" 2 分钟
    接到新高层需求 "用 se-requirements 帮我分解这个需求" 10-15 分钟
    架构设计 "用 architecture-design 基于需求集生成架构方案" 15-20 分钟
    设计审查 "用 design-review 从 4 个视角审查架构设计" 10-15 分钟
    代码规范审查 "用 code-static-review 对照 MISRA-C:2012 审查 otp_engine.c" 4-10 分钟
    追溯矩阵 "用 traceability-matrix 生成需求→设计→测试追溯矩阵" 4-10 分钟
    发布审查 "用 release-review 做 v1.0 发布审查" 10-15 分钟
    下午关闭发现 "帮我逐个关闭 design-review 中的 HIGH 发现——生成修复方案" 每发现 5-15 分钟
