    architecture-design（SE 场景）¶

    这个 skill 做什么¶

    architecture-design 是一个系统架构设计引擎——你给它一组已确认的系统需求，它帮你设计系统的模块结构、HW/SW 分区、组件选择、接口定义和数据流。在 SE 场景下，这个 skill 的核心价值是把"系统必须满足 N 条需求"翻译成"系统由 M 个模块组成，每个模块的边界和交互方式如下"。

    类比：需求分解是"这个房子需要 3 间卧室、2 个卫生间、1 个厨房"。架构设计是画出平面图——卧室在哪、水管怎么走、承重墙在哪、强弱电怎么分区。前者说什么，后者说怎么做。

    触发方式：/se-architecture 或说"帮我做架构设计"

    相关 SE skill： - architecture-design：主架构设计引擎——输入需求集，输出系统架构（本页） - software-architecture-design：软件架构设计——固件线程模型、IPC、内存预算（见 4-3 规格与详细设计） - hardware-architecture-design：硬件架构设计——引脚分配、电源域、PCB 约束（见 4-3） - software-detailed-design：软件详细设计——函数签名、状态机、错误处理（见 4-3） - hardware-detailed-design：硬件详细设计——原理图约束、PDN、热分析（见 4-3）

    什么时候用 / 什么时候不该用¶

    ✅ 应该用（SE 场景）¶

    场景 为什么需要 architecture-design
    系统需求已确认，需要设计系统架构 需求说"什么"，架构说"怎么做"——这是 SE 的核心产出
    系统涉及 HW/SW 协同设计——需要明确哪些功能放硬件、哪些放软件 错误的 HW/SW 分区是系统级 bug 的最大来源——改动成本极高
    需要评估多个架构方案并做技术选型 同样的需求可以用不同架构实现——SE 需要比较并选出最优
    架构需要在评审中展示和沟通 结构化的架构文档是跨团队（硬件/软件/测试）沟通的共同语言

    ❌ 不该用¶

    场景 为什么不该用 替代方案
    需求还没分解就被要求做架构 基于模糊需求做架构 → 大概率返工 先 se-requirements
    改动范围极小（增加一个 GPIO 输出） 不需要架构设计——现有架构能容纳 直接在详细设计层面处理
    项目是已有产品的维护迭代，架构已确定 architecture-design 是新系统/新子系统设计——不是小改动 在现有架构文档中补充变更说明

    ## 2. HW/SW 分区

| 功能 | 硬件 | 软件 | 理由 |
|------|------|------|------|
| 温度信号调理（NTC 分压 + RC 滤波） | ✅ 硬件 | — | 模拟信号处理必须在 ADC 之前。LPF 的截止频率选择需权衡 PWM 噪声衰减和 ADC 输入阻抗约束——见硬件详细设计 |
| 温度采集与转换 | ADC1（硬件） | DMA 传输 + NTC 转换算法 | ADC 采样由硬件完成，软件负责触发、数据搬运和温度转换 |
| 预警/降功率判断 | — | ✅ 软件 | 阈值比较和状态切换是软件逻辑——需要灵活性和可配置性 |
| 软件路径紧急关断 | — | ✅ 软件 → TIM1 BRK 软件触发 | 软件判断温度超标 → 写 TIM1_EGR 的 BG 位触发软件刹车 |
| **硬件冗余紧急关断** | ✅ COMP1 + DAC1 → TIM1 BRK | — | **完全独立的硬件路径**——不经过任何软件代码。Comparator 输出直接连到 BRK 输入（内部互连） |
| DAC 阈值电压设定 | — | ✅ 软件（初始化时写一次） | DAC 输出值在初始化时设定（对应 130°C 阈值电压），运行期间不变 |
| 故障记录（EEPROM） | I2C1（硬件） | ✅ 软件（I2C 写入） | EEPROM 写入由软件控制——需要协调写入时序（关断后 500ms 内完成） |
| LED 指示 | GPIO（硬件） | ✅ 软件（GPIO 控制） | 闪烁频率由软件控制 |
| 事件上报（UART） | UART2（硬件） | ✅ 软件（格式化 + 发送） | UART 协议由软件实现 |

**关键决策**：硬件冗余路径（COMP1 → TIM1 BRK）是系统安全架构的核心亮点——它确保即使软件完全失效，硬件仍能在 130°C 触发关断。

## 3. 软件模块设计

### 3.1 任务划分

| 任务 | 优先级 | 周期/触发 | 栈大小 | 职责 |
|------|--------|----------|--------|------|
| `otpAdcTask` | 5（高） | 周期 100ms | 512B | 触发 ADC1 扫描采样（3 通道 + 内部温度），DMA 完成中断后执行 NTC 转换，写入温度环形缓冲区 |
| `otpEngineTask` | 4 | 周期 100ms（跟随 ADC 完成信号） | 1024B | 从环形缓冲区读最新温度 → 三级阈值比较 → 状态切换 → 触发响应动作（LED / PWM 限幅 / 刹车 / 事件日志） |
| `otpEventTask` | 3 | 事件驱动（队列接收） | 768B | 接收 otpEngineTask 发来的事件 → 格式化 UART 消息 → 发送。紧急关断时额外执行 EEPROM 写入 |
| `classBSelfTest` | 6（最高） | 启动时 + 运行时定时器 | 512B | 执行 IEC 60730 Class B 自检（CPU 寄存器 / PC / 栈 / ADC 参考电压） |

**优先级说明**：
- classBSelfTest（pri 6）最高——安全自检不能被任何任务阻塞
- otpAdcTask（pri 5）次高——保证温度采集周期
- PWM 控制任务（现有，pri 4）与 otpEngineTask（pri 4）同级——因为 PWM 限幅通过写寄存器完成（临界区保护），不需抢占
- 现有电机控制主循环任务（pri 2）和 BLE 通信任务（pri 1）不受影响

    ### 3.3 状态机设计
    ┌──────────┐ │ NORMAL │ ← 所有温度 < 预警阈值 └────┬─────┘ │ 任一通道 ≥ 85°C ▼ ┌──────────┐ │ WARNING │ ← LED 闪烁 + UART 上报 └────┬─────┘ │ 任一通道 ≥ 105°C ▼ ┌───────────┐ │ DERATING │ ← PWM 50% 限幅 + LED 常亮 + UART 上报 └─────┬─────┘ │ 任一通道 ≥ 125°C (SW 路径) │ 或 COMP1_OUT 触发 (HW 路径, 130°C) ▼ ┌──────────────┐ │ SHUTDOWN │ ← PWM 关断 + 故障记录 + LED 快闪 │ (不可自恢复) │ └──────────────┘ ↑ 手动复位或断电重启 
    **回退逻辑**：
- WARNING → NORMAL：所有通道 < 80°C（5°C 回滞）
- DERATING → NORMAL：所有通道 < 85°C
- SHUTDOWN：不可自动回退——需人工干预

### 4.1 引脚分配

| 功能 | 引脚 | 外设 | 说明 |
|------|------|------|------|
| NTC_ Motor_Winding | PA0 | ADC1_IN1 | 电机绕组 NTC 分压输入 |
| NTC_MOSFET_Bridge | PA1 | ADC1_IN2 / COMP1_IN+ | MOSFET NTC 分压输入（复用为 COMP1 正输入） |
| LED_OTP | PB0 | GPIO | OTP 状态 LED（推挽输出） |

| I2C1_SDA | PB7 | I2C1 | EEPROM 数据线 |
| UART2_TX | PA2 | UART2 | 事件上报输出 |
| UART2_RX | PA3 | UART2 | 指令接收 |
| TIM1_BRK | PB12 | TIM1 BKIN | 外部刹车输入引脚；COMP1 输出通过**内部互连**（不经过 PB12）直接连接到 TIM1 BRK |

### 4.2 NTC 信号调理电路
    VREF+ (3.3V) │ ├── R_pullup (10kΩ, 1%) │ ├──→ NTC (连接器) → GND │ ├──→ R_series (100Ω) ──┬──→ C_filter (100nF) → GND │ │ │ └──→ ADC 输入 (PA0 / PA1) │ └── R_series (100Ω) ──┬──→ COMP1_IN+ (PA1) │ └──→ C_filter (100nF) → GND 
    截止频率 f_c = 1/(2π × 100Ω × 100nF) ≈ 15.9 kHz ——但对于 NTC 信号（变化极慢），实际有效滤波取决于 ADC 采样时间 + 软件过采样。

### 4.3 COMP1 + DAC1 硬件冗余路径
    DAC1_CH1 (内部连接, 不使用 PA4 外部引脚) ──→ 阈值电压 (对应 130°C) │ ▼ COMP1_IN- (PA4 via internal connection) NTC_MOSFET (PA1) ──→ COMP1_IN+ │ ▼ COMP1_OUT ──→ TIM1_BRK (内部直连) │ ▼ PWM 输出强制关断 (Hi-Z) 
    配置参数：
- COMP1 迟滞：HYST=0（无迟滞）——若近阈值处出现振荡，可设 HYST=1（9mV typ）消除噪声抖动
- COMP1 输出极性：正常（温度超标 = NTC 电压下降 < 阈值 → COMP 输出高 → 刹车触发）
- COMP1 速度：STM32G4 比较器为固定超快速模式（传播延迟 ~16.7ns typ），速度不可配置（与部分 STM32F3 系列不同）
- DAC1 输出：由 NTC 分压公式 + 130°C 对应电阻值计算得到

## 5. 关键设计决策记录

### ADR-001: 选 STM32G4 内部比较器而非外部比较器

- **决策**：使用 MCU 内置 COMP1 + DAC1 实现硬件冗余路径
- **备选**：外部独立比较器 IC（如 LM393）+ 外部参考电压（如 TL431）
- **选择理由**：
  1. 零 BOM 成本增加（G4 系列已内置 COMP）
  2. 内部互连到 TIM1 BRK——无需外部走线（PCB 已定型，改板成本高）
  3. 减少元件数量 → 降低失效概率 → 提高整体可靠性
- **风险**：COMP1 和 ADC1_IN2 共用 PA1 引脚——两个功能在硬件上共存但不互斥（COMP 持续监测，ADC 间歇采样）

### ADR-002: 温度采集用 ADC 扫描 + DMA 而非注入通道

- **决策**：使用 ADC1 规则组扫描模式 + DMA 循环缓冲
- **备选**：ADC 注入通道（Injected channel）——可以打断规则组自动插入采样
- **选择理由**：
  1. OTP 的温度采集有固定周期（100ms），不需要"随时打断插入"的能力
  2. DMA 循环缓冲减少软件干预——CPU 只在 100ms 定时器触发时读取最新数据
  3. 注入通道留给更紧急的模拟量（如果有电流保护等更高优先级的 ADC 需求）
- **风险**：DMA 循环缓冲的读写竞态——otpAdcTask 读缓冲时 DMA 可能正在写入。缓解：双缓冲 + 完成标志位

### ADR-003: OTP 任务优先级仅比 PWM 控制高 1 级——而非最高

- **决策**：otpEngineTask 优先级 = 4（与 PWM 控制同级），而非最高优先级
- **备选**：OTP 任务优先级为最高（pri 6），可以抢占一切
- **选择理由**：
  1. 紧急关断的真正硬实时路径是 HW 路径（COMP1 → BRK, 延迟 < 10µs）
  2. SW 路径的关断延迟目标是 50ms——在 FreeRTOS 中 task 4 足以满足
  3. 过高的 OTP 优先级会导致 I2C（EEPROM 写入）抢占其他关键任务——造成不必要的任务饥饿
- **风险**：如果 PWM 控制任务死循环占用 pri 4，SW 关断路径将无法执行。缓解：HW 冗余路径兜底 + 看门狗检测

## 架构设计的关键原则

### 原则 1：HW/SW 分区先于一切

在做任何模块设计之前，先明确每个功能在边界线的哪一边：

| 应该在硬件 | 应该在软件 |
|-----------|-----------|
| 必须在 µs 级响应的安全关断 | 需要灵活配置的阈值比较 |
| 不依赖 CPU 就能工作的保护路径 | 需要记录的故障诊断数据 |
| 模拟信号处理（滤波、调理） | 通信协议（UART、I2C） |

**错误的分区是最贵的 bug**——如果安全关断放软件（有延迟和可靠性问题），或通信协议放硬件（无法修改），架构返工的成本是整个项目的数倍。

### 原则 2：每个设计决策要有理由

架构设计文档不只是"画了哪些模块"——更重要的是**为什么这些模块这样分**。6 个月后新加入的工程师看架构文档应该能理解当时的设计权衡，而不是问"为什么当初不选另一个方案"。

### 原则 3：数据流是架构的骨架

模块定义告诉你"系统由哪些部分组成"。数据流告诉你"这些部分怎么协作"——在嵌入式系统中，数据流直接对应中断嵌套、DMA 配置、任务间通信机制。如果数据流不清晰，互斥锁和竞态条件会在实现阶段大量出现。

## 做对了的标准是什么

### ✅ 成功输出

| 标准 | 说明 |
|------|------|
| **HW/SW 分区表完整** | 每个系统功能明确标注在哪一侧实现——不允许出现"都可以" |
| **模块职责单一** | 每个模块可以用一句话说清"我负责什么"——不能说清说明粒度太粗 |
| **接口定义明确** | 模块间交互方式（共享内存/队列/直接函数调用）和信号类型都已定义 |
| **关键决策有 ADR** | 非平凡设计选择（选哪个外设、为什么不用备选）被记录 |
| **数据流可追踪** | 从传感器输入到执行器输出——中间的每一步转换和传输都可见 |

### ❌ 失败输出

| 失败表现 | 为什么不行 | 怎么办 |
|---------|-----------|--------|
| 架构图只有方框没有数据流箭头 | 方框告诉你"有什么"——箭头告诉你"怎么协作"。只有方框不是架构 | "补充模块间的数据流——X 模块的输出以什么方式传给 Y 模块" |
| HW/SW 分区模糊——"这个功能放硬件或软件都行" | "都行"意味着设计不完整——到实现阶段就会出边界冲突 | 逐条功能确认：硬实时/不依赖CPU→HW，灵活/可配置/需记录→SW |
| 没有任务优先级分配的理由 | 优先级是系统实时性的核心——拍脑袋的优先级是时序 bug 的温床 | 每个任务标注"为什么是这个优先级——如果错了会怎样" |

## 常见误区

### 误区 1："架构设计就是画方块图"

**现象**：架构图只有几个大方块（"温度采集模块"、"保护模块"、"通信模块"），每个方块没有输入输出定义。

**正确做法**：架构图的每个模块应有明确的：输入数据（从哪来、什么格式）、输出数据（往哪去、什么格式）、触发方式（定时/事件/中断）、资源占用（DMA 通道/队列/栈）。

### 误区 2："架构定下来就不能再改了"

**现象**：架构设计做完后，在详细设计阶段发现 COMP1 输出到 TIM1 BRK 的内部连接需要特定的 GPIO 配置——但架构已经"冻结"了。

**正确做法**：架构设计是迭代的。详细设计中发现架构层面的问题 → 更新架构文档并标注变更原因。但变更必须被记录——静默地改架构是技术债务的来源。

### 误区 3："AI 给的架构方案可以直接用"

**现象**：AI 给出了架构设计，你觉得很完整，直接导入设计文档。2 周后发现 AI 把 COMP1 输出接到了错误的极性——"过温时 COMP 输出低"——而 TIM1 BRK 默认是高电平触发。

**正确做法**：AI 给的架构方案是**起点**，不是终点。SE 必须逐条验证：引脚分配正确性（参考 datasheet）、内部互连可用性（参考 reference manual）、任务优先级合理性（基于实际测量数据）。
