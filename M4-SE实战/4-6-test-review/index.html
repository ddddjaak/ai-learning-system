    test-plan-review + test-report-review（SE 场景）¶

    这些 skill 做什么¶

    这一页覆盖 SE 工作流中 Verify 阶段的 2 个测试相关审查 skill：

    Skill 做什么 审查对象 输出
    test-plan-review 审查测试方案的完整性、可追溯性、合规性 测试方案文档 分级问题报告（覆盖缺口、不可执行用例、缺失验收标准）
    test-report-review 审查测试报告的正确性、完整性和覆盖率 测试报告 分级问题报告（失败用例分析、覆盖率统计、未解决阻断项）

    类比：test-plan-review 是考试前审查"出的题对不对"——是否覆盖了所有知识点、有没有出错题。test-report-review 是考试后审查"批改对不对"——是否所有题都批了、有没有漏判。

    触发方式：/se-review 然后选择对应的审查类型，或说"帮我审查测试方案"/"帮我审查测试报告"

    什么时候用 / 什么时候不该用¶

    ✅ 应该用¶

    场景 为什么需要
    测试方案完成，正式执行前需审查 测试方案如果有遗漏——执行完才发现漏测 = 浪费一轮测试周期
    需求变更后——验证测试方案是否覆盖新需求 需求变更常导致测试方案过时——但测试团队可能不知道
    测试完成，报告生成后——发布前需确认测试质量 测试报告的通过率 ≠ 质量好——可能有未覆盖的需求、未分析的失败用例
    发布审查中发现测试覆盖不足——需单独深入审查 release-review 宏观检查 → test-report-review 微观检查

    ❌ 不该用¶

    场景 为什么不该用 替代方案
    测试方案还没写完 审查半成品——发现的问题大多是"缺失"，不是"错误" 先完成测试方案编写
    只想看测试通过率 test-report-review 不只是看数字——它审查覆盖率、失败分析、追溯完整性 直接看测试报告摘要

    test-plan-review：测试方案审查¶

    test-plan-review 基于测试方案审查清单，验证测试方案的完整性和可追溯性。

    审查维度¶

    维度 检查内容 OTP 案例示例
    需求覆盖 每条 FR/NFR 是否有对应测试用例 NFR-4（功耗 ≤ 5mW）是否有功耗测量测试？
    可执行性 测试用例是否有明确的步骤、输入、预期输出 TC-5.1 "注入 125°C"——怎么注入？用什么设备？
    验收标准 每个测试用例的通过/失败标准是否量化 "PWM 关断延迟 ≤ 50ms"——怎么测量？示波器？逻辑分析仪？
    环境要求 测试环境（设备、工具、配置）是否完整列出 需要温度源（热风枪/恒温箱）、示波器、逻辑分析仪
    追溯完整性 每个测试用例是否追溯到需求 ID TC-5.1 → FR-2.3（紧急关断）

    OTP 案例：发现的典型问题¶

    # 问题 严重度 说明
    TP-1 NFR-4（功耗 ≤ 5mW）无对应测试用例 HIGH 功耗需求无法验证——认证审计会指出
    TP-2 TC-5.1 缺少温度注入方法说明 MEDIUM "注入 125°C"不够具体——需要说明是用热风枪还是 DAC 模拟
    TP-3 TC-6.1（硬件冗余测试）缺少安全注意事项 MEDIUM 测试时 PWM 输出接真实电机——需要假负载保护
    TP-4 测试方案未标注测试顺序依赖 LOW TC-7.1（时序测试）依赖 TC-5.1（关断功能）——应标注先后

    test-report-review：测试报告审查¶

    test-report-review 基于测试报告审查清单，验证测试报告的正确性和完整性。

    审查维度¶

    维度 检查内容 OTP 案例示例
    通过率分析 不只看总数——按需求类别分析通过率 FR 通过率 100%、NFR 通过率 80%（NFR-4 未测）
    失败用例分析 每个失败用例是否有根因分析和修复计划 TC-3.2 失败——根因是 ADC 采样时间不足，已修复待复测
    覆盖率统计 需求覆盖率、代码覆盖率、分支覆盖率 需求覆盖率 95%（NFR-4 未覆盖）、代码覆盖率 87%
    未解决阻断项 有没有未解决的阻断问题 无阻断项——但有 2 个已知问题（KP-1, KP-2）
    追溯完整性 测试报告中的结果是否追溯到测试用例和需求 所有测试结果 → TC-XXX → REQ-XXX 完整链路

    OTP 案例：发现的典型问题¶

    # 问题 严重度 说明
    TR-1 NFR-4 功耗测试未执行——但标注为"N/A" HIGH NFR-4 是 MUST 需求——不可标注 N/A
    TR-2 TC-3.2 失败后未复测 MEDIUM 修复了根因但未复测——不知道修复是否有效
    TR-3 测试报告的时间戳缺少时区信息 LOW 2026-06-17 14:30——是 UTC 还是本地时间？

    做对了的标准是什么¶

    ✅ 成功输出¶

    标准 说明
    每条需求有对应测试用例 需求覆盖率 = 100%（MUST 需求）或有合理的不覆盖理由
    每个失败用例有根因分析 不是"TC-3.2: FAIL"——是"TC-3.2: FAIL, 根因: ADC 采样时间不足, 修复: 增加等待时间, 待复测"
    审查发现分级 HIGH/MEDIUM/LOW + 修复建议——不是一锅端的"问题列表"
    追溯链路完整 测试结果 → 测试用例 → 需求 ID → 设计元素

    ❌ 失败输出¶

    失败表现 为什么不行 怎么办
    测试方案覆盖了所有需求但测试用例不可执行 "验证 OTP 功能正常"不是可执行的测试用例 每个用例补充具体步骤和量化标准
    测试报告通过率 100% 但需求覆盖率只有 80% 100% 通过率 + 20% 未测 = 虚假安全感 补充未覆盖需求的测试
    审查只看通过率——没有分析失败用例的根因 失败用例的根因可能是系统性问题——不分析会在生产中复现 逐个失败用例做根因分析

    常见误区¶

    误区 1："测试通过率 100% = 质量好"¶

    现象：测试报告显示 100% 通过率——你很满意。但 20% 的需求根本没有对应测试用例。

    正确做法：通过率和覆盖率是两个维度。100% 通过率 + 80% 覆盖率 = 有 20% 的需求未被验证。两个指标都要看。

    误区 2："测试方案审查是测试团队的事——SE 不需要参与"¶

    现象：你觉得测试方案由测试团队编写和审查——SE 只管需求和架构。

    正确做法：SE 是需求的 owner——只有 SE 知道哪些需求是 MUST、哪些是 SHOULD。测试方案的覆盖优先级应由 SE 确认。

    误区 3："失败用例只要修复了就行——不需要根因分析"¶

    现象：TC-3.2 失败了，开发修了代码，复测通过。你觉得"过了就行"。

    正确做法：没有根因分析的修复可能是"碰巧修好了"——同样的问题可能在其他地方复现。根因分析的价值不只是修当前 bug——而是防止同类 bug。

    进阶¶

    与 release-review 配合¶

    test-report-review 的输出 → 是 release-review 的输入之一。发布审查时会检查测试报告的审查结论。

    与 traceability-matrix 配合¶

    测试覆盖缺口 → 直接输入到 traceability-matrix——追溯矩阵会标注哪些需求缺少测试覆盖。

    与 design-review 配合¶

    测试方案中的验收标准 → 应与 design-review 中的可行性视角一致。如果 design-review 发现"50ms 关断延迟不可行"——测试方案中不应还有"验证 50ms 关断延迟"。
